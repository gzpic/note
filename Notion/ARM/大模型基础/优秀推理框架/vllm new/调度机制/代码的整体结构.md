这份代码完全位于 `vllm/core/scheduler.py` 文件中。为了让你更清晰地理解代码结构，我将这个文件内部的类和函数按照**功能角色**进行归类整理。
你可以把这个文件看作是一个 **“交通指挥中心”**，它由以下几个核心部分组成：**
第一部分：核心数据结构 (Data Objects / DTOs)**
这部分代码定义了调度器内部流转的“货币”和对外输出的“指令单”。它们全是 `@dataclass`，用于存储状态信息。**类名 / 函数名作用描述**`**SchedulingBudget**`**“记账本”**。记录当前 Step 还能处理多少 Token (`token_budget`) 和多少并发请求 (`max_num_seqs`)。调度过程中会不断扣减这个预算。`**ScheduledSequenceGroup**`**“被选中的请求包装”**。包装了一个 `SequenceGroup`，并记录了这次 Step 打算处理它多少个 Token (`token_chunk_size`)。`**SchedulerOutputs**`**“最终指令单”**。这是 `schedule()` 函数的最终返回值。包含了所有 Worker 需要知道的信息：哪些请求要跑、哪些块要 Swap、哪些块要 Copy。`**SchedulerRunningOutputs**`**“运行队列中间结果”**。`_schedule_running` 函数的产出，记录了 Decode 阶段的调度结果（含被抢占的请求）。`**SchedulerSwappedInOutputs**`**“换入队列中间结果”**。`_schedule_swapped` 函数的产出，记录了从 CPU 换回 GPU 的请求。`**SchedulerPrefillOutputs**`**“预填充队列中间结果”**。`_schedule_prefills` 函数的产出，记录了新加入的 Prompt 请求。`**PartialPrefillMetadata**`**“分块预填充状态”**。专门用于 Chunked Prefill 策略，记录当前 Step 有多少长 Prompt 被切分处理了，防止饿死短请求。**
第二部分：枚举与配置 (Enums & Config)**
定义了调度器的行为模式和常量。**类名 / 变量名作用描述**`**PreemptionMode**`**“抢占模式枚举”**。定义了两种抢占方式：
1. `SWAP`: 搬到 CPU (保存状态)。
2. `RECOMPUTE`: 直接丢弃 (重算)。`**ENABLE_ARTIFICIAL_PREEMPT**`**“测试开关”**。用于在测试环境中人为触发抢占，验证系统的鲁棒性。**
第三部分：调度器主类 (Class **`**Scheduler**`**)**
这是文件的核心，包含了所有的业务逻辑。我们可以将其方法进一步细分：**
3.1 初始化与入口方法名作用描述**`**__init__**`初始化三条队列 (`waiting`, `running`, `swapped`)，实例化 `BlockSpaceManager` (显存管理员)，初始化对象缓存池。`**schedule**`**外部调用的总入口**。触发一轮调度，生成 `SchedulerOutputs`，并更新所有请求的 Metadata (供 Worker 使用)。`**add_seq_group**`接收新请求，将其放入 `waiting` 队列。**
3.2 调度策略逻辑 (Strategy)方法名作用描述**`**_schedule**`**策略路由**。根据配置决定走 `_schedule_default` 还是 `_schedule_chunked_prefill`。`**_schedule_default**`**默认策略**。优先调度 Prefill (新请求)，再调度 Decode。容易造成长 Prompt 阻塞生成。`**_schedule_chunked_prefill**`**分块策略**。优先调度 Decode (保证生成流畅)，再调度 Swap，最后用剩余预算调度 Prefill (支持把长 Prompt 切碎执行)。**
3.3 具体阶段实现 (Phases)方法名作用描述**`**_schedule_running**`**处理 Decode 阶段**。最复杂的函数之一。包含**抢占逻辑 (Preemption)**。如果显存不足，会在这里踢人。`**_schedule_swapped**`**处理 Swap-in 阶段**。尝试将 `swapped` 队列里的请求搬回 GPU。`**_schedule_prefills**`**处理 Prefill 阶段**。从 `waiting` 队列取新请求。包含 Token 计数和 Prefix Caching 的计算。**
3.4 显存操作与抢占 (Memory & Preemption)方法名作用描述**`**_preempt**`**执行抢占**。决定是 Swap 还是 Recompute，并调用相应逻辑。`**_swap_in**`** / **`**_swap_out**`调用 `BlockManager` 进行实际的物理块映射修改，并更新请求状态。`**_allocate_and_set_running**`为新请求分配初始显存，并将其状态标记为 `RUNNING`。`**_can_append_slots**`核心守门员。询问 `BlockManager` 是否有足够显存给当前请求生成下一个 Token。**
3.5 辅助计算 (Helpers)方法名作用描述**`**_get_num_new_uncached_...**`**算账核心**。计算一个请求在这一步需要多少算力（Token Budget）。如果是 Prefix Caching 命中的，则不消耗算力预算。`**_chunk_new_tokens_to_schedule**`**切刀**。在 Chunked Prefill 模式下，根据剩余预算强行截断 Prompt 的长度。**
第四部分：性能优化 (Optimization / Caching)**
Python 的对象创建（Object Creation）开销在高性能推理 Loop 中是不可忽视的。vLLM 使用了对象池技术。**函数名作用描述**`**seq_group_metadata_builder**`构建器。`**scheduler_running_outputs_builder**`构建器。`**PyObjectCache**`** (在 **`**utils.py**`** 中)**(在 `__init__` 中被使用) **对象缓存池**。`Scheduler` 类维护了 `_seq_group_metadata_cache` 等列表。为了避免每一步都 `new` 几十个对象造成 GC 压力，它会复用上一轮的对象，只是重置其属性。**
总结**
这就构成了完整的 `scheduler.py`：
1. **数据类**定义了输入输出协议。
2. **Scheduler类** 维护队列状态。
3. **调度策略方法** (`_schedule_*`) 决定谁先跑。
4. **显存交互方法** (`_allocate`, `_preempt`) 负责与 `BlockManager` 协商资源。
5. **缓存池** 保证了 Python层面的执行效率。
