# Eager 模式 vs 计算图模式

## 1. 什么是 Eager 模式

Eager 模式（即时执行模式）是指算子逐条执行、立即求值，不需要先构建完整的计算图再统一调度。

PyTorch 默认就是 eager 模式，写法接近普通 Python，执行到哪一行就算哪一行。

## 2. 为什么服务端推理常用 Eager 模式

- 灵活性高：服务端模型迭代快，动态 shape、条件分支、MoE、early exit 这类逻辑用 eager 天然支持
- 调试方便：线上排查问题时可以直接看中间 tensor 和执行路径
- 硬件资源更充足：服务端 GPU 显存大、算力强，eager 的额外开销占比通常没那么致命
- 生态成熟：很多服务端推理框架直接基于 PyTorch eager 构建
- 快速上线：训练完模型后不一定要先 export / compile，直接加载权重就能跑

## 3. 为什么端侧更常用计算图

- 资源受限：端侧 CPU / GPU / NPU 算力和内存都紧，必须做图级优化
- 去掉 Python 依赖：端侧通常不希望带完整 Python runtime，更常导出成 ONNX、TFLite、MNN、CoreML 这类图
- 方便提前编译：计算图可以做 AOT 编译，针对 конкрет 硬件生成更合适的 kernel
- 调度更稳定：静态图可以提前规划内存分配、算子顺序和 buffer 复用，延迟更可控

## 4. 两者对比

| 维度 | Eager 模式 | 计算图模式 |
| --- | --- | --- |
| 执行方式 | 逐算子即时执行 | 先构图再统一优化执行 |
| 灵活性 | 高，支持动态控制流 | 低，动态逻辑需要额外处理 |
| 调试 | 方便，可直接看中间结果 | 相对更难，图更像黑盒 |
| 性能 | 有 Python overhead 和 kernel launch 开销 | 可做融合、常量折叠、内存复用，峰值性能更高 |
| 部署依赖 | 常常依赖 Python runtime | 更容易纯 C++ / runtime 执行 |
| 典型场景 | 服务端 GPU 推理 | 端侧 / 嵌入式 / 编译器路线 |

## 5. 工程上不要说得太绝对

实际工程中两者不是完全对立的：

- `torch.compile`：在 eager 基础上做图捕获和编译优化
- `CUDA Graph`：把固定 shape 的 eager 执行录制成 graph replay，减少 launch overhead
- ONNX Runtime / TensorRT：服务端也会用图模式加速特定模型

所以更稳妥的说法是：

> 服务端通常更偏 eager 和运行时灵活性，端侧通常更偏计算图和静态优化。

## 5.1 一个非常关键的理解点

很多人会误以为：

> eager 在执行某个算子前，不知道这个算子是什么。

这个说法方向对，但不准确。

更准确的说法是：

> eager 知道当前这个算子是什么，但不知道后面整条计算链会长什么样；graph 在执行前就知道整张图的结构。

可以直接记这张表：

|  | Eager | Graph |
| --- | --- | --- |
| 当前算子是什么 | 知道 | 知道 |
| 输入 shape / dtype / device | 知道 | 知道 |
| 后面还有哪些算子 | 不知道 | 知道 |
| 整体拓扑结构 | 看不到全局 | 提前拿到全局 |

### 这就导致

eager 不容易提前做：

- 算子融合
- 全局内存规划
- 跨算子调度
- 常量折叠

而 graph 更容易做：

- fusion
- memory planning
- 静态调度
- layout 优化

一句话重点：

> 不是 eager 不知道当前算子是什么，而是它不知道“未来”；graph 的优势在于它提前知道整条计算路径，所以能做全局优化。

## 5.2 Eager 模式的优点

这块不用背太多，抓住下面几点就够了：

- 灵活性高  
  动态 shape、条件分支、MoE、early exit 这类逻辑更自然。
- 调试方便  
  可以直接看中间 tensor、逐步排查执行路径。
- 开发迭代快  
  不需要先 export / compile，模型改完就能直接跑。
- 更适合研究和服务端快速上线  
  模型结构变化频繁时，eager 成本更低。

一句话记：

> eager 的最大优点不是绝对性能，而是灵活、好调、上线快。

## 5.3 什么是动态 Shape

一句话定义：

> 模型中某些张量的维度在编译期未知，只有运行时才能确定。

### 核心区别

- 静态 shape：所有维度编译时已知，例如 `(1, 3, 224, 224)`
- 动态 shape：至少有一个维度是变化的，例如 `(B, 3, H, W)`，其中 `B/H/W` 运行时才确定

### 为什么会出现动态 shape

- NLP 场景：不同句子长度不同，`seq_len` 天然是动态的
- Batch 维度：推理服务中请求量波动，`batch size` 不固定
- 检测 / 分割模型：输入图片分辨率不同，或者后处理结果数量不定

### 直观例子

```python
# 静态 shape：固定输入，编译一次就行
model(input)  # input shape 永远是 (1, 3, 640, 640)

# 动态 shape：输入会变
model(input)  # 第一次 (1, 3, 640, 640)
model(input)  # 第二次 (4, 3, 320, 320)
```

### 工程上的 trade-off

|  | 静态 Shape | 动态 Shape |
| --- | --- | --- |
| 性能 | 最优，算子可深度优化 | 有一定运行时开销 |
| 灵活性 | 差，换 shape 往往要重新编译 | 好，一个引擎适配多种输入 |
| 内存 | 可精确预分配 | 往往按 max shape 预留，可能浪费 |

一句话总结：

> 动态 shape 的本质是用一部分性能和内存效率，换部署灵活性。

## 6. 一句话总结

服务端算力富余、重灵活迭代，eager 更常见；端侧资源紧张、重极致优化和稳定性，计算图更常见。
