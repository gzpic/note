---

---
### **1) Attention 系列高性能 Kernel**

必须支持：

- FlashAttention / Multi-Query Attention (MQA)
- Grouped Query Attention (GQA)
- 流式 Attention（分块）
- 高效 Tile 调度 / Tensor Core 调度

**否则算力直接废掉**，大模型不会跑得快。

---

### **2) KV Cache 管理体系（核心！）**

LLM 解码的关键是：

```plain text
每生成一个 token，就要重复读前面所有 token 的 K/V


```

所以需要：

| 能力 | 描述 |
| --- | --- |
| KV Cache 动态分配/回收 | 避免内存爆炸 |
| Paged KV / Chunk KV | **避免连续大 buffer 导致 OOM** |
| KV 精度压缩 | FP16 → BF16 → FP8 → INT8（可选） |
| 序列分组 | 支持不同长度上下文并行 |

没有 KV 管理，就 **不是大模型框架**。

---

### **3) Prefill / Decode 两阶段执行调度**

大模型推理 **不是一次 forward**，而是：

```plain text
阶段一：Prefill（大矩阵，吞吐优先）
阶段二：Decode（小矩阵，高延迟敏感）


```

框架必须有能切、能复用 buffer、能 pipeline 的 **两阶段执行图（Execution Graph）**。

---

### **4) Continuous Batching & 多序列调度**

如果框架无法做到：

```plain text
用户 A、用户 B、用户 C 的生成请求动态混合在同一 batch 中推
而不是一个一个顺序推


```

那么吞吐量会下降 **10x～100x**。

这就是：

- **vLLM：PagedAttention**
- **TensorRT-LLM：Context Phase & Token Phase Scheduler**
的核心价值。

---

### **5) 大规模并行能力（可选，视目标设备而定）**

| 框架定位 | 并行策略重点 |
| --- | --- |
| 服务器 / 数据中心 | TP / PP / DP / MoE Dispatcher |
| 边缘设备 | 不做并行，但做 **内存复用** + **低精度计算** |

传统框架基本 **缺少模型流水线、张量切分执行能力**，需要补。

---

## ⭐ 总结成一句话

> 大模型推理不是“算得快”问题，是“如何长期、增量、流式地生成 token，并在显存/算力受限下保持高吞吐和低延迟”的问题。

因此需要补足：

```plain text
Attention Kernel + KV Cache + Prefill/Decode调度 + Continuous Batching


```

这 4 件事 **决定你能不能叫“大模型推理框架”**。
