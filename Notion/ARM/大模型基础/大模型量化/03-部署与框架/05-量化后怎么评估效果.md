# 量化后怎么评估效果

## 1. 评估目标

量化后评估的核心不是只看“能不能跑”，而是看三件事：

- 模型质量是否下降
- 下游任务效果是否下降
- 推理性能和资源占用是否真的改善

如果按实际重要性来排，常见可以理解成：

1. 任务效果评估
2. 输出一致性评估
3. 推理性能与资源评估

## 2. 第一层：PPL

最常见的基础评估是困惑度（Perplexity，PPL）。

它主要反映：

- 模型对 token 预测能力有没有明显变差

常用方式：

- 用同一数据集评估原模型
- 再评估量化模型
- 对比 PPL 变化

常见数据集：

- WikiText2
- PTB
- C4

PPL 适合做第一层筛查，因为简单且稳定。

## 3. 第二层：下游任务评估

PPL 不等于真实任务效果，所以还要看 benchmark。

常见任务：

- `MMLU`
- `GSM8K`
- `HellaSwag`
- `BoolQ`

它们可以帮助判断：

- 知识能力是否下降
- 推理能力是否下降
- 常识与阅读理解是否下降

## 4. 第三层：输出一致性评估

这类评估关注的是：

- 量化前后生成内容差了多少
- 差异是“可接受变化”还是“明显失真”

常见做法包括：

- token 一致率
- 首 token / 前若干 token 一致率
- 固定 prompt 下的生成对比

它的价值在于：

- 比纯 benchmark 更直观
- 比人工感觉更容易量化
- 很适合做工程上的快速回归检查

需要注意：

- 生成不完全一致不等于模型退化
- 但如果早期 token 就大幅漂移，通常说明量化误差已经比较明显

## 4. 第三层：人工 prompt 检查

端侧项目里，人工检查非常重要。

可以固定一批 prompt，例如：

- 问答
- 总结
- 翻译
- 数学推理
- 代码生成

重点观察：

- 是否重复输出
- 是否逻辑断裂
- 是否异常变短
- 是否出现乱码或明显胡说

很多量化问题，人工 prompt 反而比 benchmark 更快看出来。

## 5. 第四层：logits 差异

如果你在做推理框架、自研算子或量化实现，logits diff 很有价值。

常见对比方式：

- cosine similarity
- L2 distance
- KL divergence

它的意义是：

- 快速判断量化实现是否有 bug
- 快速定位数值漂移是否异常

## 6. 第五层：性能指标

量化评估不只看精度，还要看部署收益：

- 吞吐
- 延迟
- 显存占用
- KV cache 占用
- 功耗

如果精度掉了，但性能和显存没有明显收益，那量化通常就不值得。

## 7. 一个比较实用的工程流程

对端侧 / Jetson 项目，一个够用的最小流程通常是：

1. 跑 `PPL`
2. 跑一两个 benchmark，例如 `MMLU`
3. 做输出一致性对比
4. 做固定 prompt 人工检查
5. 必要时补 logits diff
6. 同时记录延迟、吞吐和显存

## 8. 一个重要认识

不要把 PPL 和任务效果完全画等号。

因为：

- `PPL` 看的是 token 预测
- benchmark 看的是任务完成能力
- 人工测试看的是实际可用性

它们彼此相关，但不完全等价。

## 9. 面试速记版

- 量化评估至少看：任务效果、输出一致性、部署收益
- 快速落地时可以用：PPL + benchmark + 输出对比 + 人工检查
- 做框架和算子时，logits diff 很关键
- 最终判断标准不是“精度单指标”，而是质量、性能、显存一起看
