# Runner v3.0 架构白皮书

**平台：Jetson Orin Nano（UMA）**

**设计语言：C++**

**核心目标：执行确定性（Execution Determinism）**

---

## 摘要（Abstract）

Runner v3.0 是一个为 Jetson Orin Nano 等端侧 UMA 平台设计的 **确定性推理执行引擎**。

其核心目标并非追求极限算子性能，而是通过严格的架构边界、不变量约束与执行路径分离，使 GPU 执行行为成为 **可预测、可调度、可恢复的系统组件**。

Runner v3.0 通过“**物理执行器（Physical Executor）**”设计哲学，彻底剥离逻辑判断与状态管理，确保在复杂系统负载（Camera / ISP / NVENC / TTS 并行）下仍能提供稳定、低抖动的 Decode 延迟。

---

## 1. 核心设计哲学：物理执行器（The Physical Executor）

### 1.1 Runner 的本质定位

> Runner v3.0 是一个无状态的、由物理地址驱动的算子发射器，而不是逻辑体、策略体或状态机。

Runner 永远不参与以下任何逻辑决策：

- 当前是第几轮对话
- Token 的语义位置或历史关系
- 请求归属
- Cache 是否可复用
- Cache 是否需要回收

Runner **只接受已经完成决策的物理信息**：

- 物理指针（void*）
- 物理 block 表（block_tables）
- slot mapping
- 预先计算好的 mask / position_ids

📌 所有判断、策略与决策，必须在 **Scheduler / KV Manager / Radix Tree** 层完成。

---

### 1.2 绝对无状态与幂等性（Stateless & Idempotent）

**设计原则**

- Runner 内部不维护任何 Session State
- 不保存 token / step 计数
- 不记录 cache 使用情况

**C++ 行为约束**

- 每一次 `run_*()` 调用都是独立的原子操作
- 任意调用均可被：
    - 打断
    - 丢弃
    - 重试

**Runner 明确不拥有：**

- 线程模型（不起线程池、不 sleep、不轮询）
- 隐式 counter
- ring head / tail
- radix tree 节点
- reference count

> 执行即真相（Execution is Truth）
> Runner 的执行行为完全由输入的物理参数决定，Debug 时不存在任何“隐藏状态”。

---

### 1.3 逻辑与物理的彻底解耦（核心不变量）

| 维度 | 负责模块 |
| --- | --- |
| Token 语义时间 | Scheduler |
| 前缀匹配 | Radix Tree |
| Block 回收策略 | KV Manager |
| Block 物理位置 | Physical Layer |
| Kernel 执行 | Runner |

Runner **只看物理地址，不看历史语义**。

这保证了执行行为在系统演进、策略替换下仍保持稳定与确定。

---

## 2. 内存与资源架构（针对 Orin Nano UMA）

Jetson Orin Nano 上的主要瓶颈不是算力，而是：

- CPU 调度开销
- 内存一致性与页迁移
- DVFS 引发的延迟抖动

Runner v3.0 采用 **静态池化 + 强制零拷贝** 的内存架构。

---

### 2.1 一次性资源占有（One-time Reservation）

在 `Runner::init()` 阶段，必须完成全部资源分配：

- KV Cache（FP16 / Int8）
- KV Scales
- Static Input Buffers
- Pinned Output Buffer
- CUDA Streams
- CUDA Graph Executable

**运行期红线（严格禁止）：**

- ❌ malloc / new
- ❌ cudaMalloc
- ❌ Tensor resize

> 物理层一旦初始化完成，在 Runner 生命周期内 不可变。
> 所有动态性必须上移到 Scheduler / Manager。

---

### 2.2 KV Cache 的物理化定义

KV Cache 被定义为 **物理池（Physical Pool）**，而非逻辑容器：

- 连续大块内存
- 固定 Block Size（16 / 32）
- Block ID → 物理偏移：O(1)

Ring Buffer / Radix Tree **只是地址分配策略**，

不属于 Runner 的职责范围。

---

### 2.3 Zero-Copy：UMA 平台的强约束

在 UMA 架构下：

- Logits 由 GPU 直接写入 CPU pinned memory
- Sampler 通过 CPU 指针直接读取
- ❌ 严禁使用 cudaMemcpy D2H

Zero-Copy 不是性能优化选项，而是 **平台正确性约束**。

---

### 2.4 Int8 KV 原生支持

Runner 内部原生支持：

- Int8 KV Cache
- 独立 Scale 管理
- Attention 前后处理在同一执行流完成

目的：最大化 Orin Nano（4GB / 8GB）有限显存的可用上下文长度。

---

## 3. 三态执行路径（Three-State Executor）

### 设计前提

> 在端侧系统中，“吞吐最优”与“延迟稳定”是天然冲突的目标。
> 任何试图用一条执行路径同时优化二者的设计，都会在系统负载下失败。

Runner v3.0 内部只存在 **三条完全隔离的执行路径**。

---

### 3.1 Decode Executor（延迟优先 · 核心路径）

**目标**

- μs 级稳定延迟
- 极低 jitter

**核心技术：CUDA Graph Replay**

- 在系统态（地址固定、allocator 稳定）完成 Graph Capture
- 运行期仅进行 Graph Replay
- 使用 Input Binding 更新输入数据，而非重新构图

**Graph 不变量**

- Batch = 1
- Block Size 固定
- Kernel 拓扑固定
- Tensor 地址固定

**运行期唯一变化**

- input buffer 内容
- block_tables 内容

**并发策略**

- 绑定 `cudaStreamPriorityHigh`
- 防止被 Camera / ISP / NVENC / TTS 抢占

**效果**

- CPU launch 开销降低 10×+
- Decode 延迟稳定在微秒级
- 几乎消除延迟抖动

---

### 3.2 Prefill Executor（吞吐优先）

**目标**

- 最大化 FLOPS
- 容忍延迟

**特征**

- 动态 kernel enqueue
- Normal Priority Stream
- 不使用 CUDA Graph

**StreamLLM 语义**

> Compute-All, Store-Sparse

- Attention 计算全上下文
- KV Cache 只写入 mask 允许的 block
- Prefill 不参与 Ring 回收

---

### 3.3 Agent Executor（分层存储感知）

**职责**

- 将“前缀匹配结果”转化为“物理数据是否在 GPU”

**行为**

- 在 GPU → suffix prefill
- 不在 GPU → 发起 async DMA
- load 未完成 → 返回 PENDING

**原则**

- 绝不等待
- 绝不阻塞
- 不做淘汰或 fallback 决策

Agent Executor 是 **Proxy + DMA 发射器**。

---

## 4. 系统稳定性与边界约束（Predictability）

### 4.1 Step Boundary 原子性约束

以下行为 **只能发生在 Step Boundary**：

- KV Block 覆写
- Radix Tree 结构变更
- Ring 回收

即：

> 上一个 Decode Kernel 完全结束之后。

**效果**

- 消除 Attention 与 Cache 管理的竞态
- 显著降低 Debug 难度
- 保证执行幂等性与可恢复性

---

### 4.2 频率锁定（Frequency Locking）

Runner 生命周期包含频率管理：

- init → 锁定 CPU / GPU 频率
- destroy → 解锁

目的不是极限性能，而是 **Decode 延迟的绝对确定性**，避免 DVFS 抖动。

---

### 4.3 内存对齐是功能正确性约束

- Block Size = 16 / 32
- L2 Cache Line 对齐
- Warp 对齐

这是避免隐藏性能退化与异常行为的必要条件，而非优化建议。

---

## 5. 结论（Conclusion）

Runner v3.0 并不是一个“更聪明的执行器”，而是一个：

> 明确声明自己永远不做逻辑判断的物理执行引擎。

正是这些严格的不变量、边界与红线，使其在 Jetson Orin Nano 等端侧平台上实现：

- 可预测的延迟
- 可调度的执行
- 可恢复的运行
- 可长期稳定部署
