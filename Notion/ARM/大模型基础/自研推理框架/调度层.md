# 一、Scheduler 行为规范（升级版）

**NanoFlow Scheduler Specification v1.0**

> 适用目标：Jetson Orin Nano Super（UMA）
> 推理模式：StreamLLM over TensorRT-LLM
> 
> 核心目标：**Decode 实时性 > Prefill 吞吐量**

---

## 1. Scheduler 的核心职责（唯一权威）

Scheduler 是系统中**唯一的“逻辑状态所有者”**，负责：

- 决定 **P / D 谁运行**
- 维护 **KV Cache 逻辑指针**
- 组装 Batch
- 分配 Pinned Memory
- 确认 Step Commit / Rollback

GPU **不参与任何逻辑决策**，只执行 Kernel。

---

## 2. 全局调度不变式（Invariants）

**以下规则任何情况下不可违反：**

1. **Decode 优先级永远高于 Prefill**
2. Decode Step 是 **原子操作**
3. Ring Buffer 的 Block 回收/覆写 **只能发生在 Decode Step 边界**
4. UMA 上 **禁止任何大规模 memcpy**
5. GPU 只看到 **整理好的 Block Table**
6. 逻辑指针只由 Scheduler 更新

---

## 3. 执行模型（Execution Model）

### 3.1 两类任务队列

```plain text
D_Queue : Decode Requests (实时)
P_Queue : Prefill Requests (后台)


```

### 3.2 主调度循环（Canonical Logic）

```c++
while (engine_running) {

    if (!D_Queue.empty()) {
        run_decode_step();
        continue;
    }

    if (!P_Queue.empty()) {
        run_prefill_chunk();
        continue;
    }

    idle_or_sleep();
}


```

> 该逻辑不可被“公平调度”“轮询”等策略替代。

---

## 4. Prefill (P) 执行规范

### 4.1 执行粒度

- Prefill 必须 **Chunked 执行**
- 推荐 Chunk Size：`128 ~ 256 tokens`
- Chunk 之间：
    - 可中断
    - 可被 Decode 抢占

---

### 4.2 数据写入规则（UMA Red Line）

**Prefill Kernel：**

- 必须执行 **全量 Attention**
- KV 写入规则：
    - Sink Tokens → 写入 Ring Buffer
    - Rolling Window Tokens → 写入 Ring Buffer
    - 中间 Tokens → **只算不存**

❌ 禁止：

- 临时 KV Buffer
- Prefill 结束后的 memcpy

✅ 正确模式：

```plain text
Compute All → Store Sparse → Zero Copy


```

---

### 4.3 Prefill Batching 策略

- 允许 Aggressive Batching
- 优化目标：
    - Tensor Core 利用率
    - FLOPS 最大化
- 精度建议：
    - W8A16 / FP16

---

## 5. Decode (D) 执行规范

### 5.1 Decode Step 原子性

- 一个 Decode Step = 生成 1 个 Token
- Step 内：
    - 不可拆分
    - 不可被抢占
- Step 结束后：
    - 才允许更新 Ring Buffer 指针

---

### 5.2 Decode Micro-Batching

- 允许 Batch Size ≤ 4
- 必须满足：
    - 固定 Batch Size
    - 固定访问模式
    - 固定 Kernel Shape

⚠️ 动态波动 Batch Size = 违规调度

**退化策略：**

- 若带宽饱和 / Latency 抖动：
    - 强制降级为 Batch=1

---

### 5.3 Decode 输出 & CPU 协作

- Logits 直接写入：
    - CPU 可见的 Pinned Memory
- GPU 与 CPU Pipeline：

```plain text
GPU: Step N+1 Attention / KV
CPU: Step N Logits Sampling


```

---

### 5.4 Step Commit 规则（Lazy Commit）

7. GPU 完成 Step N 的 KV 计算
8. CPU 进行采样
9. 若采样接受：
    - Scheduler 更新逻辑指针
10. 若拒绝 / EOS：
    - 指针不动
    - 当前 Step 数据视为 Dirty

---

## 6. 硬件与系统约束

### 6.1 频率管理

- Decode Loop 期间：
    - 锁定 GPU / CPU 频率
- 防止 DVFS 导致的 Latency Jitter

---

### 6.2 CUDA Stream 策略

- Prefill Stream：Normal Priority
- Decode Stream：High Priority
- 禁止混用

---

### 6.3 Python Runtime（如适用）

- Decode Loop 内：
    - 禁止 GC
    - 禁止动态对象分配

---

## 7. 无锁同步原则（CPU / GPU）

- 允许：
    - `std::atomic<bool>` 标志位
- 仅用于：
    - Step 完成
    - Logits Ready
- ❌ 禁止：
    - 复杂状态机
    - 多阶段依赖

## 📄 Scheduler Design Prompt（Implementation-Oriented）

> 你正在实现一个端侧大模型推理引擎的调度层，目标平台是 Jetson Orin Nano Super（UMA 架构），推理框架基于 TensorRT-LLM + StreamLLM。
> 系统需要支持 **多轮对话 + 少量并发**，但必须保证 **Decode 的实时性高于一切**。

### 核心要求

11. **调度优先级**
    - Decode 请求为实时任务
    - Prefill 请求为后台任务
    - 只要 Decode 队列非空，必须立即执行 Decode Step
12. **执行粒度**
    - Prefill：Chunked 执行（128~256 tokens），Chunk 边界可被中断
    - Decode：Step-based 执行（1 token），Step 原子不可拆分
13. **主调度逻辑（不可修改）**
```c++
if (D_Queue not empty) {
    run_decode_step();
} else if (P_Queue not empty) {
    run_prefill_chunk();
}


```
14. **KV Cache 规则**
    - 使用静态 Ring Buffer
    - Prefill 阶段：
        - 所有 token 参与 Attention
        - 仅 Sink + Rolling Window 的 KV 写入显存
        - 禁止任何 memcpy
    - Decode 阶段：
        - 每步只写当前 token 的 KV
        - Block 回收只能发生在 Step 边界
15. **Batching**
    - Prefill：允许 Aggressive Batching
    - Decode：允许 Micro-Batching（≤4），必须固定模式
    - 若带宽或延迟不稳定，Decode 自动退化为 Batch=1
16. **CPU / GPU 协作**
    - GPU 负责 Attention / KV
    - CPU 负责 Logits Sampling
    - 允许 CPU/GPU pipeline 并行
17. **一致性模型**
    - Scheduler 是唯一逻辑状态源
    - GPU 只执行 Kernel，不维护逻辑状态
    - Decode Step 采用 Lazy Commit
18. **系统约束**
    - UMA 架构下禁止大规模 memcpy
    - Decode Loop 期间建议锁定 CPU/GPU 频率
    - Python 环境下禁止 GC

### 输出期望

- Scheduler 类设计
- 主循环伪代码
- Prefill / Decode 执行函数
- 队列与状态管理逻辑
- 明确标注不可违反的约束

# 一、Prefill（P）阶段：目标 = **最大吞吐、最少干扰**

> Prefill 的唯一 KPI：tokens / second 最大化
> 延迟不是 KPI，稳定性也不是（前提是不影响 Decode）

---

## 1️⃣ Prefill 必须 Chunk 化（这是前提）

### 推荐参数

- **Chunk Size**：256 / 512 tokens
- **调度语义**：
> Prefill = 一系列可中断的子任务

### 工程收益

- 避免长时间占用 SM
- Scheduler 可以在 chunk 边界插入 Decode step
- 避免 UMA 带宽长时间饱和

---

## 2️⃣ Prefill 批处理（Batching）只限 Prefill

### 正确做法

- 多个请求的 Prefill **可以 batch**
- Decode **禁止 batch（或 batch=1）**

```plain text
PrefillBatch = merge(P1_chunk, P2_chunk, P3_chunk)


```

### 为什么安全？

- Prefill 之间 **无状态依赖**
- KV 写入位置已确定
- 不涉及 Ring Buffer 回收

---

## 3️⃣ Prefill 使用“非 Ring Buffer KV”

你之前的设计已经暗含了这一点，这里明确化：

- Prefill：
    - 使用临时 KV
    - 或独立 Linear KV
- Prefill → Decode 迁移时：
    - 只拷贝 **最后 Sink + Rolling 所需的部分**

👉 **减少无意义 KV 写入**

---

## 4️⃣ Prefill 允许更激进的算子配置

在 TRT-LLM / Engine 层：

- 更大的 GEMM tile
- 更高 occupancy
- 允许更高 register 压力
- 允许 latency spikes

Prefill 的 kernel：

> “算得猛” > “算得稳”

---

## 5️⃣ Prefill 的 CUDA Stream / Priority

- 使用 **低优先级 CUDA stream**
- Decode 使用默认 / 高优先级 stream

👉 NVIDIA driver 会自动偏向 Decode

---

## 6️⃣ Prefill 的提前终止优化（实用）

如果用户输入极长：

- 只 Prefill 到：
```plain text
max_len = sink + rolling


```
- 前面的 token：
    - 直接丢弃
    - 或只做 embedding，不落 KV

👉 Prefill 从一开始就为 StreamLLM 服务

---

# 二、Decode（D）阶段：目标 = **稳定、确定、低抖动**

> Decode 的 KPI：step latency 稳定性

---

## 1️⃣ Decode 永远是 Step-Granularity

**禁止**：

- 合并多个 Decode step
- speculative decode（端侧 Orin 不划算）

Decode 的节奏必须是：

```plain text
D(step 1) → D(step 2) → D(step 3)


```

---

## 2️⃣ Decode 的 KV 行为必须完全确定

- Block ID 在 step 开始前就确定
- 写入位置固定
- 回收只在 commit 阶段

👉 **无条件避免任何 runtime decision**

---

## 3️⃣ Decode Kernel 强约束

### Decode Kernel 必须：

- Block size = 16 / 32
- Memory access 连续
- Warp 无分歧
- 无动态 shape 分支

Decode 的 kernel目标不是 FLOPS，而是：

> 最小 tail latency

---

## 4️⃣ Decode 禁止任何数据搬运

- ❌ cudaMemcpy
- ❌ swap
- ❌ KV 重排

Decode step 唯一允许修改的“数据”：

- Host 端 `block_ids` 表
- `position_ids`

---

## 5️⃣ Decode 的 Logits 计算内嵌

你前面已经问到，这里再强调一次：

- Logits 是 Decode graph 的一部分
- 不拆 kernel
- Scheduler 只“消费 logits”

---

## 6️⃣ Decode 的 Sampler 与 Engine 解耦

- Engine：
    - 只产 logits
- Sampler：
    - CPU / 轻量 GPU
    - 可异步

但：

> ❗ Sampler 完成前，禁止 commit KV

---

## 7️⃣ Decode 的 Stop 优先级最高

一旦：

- EOS
- Max token
- User interrupt

👉 **立即终止 Decode loop**

- 不等 Prefill
- 不等其他请求

---

# 三、P / D 协同优化（很多人会忽略）

## 1️⃣ Decode 感知 Prefill 压力

Scheduler 可以动态调整：

- Prefill chunk size
- Prefill batch size

依据：

- 最近 N 个 Decode step latency

---

## 2️⃣ Prefill 感知 Decode 窗口大小

如果 Rolling Window 较大：

- Prefill 自动减速
- 减少并发 Prefill 数

---

# 四、最终对照表（给你一个“工程 Checklist”）

| 项目 | Prefill (P) | Decode (D) |
| --- | --- | --- |
| KPI | Throughput | Latency Stability |
| 是否 batch | ✅ | ❌ |
| 是否 chunk | ✅ | ❌ |
| CUDA Stream | 低优先级 | 高优先级 |
| KV 策略 | 临时 / 线性 | Ring Buffer |
| 数据搬运 | 允许（有限） | 禁止 |
| 调度优先级 | 低 | 最高 |

---

## 五、拍板级总结（一句话）

> 在端侧 StreamLLM 架构中，Prefill 应被设计为可中断、可批处理、追求吞吐的后台任务，而 Decode 必须被设计为严格 step 化、零拷贝、零抖动的前台实时任务；二者在调度与执行路径上彻底分离、分别极致优化，才能在 Orin Nano 的 UMA 约束下同时获得可用的长上下文与稳定的交互体验。
