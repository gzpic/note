# 布局优化

## 1. 布局到底在优化什么

像 `BSHD`、`BHSD` 这类布局，不改变数学结果，改变的是：

- 数据在内存里的排列方式
- kernel 读取时是否连续
- 是否需要额外 `transpose / reorder / copy`
- 是否更容易做向量化和 Tensor Core / SIMD 加速

一句话：

> 布局优化，本质是在优化访存，而不是在优化公式。

---

## 2. 四个维度先统一

在大模型里常见记法：

- `B`：batch
- `S`：sequence / token 位置
- `H`：head
- `D`：head_dim

最常见两种布局：

- `BSHD`
- `BHSD`

它们的区别，本质就是：

> 到底是让 token 相关访问更顺，还是让 head + 历史序列扫描更顺。

---

## 3. 为什么隐藏层等重要中间数据常常更偏 `BSHD`

这是最重要的一题。

### 3.1 先说结论

隐藏层、QKV 投影前后、RoPE 临时张量这类“计算中间态”，很多实现更偏 `BSH` 或 `BSHD`，核心原因是：

- 它们大多数是按 token 处理的
- 每个 token 对应一个 hidden vector
- norm / linear / rope 这些算子的热点访问更接近 token-major

所以更自然的组织方式常常是：

```text
[B, S, H, D]
```

也就是 `BSHD`。

### 3.2 为什么 `BSHD` 更顺手

因为大部分中间算子的处理逻辑都是：

- 固定一个 token
- 取这个 token 的整条 hidden 向量
- 做 norm、线性变换、位置编码

例如：

- Embedding 输出更像 `BSH`
- RMSNorm / LayerNorm 是对单个 token 的 hidden 维做归一化
- QKV Projection 也是把每个 token 的 hidden 向量送进 GEMM
- RoPE 是对每个 token、每个 head 的小向量做旋转

这些算子里，最自然的计算单位是：

- “一个 token 的向量”

因此中间表示更偏：

- `BSH`
- 或者把 hidden 再拆头，变成 `BSHD`

### 3.3 为什么它很重要

因为 hidden state 是整个模型里最常流经各层的主干数据。

如果这条主干布局不顺，会导致：

- norm 读写不连续
- linear 前后频繁转置
- RoPE 需要额外数据重排
- prefill 阶段 kernel 不好做大吞吐

所以：

> 隐藏层等重要中间张量常用 `BSHD`，不是因为名字好看，而是因为 token-major 的算子链路更自然、更省重排、更容易跑快。

### 3.4 一版面试答案

可以直接答：

> 因为 hidden state、QKV 投影、RoPE 这些中间计算本质上是按 token 处理向量的，`BSHD` 更符合 token-major 的访问方式。这样 norm、linear、RoPE 这类算子组织更自然，也能减少额外转置和数据重排，所以很多重要中间张量会更偏 `BSHD`。

---

## 4. 为什么又不是所有东西都用 `BSHD`

因为 attention，尤其 decode 阶段，热点访问模式变了。

decode 时更像：

```text
Q[b,h,1,d] × K[b,h,0:t,d]
```

访问模式是：

- `(b,h)` 固定
- 扫描历史 `s`
- 每次读完整个 `d`

这时更顺手的布局是：

```text
[B, H, S, D]
```

也就是 `BHSD`。

原因：

- 固定 `(b,h)` 后，扫描历史 token 更连续
- 更适合按 head 并行
- 更适合 decode 阶段的 KV cache 读取

所以真实框架通常不是一个布局走到底，而是：

- 中间计算张量更偏 `BSHD`
- attention / KV cache 更偏 `BHSD`

---

## 5. Prefill 为什么更偏 `BSHD`

prefill 的特点：

- 一次处理整段 prompt
- token 数量大
- attention 更像大矩阵计算
- GEMM 是主力

这时：

- token 维并行度大
- hidden state 更像一批 token 向量

所以 prefill 阶段的很多算子更偏：

```text
BSHD
```

---

## 6. Decode 为什么更偏 `BHSD`

decode 的特点：

- 一次常常只生成 1 个 token
- token 维并行度很低
- 核心热点变成扫描历史 KV cache

这时最自然的是：

- 一个 `(b,h)` 负责一段历史扫描

因此 decode attention / KV cache 路径更偏：

```text
BHSD
```

---

## 7. 为什么 KV Cache 往往用 `BHSD`

因为 decode 阶段要反复读取历史 K/V。

如果 KV cache 是：

```text
[B, H, S, D]
```

那么固定 `(b,h)` 后：

- `D` 连续
- `S` 递增扫描更顺
- 向量化更自然

如果硬存成 `BSHD`，就更容易出现：

- 跨 stride 跳跃
- cache locality 变差
- coalesced access 更差

所以：

> hidden state 常偏 `BSHD`，KV cache 常偏 `BHSD`，是由两条链路完全不同的热点访问模式决定的。

---

## 8. 一张总表

| 模块 | 常见布局倾向 | 原因 |
| --- | --- | --- |
| Embedding / Hidden State | `BSH` / `BSHD` | 按 token 处理更自然 |
| RMSNorm / LayerNorm | `BSH` / `BSHD` | 对单 token 向量做归一化 |
| QKV Projection | `BSH` / `BSHD` | token-major 的 GEMM |
| RoPE | `BSHD` | 对每个 token、每个 head 的小向量处理 |
| Prefill Attention 临时张量 | 更偏 `BSHD` | 序列大、并行度来自 token |
| KV Cache | `BHSD` | decode 扫描历史序列更顺 |
| Decode Attention 热点访问 | `BHSD` | 固定 `(b,h)` 扫 `s` |

---

## 9. 最后怎么判断一个布局好不好

不要只看名字，要看这几个问题：

- 最热点的访问维度是不是连续
- 是否减少了 `transpose / reorder / copy`
- 是否更利于向量化
- 是否更适合 Tensor Core / SIMD
- 是否更适合 prefill 或 decode 当前阶段的主要瓶颈

一句话：

> 谁是热点访问维度，谁就应该尽量更连续。

---

## 10. 一版最终可直接背的总结

大模型里不是所有张量都统一用 `BSHD`。  
更准确地说：

- hidden state、QKV 投影、RoPE 这类中间计算张量更偏 `BSHD`
- attention 尤其 decode 阶段，以及 KV cache，更偏 `BHSD`

原因是：

- 前者主要按 token 处理向量，`BSHD` 更自然
- 后者主要固定 `(b,h)` 扫描历史序列，`BHSD` 更高效

所以布局优化的本质不是选一个“万能布局”，而是：

> 让不同阶段的热点访问模式都尽量顺着内存来。
