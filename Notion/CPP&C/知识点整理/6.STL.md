# 6.1 容器分类

## 序列式容器

- `vector`
- `deque`
- `list`
- `array`
- `forward_list`

特点：

- 强调元素顺序
- 按位置访问

## 关联式容器

- `map`
- `set`
- `multimap`
- `multiset`

特点：

- 强调键和值或键本身
- 底层通常是红黑树

## 无序关联式容器

- `unordered_map`
- `unordered_set`

特点：

- 底层通常是哈希表

## 容器适配器

- `stack`
- `queue`
- `priority_queue`

特点：

- 借助底层容器，限制接口后对外使用

# 6.1.1 STL 的优势

- 数据结构和算法分离
- 复用性高
- 泛型能力强
- 性能通常较好
- 生态成熟、接口统一

# 6.1.2 STL 的基本组成部分

狭义常答三部分：

- 容器
- 算法
- 迭代器

更完整常答六部分：

- 容器（Container）
- 算法（Algorithm）
- 迭代器（Iterator）
- 仿函数（Function Object）
- 适配器（Adaptor）
- 空间配置器（Allocator）

# 6.1.3 STL 部件关系

- 容器负责存数据
- 算法负责处理数据
- 迭代器把算法和容器连接起来
- 仿函数为算法提供可定制策略
- 适配器对现有接口进行封装和变形
- 空间配置器负责内存获取与释放

# 6.1.4 谓词

谓词本质上是“返回 `bool` 的可调用对象”。

常见分为：

- 一元谓词
  - 接收一个参数，返回真假
- 二元谓词
  - 接收两个参数，返回真假

例如：

```cpp
struct Less {
    bool operator()(int a, int b) const { return a < b; }
};
```

谓词常用于：

- 排序规则
- 查找条件
- 过滤条件

# 6.1.5 内建函数对象

标准库提供了一批函数对象，例如：

- 算术仿函数
  - `std::plus`
  - `std::minus`
- 关系仿函数
  - `std::less`
  - `std::greater`
- 逻辑仿函数
  - `std::logical_and`
  - `std::logical_or`

它们常用于算法、关联容器排序规则、自定义比较策略。

# 6.2 `vector`

- 底层是动态数组
- 支持随机访问
- 尾插效率高

常见内部状态可理解为：

- `start`
- `finish`
- `end_of_storage`

## 扩容机制

- 申请更大空间
- 搬运旧元素
- 释放旧空间

尾插摊还复杂度通常是 `O(1)`。

## `vector` 和 `array` 的区别

- `vector`
  - 动态数组
  - 大小可变
  - 可能扩容
- `array`
  - 定长数组封装
  - 大小编译期确定
  - 不会自动扩容

## 添加元素

### `push_back`

- 把一个已有对象拷贝或移动到尾部

### `emplace_back`

- 直接在尾部原地构造对象
- 对复杂对象通常可少一次临时对象开销

## `string::at()` 和 `operator[]`

- `operator[]`
  - 不做边界检查
- `at()`
  - 会做边界检查
  - 越界时可能抛出 `std::out_of_range`

## 删除元素

### `pop_back`

- 删除尾元素

### `erase`

- 删除指定位置或区间
- 删除点之后的元素通常需要前移

### `remove` / `erase-remove`

`remove` 本身不真正缩容容器，而是把“保留元素”前移，返回新的逻辑结尾迭代器。

常见写法：

```cpp
v.erase(std::remove(v.begin(), v.end(), x), v.end());
```

## 遍历方式

### 下标遍历

```cpp
for (size_t i = 0; i < v.size(); ++i) {
    std::cout << v[i] << ' ';
}
```

### 迭代器遍历

```cpp
for (auto it = v.begin(); it != v.end(); ++it) {
    std::cout << *it << ' ';
}
```

### 范围 `for`

```cpp
for (const auto& x : v) {
    std::cout << x << ' ';
}
```

## 范围 `for` 与迭代器遍历区别

- 范围 `for` 写法更简洁
- 迭代器遍历更灵活，便于做插入、删除、与算法配合
- 需要精细控制当前位置时，迭代器更合适

## 手动释放 `vector` 内存的几种方式

```cpp
v.clear();
v.shrink_to_fit();
```

或：

```cpp
std::vector<int>().swap(v);
```

## `resize` 和 `reserve` 的区别

- `resize(n)`
  - 改变的是 `size`
  - 可能创建或删除元素
- `reserve(n)`
  - 改变的是预留容量 `capacity`
  - 不会直接改变当前元素个数

## 一句话

- `resize` 改“已有多少元素”
- `reserve` 改“最多先预留多少空间”

## 迭代器失效

### 插入/扩容

- 若触发扩容，原有指针、引用、迭代器通常全部失效
- 若不扩容，尾插前的元素通常仍有效，但 `end()` 会变化

### 删除

- 删除末尾元素
  - 指向被删元素及旧 `end()` 的迭代器失效
- 删除中间元素
  - 被删位置及其后的迭代器、引用、指针通常失效

# 6.2.1 `vector` 的实现原理

- 底层是一段连续内存
- 不够用时重新申请更大空间
- 把旧元素搬运到新空间
- 释放旧空间

这也是为什么：

- 随机访问快
- 中间插删慢
- 扩容会导致迭代器失效

# 6.3 `list`

- 底层通常是双向链表
- 已知迭代器位置时插入删除是 `O(1)`
- 不支持高效随机访问

## 原理理解

- 每个节点通常包含数据、前驱指针、后继指针
- 插删高效，但缓存局部性不如 `vector`

## 迭代器失效

- 删除某个元素时，通常只有指向该元素的迭代器失效
- 其他元素的迭代器通常保持有效

# 6.4 `deque`

- 分段连续空间
- 首尾插删高效
- 支持随机访问，但常数通常大于 `vector`

## 原理理解

- 不是单块连续大数组
- 一般由中控结构维护多个缓冲区
- 因此首尾扩展比 `vector` 更灵活

## `deque` 和 `vector` 最大差异

- `deque` 擅长首尾插删
- `vector` 擅长连续存储和缓存局部性
- `deque` 没有单块连续大内存这一前提

# 6.5 `stack` 和 `queue`

- 都是容器适配器
- 默认底层容器通常是 `deque`

# 6.6 `priority_queue`

- 本质是堆
- 默认底层容器通常是 `vector`
- 默认是大顶堆

## 原理

- 堆通常用数组表示
- 队首始终是优先级最高元素

## 大顶堆和小顶堆

- 大顶堆
  - 父节点值大于等于子节点
- 小顶堆
  - 父节点值小于等于子节点

## 为什么堆底层常用数组

- 完全二叉树很适合顺序存储
- 下标关系简单
- 常见关系：
  - 左孩子 `2*i + 1`
  - 右孩子 `2*i + 2`
  - 父节点 `(i - 1) / 2`

# 6.7 `map` 和 `set`

- 底层通常是红黑树
- 查找、插入、删除复杂度通常为 `O(log n)`

## 红黑树记忆点

- 根是黑色
- 红节点不能连续
- 从任一点到所有叶子空节点的黑节点数相同

## 为什么 `map/set` 常用红黑树

- 需要有序性
- 需要稳定的 `O(log n)` 查找、插入、删除
- 相比 AVL，红黑树在插入删除时通常旋转更少，综合表现更均衡

## `set` 和 `map` 的区别

- `set`
  - 只有键
  - 键本身就是值
- `map`
  - 存键值对 `key-value`

## `multiset` / `multimap`

- 允许键重复

## 为什么 `set` 能保证不重复

- 底层有序树插入时会按键比较
- `set` 语义要求唯一键
- 已存在相同键时，不再插入新键

## 为什么插入删除效率高

- 关联容器按节点组织
- 插删主要是树结构调整和指针/链接变化
- 不像 `vector` 那样常常需要整体搬移后续元素

## 为什么插入后旧迭代器通常不失效

- 树节点一般独立分配
- 插入新节点通常不会整体搬移已有节点
- 因此原节点地址通常保持稳定

## `map` 和 `unordered_map` 的区别

- `map`
  - 有序
  - 底层通常是红黑树
  - 查找、插入、删除通常是 `O(log n)`
- `unordered_map`
  - 无序
  - 底层通常是哈希表
  - 平均查找、插入、删除通常是 `O(1)`

## 为什么 `unordered_map` 常常比 `map` 快，但有时反而更慢

`unordered_map` 常快的原因：

- 平均复杂度通常是 `O(1)`
- 不需要维护全局有序性
- 对只查找/插入、不关心顺序的场景很友好

但它有时会更慢，常见原因有：

- 哈希冲突严重时会退化，最坏可能到 `O(n)`
- 扩容和 rehash 代价不小
- 桶数组会带来额外内存开销
- 哈希函数质量差时性能会明显下降
- 实际常数和内存局部性不一定总比树结构好

### 1. 哈希冲突严重时会退化

如果很多 key 都落到同一个 bucket：

```text
bucket

k1 -> k2 -> k3 -> k4 -> k5
```

这时查找不再接近常数时间，而会退化成沿链逐个比对：

- 平均情况变差
- 极端情况可退化到 `O(n)`

所以：

- `unordered_map` 并不是永远都比 `map` 快

### 2. rehash 代价不小

当元素越来越多、负载因子过高时，哈希表往往需要扩容并 rehash。

典型过程：

1. 分配新的 bucket 数组
2. 重新计算或重新分布元素位置
3. 把旧元素迁移到新 bucket

这一轮代价通常接近：

- `O(n)`

如果业务中频繁触发 rehash：

- 性能抖动会比较明显

### 3. 内存占用通常更大

`unordered_map` 往往不仅要存节点，还要维护 bucket array：

```text
bucket array + 节点
```

因此常见现象是：

- 内存占用通常高于 `map`
- 数据量很大时，cache locality 也未必更好
- 于是理论复杂度更优，但实际跑起来不一定更快

### 4. 小数据量时常数可能更吃亏

当数据量很小时，例如几十个元素：

- `map` 的 `O(log n)` 其实层数很浅
- 比如 `n = 50` 时，`log2(50)` 也就几层

而 `unordered_map` 这边往往还要承担：

- hash 计算
- bucket 定位
- 指针跳转

所以小数据量下：

- `map` 反而可能更直接
- 性能未必输给 `unordered_map`

一句话：

- 复杂度是平均/最坏趋势
- 真正快不快，还要看数据分布、哈希函数和常数因素

## 什么时候选 `map`，什么时候选 `unordered_map`

| 场景 | 更常见选择 |
| --- | --- |
| 需要自动有序 | `map` |
| 只关心查找 / 插入效率 | `unordered_map` |
| 数据量很大，且哈希分布较好 | `unordered_map` |
| 数据量很小 | `map` 也常常足够，甚至可能更稳 |
| 需要范围查询、上下界查询 | `map` |
| 需要有序遍历 | `map` |

## 面试一句话模板

- `unordered_map` 平均复杂度更优，但它依赖哈希分布
- 一旦冲突严重、频繁 rehash、数据量很小或需要有序性，`map` 反而可能更合适

## `map` 的实现原理

- 本质是按键组织的平衡搜索树
- 节点存储键值对
- 按键自动有序
- 中序遍历可得到有序序列

## 红黑树 vs 二叉搜索树 vs AVL

- 普通二叉搜索树
  - 极端情况下会退化成链表
  - 时间复杂度可能退化到 `O(n)`
- AVL
  - 平衡更严格
  - 查找通常更优
  - 但插入删除调整代价更高
- 红黑树
  - 平衡要求稍松
  - 插入删除时旋转更少
  - 更适合频繁修改的关联容器

## 为什么不是普通二叉搜索树

- 普通 BST 不保证平衡
- 遇到有序插入时容易退化
- 红黑树能把高度控制在对数级别

# 6.8 `unordered_map` 和 `unordered_set`

- 底层通常是哈希表
- 平均查找、插入、删除复杂度通常是 `O(1)`
- 最坏情况可能退化

## 哈希表理解

- 通过哈希函数把键映射到桶
- 冲突时需要额外策略处理，例如链地址法

## 扩容和冲突解决

### 为什么要扩容

- 装载因子过高时，桶变拥挤
- 冲突增多，查找性能下降

### 扩容时会做什么

- 申请更多桶
- 重新计算元素桶位置
- 把元素重新分布到新桶中

### 常见冲突解决方式

- 链地址法
- 开放地址法
- 再哈希法
- 公共溢出区法

### 面试稳妥说法

- C++ 标准库的 `unordered_map` 常见实现通常基于桶数组 + 链式结构
- 具体扩容策略和桶数量增长规则属于实现细节，不要背死
- 扩容后元素通常要重新分桶，因此相关迭代器通常会失效

## 动态链接下容器可能出现的问题

跨动态库传递 STL 容器时，若双方编译器、运行库、ABI 或分配器配置不一致，可能带来兼容性问题。

面试里可简答为：

- STL 容器二进制布局和运行库实现可能相关
- 跨模块边界传值时要注意 ABI 和运行库一致性

## STL 线程安全性

- 标准容器默认不提供整体线程安全保证
- 多线程同时读同一个容器，若没有并发写入，通常问题较小
- 一旦有线程写容器，就需要同步保护
- 即使只拿着迭代器遍历，若另一线程同时修改容器，也可能出现未定义行为

## 为什么默认不是线程安全的

- 加锁会带来额外开销
- 不同场景需要的同步粒度不同
- 标准库更倾向把同步策略交给使用者

# 6.9 常见复杂度直觉

- `vector`
  - 随机访问快
  - 尾插快
  - 中间插删慢
- `list`
  - 插删快
  - 查找慢
- `map/set`
  - 稳定 `O(log n)`
- `unordered_map/unordered_set`
  - 平均更快，但不保证有序

## `empty()` 和 `size() == 0`

现代标准库里，大多数标准容器的 `size()` 也是常数时间。

但工程上仍常建议优先写：

```cpp
if (c.empty()) { ... }
```

原因是：

- 语义更直接
- 可读性更好
- 不依赖你额外记忆不同容器的复杂度细节

## 面试稳妥说法

- 现代标准容器里 `empty()` 和 `size()==0` 通常都没问题
- 但 `empty()` 更符合意图表达，因此更推荐

# 6.9.1 常见容器查找复杂度

- `vector`
  - 按下标访问 `O(1)`
  - 无序查找 `O(n)`
  - 有序后二分查找可到 `O(log n)`
- `deque`
  - 按下标访问 `O(1)`
  - 查找通常 `O(n)`
- `list`
  - 查找 `O(n)`
- `map/set`
  - 查找 `O(log n)`
- `unordered_map/unordered_set`
  - 平均 `O(1)`，最坏可能退化

# 6.9.2 `vector` 和 `list` 的区别与适用场景

- `vector`
  - 连续内存
  - 随机访问快
  - 尾插快
  - 中间插删慢
- `list`
  - 链式存储
  - 插删快
  - 随机访问慢

## 适用场景

- 读多、随机访问多：优先 `vector`
- 频繁在中间位置插删：可考虑 `list`

# 6.10 allocator / 空间配置器

空间配置器负责：

- 内存获取
- 内存释放
- 对象构造
- 对象销毁

## 面试常见说法

- `allocator` 把“内存管理”和“对象构造”拆开处理
- 标准容器默认都依赖分配器工作

## 注意

- 面试里知道它是“容器背后的内存管理组件”通常就够了

# 6.11 仿函数和普通函数的区别

- 仿函数是对象，函数只是函数
- 仿函数可以携带状态
- 仿函数可被模板更灵活地内联和优化
- 普通函数更直接、写法更简单

## 一句话

- 仿函数 = 重载了 `operator()` 的对象

# 6.12 迭代器

## 作用

- 统一访问不同容器元素的方式
- 把算法和容器解耦

## 为什么有指针还要迭代器

- 指针只天然适合连续内存
- 迭代器可以抽象不同底层结构
- 让同一套算法适配 `vector`、`list`、`map` 等不同容器

## 常见迭代器类别

- 输入迭代器
- 输出迭代器
- 前向迭代器
- 双向迭代器
- 随机访问迭代器

## 常见容器对应迭代器能力

- `vector`
  - 随机访问迭代器
- `deque`
  - 随机访问迭代器
- `list`
  - 双向迭代器
- `map/set`
  - 双向迭代器
- `unordered_map/unordered_set`
  - 前向迭代器

## 删除元素时如何处理迭代器

- `vector/deque`
  - 常用 `it = c.erase(it);`
- `list`
  - 也常用 `it = c.erase(it);`
- `map/set`
  - 删除当前元素前先保存下一个迭代器，或使用支持返回下一个迭代器的接口

## 什么时候会失效

- `vector`
  - 扩容时大范围失效
  - 删除点之后通常失效
- `deque`
  - 插删时失效规则比 `vector` 更复杂
- `list`
  - 通常只有被删节点对应迭代器失效
- `map/set`
  - 插入通常不使旧迭代器失效
  - 删除时通常只有被删元素对应迭代器失效

## 容器删除元素时的常见写法

### `vector/deque/list`

```cpp
for (auto it = c.begin(); it != c.end(); ) {
    if (need_remove(*it)) {
        it = c.erase(it);
    } else {
        ++it;
    }
}
```

### `map/set`

同样推荐优先使用返回下一个有效迭代器的 `erase` 形式；不要依赖过时写法。

# 6.13 数据结构的存储方式

常见可答四种：

- 顺序存储
- 链式存储
- 索引存储
- 散列存储

## 顺序存储

- 逻辑上相邻，物理上也相邻
- 例如数组、`vector`

## 链式存储

- 节点物理位置可不连续，通过指针连接
- 例如链表
