### **8.1 设计模式总览**

设计模式本质上是“在特定场景下反复验证过的设计经验总结”，不是固定模板，也不是越多越好。

面试里常答：

- 目的：提升可维护性、可复用性、可扩展性、灵活性
- 本质：高内聚、低耦合、面向抽象

### **8.2 设计模式分类**

经典 GoF 设计模式通常分为三大类：

- 创建型
  - 关注对象怎么创建
  - 例如：单例、工厂
- 结构型
  - 关注类和对象怎么组合
  - 例如：装饰器
- 行为型
  - 关注对象之间怎么协作、通信
  - 例如：观察者、策略

### **8.3 类图里常见关系**

常见关系有：

- 继承
  - `is-a`
- 实现
  - 类实现接口/抽象约定
- 依赖
  - 临时使用，耦合较弱
- 关联
  - 一个类长期持有另一个类引用
- 聚合
  - 整体和部分关系，但部分可独立存在
- 组合
  - 更强的整体和部分关系，部分通常随整体生命周期结束

### **8.4 软件设计原则**

常见目标：

- 可维护性
- 可复用性
- 可扩展性
- 灵活性

常见原则：

- 单一职责原则
  - 一个类只负责一类职责
- 开闭原则
  - 对扩展开放，对修改关闭
- 里氏替换原则
  - 子类应该能替换父类并保持行为正确
- 迪米特法则
  - 一个对象应尽量少了解其他对象细节
- 合成复用原则
  - 优先使用组合，而不是滥用继承
- 依赖倒置原则
  - 依赖抽象，不依赖具体实现
- 接口隔离原则
  - 不应强迫使用者依赖它不需要的接口

### **8.5 继承的优缺点**

优点：

- 能复用基类已有实现
- 易于表达层次关系

缺点：

- 子类和父类耦合更强
- 父类改动可能影响子类行为
- 容易形成过深继承树

面试稳妥回答：

- 继承不是不能用，但工程上通常强调“组合优于继承”

### **12.1 单例模式**

保证一个类只有一个实例，并提供一个访问它的全局访问点。

> 回顾：类内的
> ```plain text
> 回顾：类内的 static 变量属于类级别而非对象级别，因此所有对象的 static 变量都共享一份内存，且静态函数只能访问静态成员，因为非静态成员属于对象级别，级别不同没办法。
> ```
> 
> **类级别**
> 
> ```plain text
> static
> ```

由于只能包含一个，因此需要`delete` 拷贝构造函数和赋值运算符

**懒汉版**

在第一次调用时才创建实例，以前创建实例用的是 `new` ，线程不安全，需要自己加锁。

C++11 之后常用 Meyers’ Singleton（如下），采用了局部静态变量，这种方法是线程安全的，。

`class SingletonLazy {
public:
    static SingletonLazy& get() {
        static SingletonLazy instance;  // *只会在第一次调用时执行，后续忽略
        return instance;
    }
    
    void show() { cout << "Lazy Singleton" << endl; }

private:
    SingletonLazy() = default;
    SingletonLazy(const SingletonLazy&) = delete;
    SingletonLazy& operator=(const SingletonLazy&) = delete;
};`

**饿汉版**

程序一开始就创建实例，可能影响启动性能

`class SingletonEager {
public:
    static SingletonEager& get() {
        return instance;  // 直接返回已创建的实例
    }
    
    void show() { cout << "Eager Singleton" << endl; }

private:
    SingletonEager() = default;
    SingletonEager(const SingletonEager&) = delete;
    SingletonEager& operator=(const SingletonEager&) = delete;
    
    static SingletonEager instance;  // 静态成员声明
    
    // C++17 之后可以用 inline 在声明的同时完成定义
    // inline static SingletonEager instance;
};

// 静态成员定义（程序启动时创建）
SingletonEager SingletonEager::instance;`

### **12.1.1 单例模式定义**

单例模式要求：

- 保证一个类只有一个实例
- 提供一个全局访问点

实现时通常要保证：

- 构造函数不能公开创建多个对象
- 拷贝构造和赋值操作要禁掉

### **12.1.2 懒汉式优缺点**

优点：

- 延迟创建，用到时才初始化
- 对启动阶段更友好

缺点：

- 传统写法在多线程下要自己处理同步
- 若实现不好，容易出现竞态或重复创建

### **12.1.3 单例常见应用**

- 数据库连接池
- 线程池
- 配置中心
- 日志系统

## 面试注意

- 不要把所有“全局唯一对象”都机械设计成单例
- 单例会引入全局状态，测试和扩展时也有代价

### **12.2 工厂模式**

通过工厂来返回所需的对象，目的是实现创建者和调用者的分离。

**简单工厂**

一个工厂类生产多种产品，通过参数指定类型

`// 产品基类
class Product {
public:
    virtual void use() = 0;
};

// 具体产品
class ProductA : public Product { /*...*/ };
class ProductB : public Product { /*...*/ };

// 简单工厂
class SimpleFactory {
public:
    static Product* createProduct(const std::string& type) {
        if (type == "A") return new ProductA();
        if (type == "B") return new ProductB();
        return nullptr;
    }
};

// 调用者
Product* obj = SimpleFactory::createProduct("A");
obj->use();`

但扩展性差，如果新增产品，需要修改工厂类，改这个大类一不小心其他产品也寄了。

**普通工厂**

每个产品对应一个专属工厂子类，新增产品只需添加新工厂，这样就不用修改原有的工厂类，更符合开闭原则。

`// 工厂基类
class Factory {
public:
    virtual Product* createProduct() = 0;
};

// 具体工厂A → 生产产品A
class FactoryA : public Factory {
public:
    Product* createProduct() override { return new ProductA(); }
};

// 具体工厂B → 生产产品B
class FactoryB : public Factory {
public:
    Product* createProduct() override { return new ProductB(); }
};

// 调用者
Factory* factory = new FactoryA(); // 搭建A产线
Product* obj = factory->createProduct(); // 得到ProductA`

**抽象工厂**

仍然有多个工厂，但一个工厂可以生产多个相关产品。主要用于解决“产品族”问题，下面以“现代风格UI”和“古典风格UI”为例：

`// 抽象产品：按钮
class Button {
public:
    virtual void render() = 0;
    virtual ~Button() = default;
};

// 抽象产品：复选框
class Checkbox {
public:
    virtual void check() = 0;
    virtual ~Checkbox() = default;
};

// 具体产品：现代风格按钮和复选框
class ModernButton : public Button { /*...*/ };
class ModernCheckbox : public Checkbox { /*...*/ };;

// 具体产品：古典风格按钮和复选框
class ClassicButton : public Button { /*...*/ };;
class ClassicCheckbox : public Checkbox { /*...*/ };;

// 抽象工厂接口：定义创建一族产品的方法
class GUIFactory {
public:
    virtual Button* createButton() = 0;
    virtual Checkbox* createCheckbox() = 0;
    virtual ~GUIFactory() = default;
};

// 具体工厂：生产整套现代风格控件 (一个产品族)
class ModernFactory : public GUIFactory {
public:
    Button* createButton() override {
        return new ModernButton();
    }
    Checkbox* createCheckbox() override {
        return new ModernCheckbox();
    }
};

// 具体工厂：生产整套古典风格控件 (另一个产品族)
class ClassicFactory : public GUIFactory {
public:
    Button* createButton() override {
        return new ClassicButton();
    }
    Checkbox* createCheckbox() override {
        return new ClassicCheckbox();
    }
};

// 客户端代码
int main() {
    GUIFactory* factory = nullptr;
    Button* myButton = nullptr;
    Checkbox* myCheckbox = nullptr;

    factory = new ModernFactory(); // 切换到现代风格族

    myButton = factory->createButton();
    myCheckbox = factory->createCheckbox();

    myButton->render();
    myCheckbox->check();

    delete myButton;
    delete myCheckbox;
    delete factory;
}`

### **12.2.1 简单工厂对外提供什么接口**

最常见的是提供一个统一创建入口，例如：

```cpp
Product* create(Type t);
```

或者静态工厂接口：

```cpp
static Product* createProduct(const std::string& type);
```

核心点是：

- 调用者不直接 `new` 具体产品
- 由工厂统一决定返回哪种对象

### **12.2.2 工厂模式优缺点**

优点：

- 把对象创建和对象使用解耦
- 调用者不依赖具体类名
- 便于统一管理创建逻辑

缺点：

- 简单工厂扩展新产品时常要修改工厂类
- 工厂方法和抽象工厂会引入更多类，系统复杂度上升

### **12.3 装饰器模式**

定义：

- 在不改变原对象结构的前提下，动态给对象增加额外职责

核心思想：

- 不靠继承无限扩子类
- 而是通过组合，把“被装饰对象”包起来

常见角色：

- 抽象构件 `Component`
- 具体构件 `ConcreteComponent`
- 抽象装饰 `Decorator`
- 具体装饰 `ConcreteDecorator`

优点：

- 比继承更灵活
- 可按需叠加功能
- 符合开闭原则

缺点：

- 装饰层次多了以后，类和组合关系会变复杂

### **12.4 观察者模式**

定义：

- 一种一对多依赖关系
- 当主题对象状态变化时，所有观察者都会收到通知并更新

常见角色：

- 主题 `Subject`
- 具体主题 `ConcreteSubject`
- 观察者 `Observer`
- 具体观察者 `ConcreteObserver`

优点：

- 主题和观察者解耦
- 便于扩展通知对象

缺点：

- 观察者过多时，通知成本高
- 若设计不当，可能出现循环依赖或更新链过长

### **12.5 策略模式**

定义：

- 把一组可互换算法分别封装起来，并提供统一接口

核心思想：

- 把“怎么做”从“谁来调用”里拆出来
- 运行时可以切换不同策略

优点：

- 避免大量 `if-else`
- 算法可以独立扩展
- 符合开闭原则

缺点：

- 会增加策略类数量
- 调用方需要理解不同策略的区别

## 一句话

- 工厂模式解决“对象怎么创建”
- 装饰器解决“功能怎么动态扩展”
- 观察者解决“状态变化如何通知”
- 策略模式解决“算法怎么灵活切换”
