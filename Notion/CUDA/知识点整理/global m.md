All accesses to global memory go through the L2 cache. Many accesses also pass through the L1 cache,
depending on the type of access and your GPU's architecture. If both L1 and L2 caches are used, a
memory access is serviced by a 128-byte memory transaction. If only the L2 cache is used, a memory
access is serviced by a 32-byte memory transaction.

这是一个非常敏锐的问题！你的理解已经触及了 GPU 性能优化的核心痛点：**“读放大”（Read Amplification）**。

简单直接的回答是：
**是的，你是对的。**
在默认情况下（L1 开启），即使你的代码只写了读取 **1个字节** (`char val = data[0];`)：

1. **L1 缓存** 会强行搬运 **128 字节**。
2. 为了满足 L1 的这 128 字节需求，**L2 缓存** 不得不也搬运 **128 字节**（由 4 个 32 字节的扇区组成）从显存给 L1。
3. **最终结果：** 你只想要 1 字节，显存带宽却消耗了 128 字节。**有效利用率仅为 0.7%**。

合并且对齐：
都是针对 *warp 访问全局内存* 的硬件行为

**合并的定义：** 一个 Warp 中的 32 个线程，访问的内存地址必须是**连续**的，

硬件会尝试把这些访问**合并成尽量少的内存事务（memory transaction）**

**对齐定义：** 访问内存的**起始地址**必须是内存事务粒度（32 字节或 128 字节）的整数倍。


- **32 字节** 来自 **显存（DRAM）的物理特性**。
- **128 字节** 来自 **计算单元（Warp）的逻辑需求**。

为什么合并能：

首先啊缓存的cache line字节决定了一次搬运所搬动的连续的内存数据大小

**未合并（浪费）：**
• 你的 Warp 里有 32 个线程，每个线程只要 4 字节。
• 如果它们访问分散的地址，GPU 必须发起 32 次传输，每次都搬运 128 字节。
• **总传输量：** $32 \times 128 \text{ Bytes} = 4096 \text{ Bytes}$。
• **你有用的数据：** $32 \times 4 \text{ Bytes} = 128 \text{ Bytes}$。
• **有效利用率：** $\frac{128}{4096} \approx \mathbf{3\%}$。
• *意味着：你占用了 100% 的道路宽度，却只运了 3% 的货。***合并（高效）：**
• 32 个线程访问连续地址。
• GPU 发现这一段刚好凑满 128 字节，只发起 1 次传输。
• **总传输量：** $1 \times 128 \text{ Bytes} = 128 \text{ Bytes}$。
• **你有用的数据：** 128 Bytes。
• **有效利用率：** $\frac{128}{128} = \mathbf{100\%}$。
• *意味着：每一比特的带宽都用来做计算了。*

有连续的32的线程访问连续的数据即使是乱序访问也是合并的
