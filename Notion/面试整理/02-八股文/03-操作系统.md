# 操作系统

## 1. 进程与线程

从 `[[操作系统/操作系统]]` 提炼出的核心点：

- 进程是资源分配基本单位
- 线程是调度和执行基本单位
- 同一进程内线程共享地址空间

### 一个线程大概占多大内存

- 面试里常见回答是“线程默认栈空间通常几 MB 级别”
- Linux 下很多发行版里，`pthread` 默认线程栈常见是 `8MB`
- 但这不是绝对常量，受系统、库实现和线程属性设置影响

稳妥回答：

- 线程本身不只是“线程栈”
- 但面试里通常说“一个线程默认主要会占一块线程栈空间，Linux 常见默认约 `8MB`”

## 2. 死锁

从 `[[操作系统/操作系统]]` 提炼出的核心点：

- 死锁定义：多个执行流互相等待对方持有的资源
- 四个必要条件：
  - 互斥
  - 请求与保持
  - 不可剥夺
  - 循环等待
- 解决思路：
  - 预防：打破条件
  - 避免：银行家算法等
  - 检测与解除：抢占资源或终止进程

补一句面试常答：

- 银行家算法的核心是“试探分配后检查系统是否仍处于安全状态”
- 若分配后进入不安全状态，则暂不分配

## 3. 分页、虚拟内存与缺页

从 `[[操作系统/操作系统]]` 提炼出的核心点：

- 逻辑地址分页，物理内存分帧
- 页表建立页到帧的映射
- 按需调页：
  - 页不在内存时触发缺页中断
  - OS 将页从外存换入
- 虚拟内存的本质是地址空间抽象

### 什么是页表，为什么需要页表

- 页表是虚拟页到物理页框的映射表
- 每个进程通常都有自己的页表或等价的地址映射结构
- 引入页表后，程序看到的是连续的虚拟地址空间，而不是直接操作物理地址

为什么要有：

- 实现虚拟地址到物理地址的转换
- 支持进程隔离和内存保护
- 支持按页装入、共享库映射、缺页换入等机制

### 虚拟地址到物理地址怎么映射

- CPU 先产生虚拟地址
- MMU 负责地址转换
- 转换时先查快表（TLB）
- 若 TLB 未命中，再查页表
- 找到物理页框后，再加上页内偏移得到物理地址

一句话：

- `虚拟地址 = 页号 + 页内偏移`
- 通过页表把“页号”翻译成“物理页框号”，再和偏移拼出物理地址

### 缺页中断

- 当进程访问的虚拟页当前不在物理内存中时，会触发缺页异常/缺页中断
- 操作系统检查该访问是否合法
- 若合法，就从磁盘把对应页调入内存，更新页表后重新执行出错指令
- 若非法，则可能向进程发送异常信号并终止进程

和一般外中断的区别可简答为：

- 缺页是在指令执行过程中产生
- 同一条指令可能触发多次缺页
- 处理完后通常要回到“原指令”重新执行

### 常见页面置换算法

- OPT
  - 理论最优，淘汰未来最长时间不会访问的页
  - 主要用于分析比较
- FIFO
  - 先进先出
  - 简单但可能效果一般
- LRU
  - 最近最久未使用淘汰
  - 更贴近局部性原理
- Clock
  - LRU 的近似实现
  - 常通过访问位降低维护成本
- LFU
  - 淘汰访问频率最低的页

### 为什么需要虚拟内存

- 给每个进程提供更大、更连续的地址空间抽象
- 实现进程隔离，提升安全性
- 让物理内存不够时也能通过换页继续运行
- 便于按需加载、共享库映射和内存保护

面试常答一句：

- 虚拟内存的核心价值是“隔离 + 扩展 + 便于管理”

## 4. 内存碎片

从 `[[操作系统/操作系统]]` 和 `[[CPP&C/知识点整理/4.内存管理]]` 提炼出的核心点：

- 物理内存角度：
  - 分页缓解外部碎片
  - 但会引入内部碎片
- 虚拟内存/堆分配角度：
  - 不连续空洞会造成外部碎片
- 结构体对齐和分配粒度会造成内部碎片

## 6. I/O 多路复用

- `select`
  - 位图描述符集合
  - 每次调用都要拷贝和扫描全部 fd
  - 常见实现还有最大 fd 数限制
- `poll`
  - 用数组描述 fd，解决了 `select` 的位图限制
  - 但仍需要线性扫描
- `epoll`
  - Linux 下常见高并发方案
  - 把“关注集合维护”和“就绪事件返回”分开
  - 常见实现会用红黑树管理关注集合，用链表组织就绪事件

面试常答：

- `select/poll` 更像每次都问一遍谁准备好了
- `epoll` 更像先登记，谁就绪再通知

## 7. 典型锁

- 互斥锁
  - 同一时刻只允许一个线程进入临界区
- 读写锁
  - 允许多读单写
- 自旋锁
  - 获取不到锁时忙等，不立即睡眠
  - 适合持锁时间很短的场景
- 条件变量
  - 不是锁的替代品，常与互斥锁配合做等待/通知

面试稳妥说法：

- 互斥锁解决互斥访问
- 条件变量解决等待某个条件成立
- 二者经常配合使用

## 8. 外中断和异常

- 外中断
  - 由 CPU 执行指令之外的外部事件引起
  - 例如时钟中断、I/O 完成中断
- 异常
  - 由当前指令执行过程中内部事件引起
  - 例如除零、缺页、非法指令、越界访问

面试一句话：

- 外中断更像“外面来事了”
- 异常更像“当前这条指令自己出事了”

## 9. 操作系统对内存管理要做什么

- 内存分配与回收
- 地址转换
- 内存保护
- 逻辑扩充

更展开可答：

- 负责逻辑地址到物理地址映射
- 提供分页/分段等机制
- 提供进程隔离和访问权限控制
- 在内存不足时通过换页等方式做扩充

## 10. 进程同步与线程同步

### 常见同步目标

- 互斥
  - 同一时刻只允许一个执行流访问临界区
- 同步
  - 保证执行顺序和协作关系

### 常见进程同步方式

- 信号量
- 管程
- 消息传递配合同步语义
- 共享内存 + 同步原语

### 什么是管程

- 管程（Monitor）可以看成把“共享数据 + 对共享数据的操作 + 同步机制”封装到一起的高级同步抽象
- 任意时刻通常只允许一个执行流进入管程内部
- 这样可以把并发控制细节隐藏起来，降低直接操作信号量时出错的概率

一句话：

- 信号量更底层
- 管程更像“带同步语义的封装对象”

### 常见线程同步方式

- 互斥锁
- 条件变量
- 信号量
- 读写锁
- 自旋锁

## 11. 进程通信和线程通信

### 进程通信（Linux 常见）

- 管道
- 有名管道
- 消息队列
- 共享内存
- 信号
- socket
- 信号量

### `socket` 的基本组成

常见创建接口：

```c
int socket(int domain, int type, int protocol);
```

三部分分别表示：

- `domain`
  - 协议族 / 地址族，如 `AF_INET`、`AF_UNIX`
- `type`
  - socket 类型，如 `SOCK_STREAM`、`SOCK_DGRAM`
- `protocol`
  - 具体协议，常可填 `0` 让系统按前两者默认选择

### 线程通信

- 线程共享同一进程地址空间
- 更常通过共享变量 + 同步原语通信
- 常见同步工具：
  - 互斥锁
  - 条件变量
  - 信号量
  - 读写锁

### 线程同步的高频答案

- 互斥锁
- 读写锁
- 条件变量
- 信号量
- 自旋锁

### 为什么共享内存最快

- 共享内存让多个进程直接访问同一段物理内存映射
- 少了额外的数据拷贝
- 通常不需要像管道/消息队列那样频繁经过内核搬运数据

### 为什么要通信

- 数据传输
- 资源共享
- 事件通知
- 协作控制

### 常见方式优缺点速记

- 管道
  - 简单
  - 适合本机、亲缘关系较强场景
- 消息队列
  - 按消息组织，解耦较好
  - 容量受系统限制
- 共享内存
  - 最快
  - 但必须额外解决同步互斥
- 信号
  - 适合事件通知
  - 不适合承载复杂数据
- socket
  - 可跨主机
  - 通用性强，但开销通常更高

### 管道的实现原理

- 操作系统在内核中维护一块缓冲区
- 一个文件描述符负责读端，一个负责写端
- 匿名管道通常用于有亲缘关系的进程
- 命名管道（FIFO）可用于无亲缘关系进程

一句话：

- 管道本质上是“内核里的字节流缓冲区 + 两端文件描述符”

## 12. POSIX 常见线程锁

- `pthread_mutex_t`
  - 互斥锁
- `pthread_rwlock_t`
  - 读写锁
- `pthread_spinlock_t`
  - 自旋锁
- `pthread_cond_t`
  - 条件变量，严格说不是“锁”

面试里可补一句：

- 自旋锁适合持锁时间极短
- 互斥锁适合竞争时间较长、允许线程睡眠的场景

## 12.1 互斥锁、自旋锁、信号量

- 互斥锁
  - 获取不到锁时，线程通常睡眠等待
  - 适合临界区较长的场景
- 自旋锁
  - 获取不到锁时忙等，不立即让出 CPU
  - 适合持锁时间极短的场景
- 信号量
  - 本质是计数器
  - 既能做互斥，也能做同步

一句话：

- 互斥锁偏“互斥访问”
- 信号量既能“限资源数”，也能“做先后顺序”
- 自旋锁是用 CPU 换等待时间

### 互斥锁和读写锁的区别

- 互斥锁
  - 任意时刻只允许一个线程进入临界区
  - 不区分读和写
- 读写锁
  - 允许多个读者并发读
  - 但写者必须独占

适用场景：

- 读多写少时，读写锁通常更合适
- 读写竞争都重或临界区很短时，互斥锁往往更直接

### 信号量的作用

- 做互斥
- 做同步
- 控制并发资源数量

面试里可答：

- 二值信号量常可近似看成互斥用途
- 计数信号量更适合描述“有 N 个资源可用”

## 12.2 生产者消费者问题

这是经典同步模型：

- 生产者向缓冲区放数据
- 消费者从缓冲区取数据
- 既要互斥访问缓冲区
- 又要满足“满了不能放，空了不能取”

常见实现：

- 互斥锁 + 条件变量
- 或信号量 + 互斥锁

## 13. 线程如何回收

常见方式：

- `pthread_join`
  - 等待目标线程结束并回收其资源
- `pthread_detach`
  - 线程结束后由系统自动回收

## 注意

- 一个线程通常要么 `join`，要么 `detach`
- 不应两种方式混用

## 线程主动结束

- 可调用 `pthread_exit`
- 也可以从线程函数里 `return`

### 服务器创建线程时还能处理新连接吗

- 一般可以
- 常见做法是：
  - 主线程或监听线程继续 `accept`
  - 工作线程负责处理已接入连接
- 更现代的高并发模型里，也常直接用线程池、事件循环或 Reactor，而不是“每来一个连接就新建一个线程”

## 13.1 三种线程实现模型

- 用户线程
  - 在线程库中实现
  - 切换通常不必每次陷入内核
  - 但一个阻塞系统调用可能拖住整个用户线程调度实体
- 内核线程
  - 由内核直接调度
  - 阻塞和调度更自然
  - 切换和管理开销更高
- 轻量级进程（LWP）
  - 可理解为连接用户线程与内核线程的执行实体

面试稳妥说法：

- 用户线程切换轻
- 内核线程调度强
- LWP 是很多系统里的折中桥梁

## 13.2 线程的优缺点

优点：

- 并发度高
- 创建和切换通常比进程轻
- 同进程内共享数据方便

缺点：

- 共享地址空间，出错更容易互相影响
- 同步、加锁、调试更麻烦
- 数据竞争、死锁等问题更常见

### 为什么有了进程还要有线程

- 进程切换和创建开销更大
- 多个执行单元若要频繁共享数据，线程比进程更方便
- 线程让“同一进程内的多个任务并发执行”成本更低

一句话：

- 线程是为了在保持并发能力的同时，降低进程级并发的开销

## 13.3 为什么线程开销通常比进程小

- 线程共享所属进程的大部分资源
- 不必像进程那样切换完整地址空间
- 通信可以直接走共享内存，不必总是跨进程搬运数据

### 为什么进程切换通常比线程慢

- 进程切换往往涉及地址空间切换
- 页表切换后，TLB 相关缓存命中也可能受影响
- 线程切换通常不需要切换整个进程地址空间

一句话：

- 进程切换多了“地址空间/页表”这一层成本

### 线程栈

- 每个线程通常都有自己独立的线程栈
- 这些线程栈位于同一进程地址空间中的不同区域
- 默认栈大小可通过线程属性调整，不要死背某个固定地址

### 单核机器写多线程程序还要不要加锁

- 仍然要
- 因为单核不代表没有并发切换
- 抢占式调度下，线程会在任意时间片边界被切走
- 只要共享数据存在并发访问，就仍然需要同步保护

## 13.4 线程安全

- 如果多个线程并发访问同一段代码或同一份数据时，程序结果仍然正确、可预期，就可以说它是线程安全的

常见实现手段：

- 不共享可变状态
- 加锁保护共享资源
- 使用原子操作
- 使用线程局部存储

## 14. 覆盖与交换

### 覆盖

- 主要发生在同一进程内部
- 把不会同时运行的程序段复用同一块内存区域
- 适合早期内存极紧张场景

### 交换

- 主要在不同进程之间进行
- 把整个进程或较大部分换出到外存，再把别的进程换入

#### 被换出的进程放在哪里

- 通常放在外存的交换区（swap space）或对应的交换文件区域

#### 哪些进程更可能被优先换出

- 阻塞进程
- 优先级较低的进程
- 一些驻留时间较长、短期内不急需运行的进程

## 注意

- 进程的 PCB 等核心管理信息通常不会整体跟着一起换出

### 区别

- 覆盖：同一进程不同部分共享内存
- 交换：不同进程之间做换入换出
- 缺页换入换出通常是“页”为单位，比整进程交换更细粒度

## 15. 局部性原理

### 时间局部性

- 刚访问过的数据或指令，之后很可能再次访问

### 空间局部性

- 访问某个地址后，附近地址也很可能被访问

## 为什么重要

- Cache、分页、预取、页面置换算法都高度依赖局部性原理

## 15.1 并发和并行

- 并发
  - 宏观上一段时间内多个任务都在推进
  - 不要求同一时刻真的同时执行
- 并行
  - 同一时刻多个任务真正同时执行
  - 更依赖多核、多处理器等硬件支持

## 一句话

- 并发强调“交替推进”
- 并行强调“同时执行”

## 16. 进程组、作业、会话

### 进程组

- 一组相关进程的集合
- 通常有一个组长进程
- 常用于信号管理和作业控制

### 作业

- Shell 视角下的一组相关进程
- 可分前台作业和后台作业

### 会话

- 一个或多个进程组的集合
- 通常对应一个控制终端上下文

## 一句话

- 会话最大
- 下面可以有多个进程组
- Shell 里的作业通常和进程组强相关

## 17. 进程终止方式

- `main` 正常返回
- 调用 `exit`
- 调用 `_exit`
- 调用 `abort`
- 收到终止信号，例如 `SIGINT`、`SIGTERM`、`SIGKILL`

## 18. 系统调用是什么

- 系统调用是用户态程序向内核请求服务的接口

常见类别：

- 进程控制
- 文件操作
- 设备操作
- 内存管理
- 进程通信

### 系统调用和库函数的区别

- 系统调用
  - 由内核提供
  - 执行时进入内核态
  - 更靠近底层资源
- 库函数
  - 由运行库提供
  - 常常是对系统调用的封装
  - 也可能完全不触发系统调用

一句话：

- 库函数是更好用的包装
- 系统调用才是真正向内核请求服务

## 18.2 为什么要区分用户态和内核态

- 保护关键内核数据和硬件资源
- 防止普通程序随意执行特权指令
- 让操作系统通过系统调用接口统一管理资源访问

## 18.1 用户态和内核态

- 用户态
  - 权限受限
  - 不能直接执行特权指令
- 内核态
  - 权限最高
  - 可访问关键硬件和内核数据

### 什么时候会进入内核态

- 系统调用
- 异常
- 中断

### 用户态陷入内核态的典型场景

- 调用 `read`、`write`、`open` 这类系统调用
- 发生缺页、除零、非法指令等异常
- 收到时钟中断、网卡中断、磁盘中断等设备中断

## 18.3 sleep 和 wait 的区别

- `sleep`
  - 让当前线程/进程挂起一段时间
  - 时间到了继续运行
- `wait` / `waitpid`
  - 父进程回收子进程退出状态
  - 解决僵尸进程问题

一句话：

- `sleep` 是“休眠”
- `wait` 是“回收子进程”

## 19. 终端退出后进程会怎样

- 终端退出时，相关前台会话里的进程通常会收到 `SIGHUP`
- 若程序没有特殊处理，往往会随终端退出而结束

### 常见信号

- `SIGHUP`
  - 终端挂起，常见于控制终端断开
- `SIGINT`
  - 中断信号，常见于 `Ctrl+C`
- `SIGTERM`
  - 请求进程正常终止，`kill` 默认发送它
- `SIGKILL`
  - 强制杀死，不能被捕获、阻塞或忽略
- `SIGCHLD`
  - 子进程状态变化时发给父进程
- `SIGSEGV`
  - 非法内存访问，常见于段错误
- `SIGABRT`
  - 调用 `abort()` 时触发

## 19.1 Linux 常见进程状态

- `R`
  - Running / Runnable，可运行状态
- `S`
  - 可中断睡眠
- `D`
  - 不可中断睡眠，常见于等待 I/O
- `T`
  - 停止或被跟踪
- `Z`
  - 僵尸进程
- `X`
  - 退出态，较少直接看到

面试里通常先抓三类：

- 运行态
- 就绪/等待态
- 阻塞/睡眠态

## 19.2 进程的五状态模型

- 新建
- 就绪
- 运行
- 阻塞
- 终止

一句话：

- 操作系统核心就是在这些状态之间调度和切换进程

## 19.3 fork 的作用

- `fork` 用来创建子进程
- 调用一次，返回两次：
  - 父进程返回子进程 PID
  - 子进程返回 `0`
  - 失败返回 `-1`

### fork 之后父子进程的关系

- 父子进程拥有不同的虚拟地址空间
- 初始时很多页面会借助写时复制（COW）共享同一份物理页
- 当某一方写入时，内核再为其分配新的物理页

稳妥回答：

- 不要说“fork 后立刻完整复制整个堆和栈”
- 更准确是“现代系统通常通过写时复制推迟真正的数据页复制”

## 19.4 守护进程、孤儿进程、僵尸进程

- 守护进程
  - 长期运行在后台、通常脱离终端的服务进程
- 孤儿进程
  - 父进程先退出，子进程还在运行
  - 后续通常由 `init/systemd` 接管
- 僵尸进程
  - 子进程已退出，但父进程还没回收其退出状态
  - 会残留进程表项等内核管理信息

### 如何避免僵尸进程

- 父进程及时调用 `wait` / `waitpid`
- 正确处理 `SIGCHLD`
- 不要长期放着退出的子进程不回收

## 20. 如何让进程后台运行

- 命令后加 `&`
- 也可配合 `nohup`、`setsid` 等方式减少终端退出影响

## 21. 抖动 / 颠簸

- 页面刚换出又要换入，刚换入又很快换出
- 说明页频繁调度，系统大量时间耗在换页上

常见原因：

- 分配给进程的物理页框太少
- 活跃工作集大于可用物理内存

## 22. malloc 的实现原理

面试里常见回答：

- 小块内存分配常通过扩展堆区来完成
- 大块分配常可能直接走 `mmap`
- 具体阈值是实现细节，不建议死背某个固定数字

更稳妥地说：

- `malloc` 先向分配器申请
- 分配器再通过 `brk/sbrk` 扩展堆，或通过 `mmap` 映射匿名内存
- 真正首次访问页面时，往往还会配合缺页异常按需建立物理页映射

### 为什么它通常不会“一申请就立刻拿到所有物理页”

- 因为现代系统普遍采用虚拟内存和按需分配
- 先建立虚拟地址范围，真正访问时再逐页映射物理内存

## 22.3 mmap 的原理和使用场景

- `mmap` 可以把文件或匿名内存映射到进程虚拟地址空间
- 建立映射后，进程可像访问普通内存一样访问这段区域
- 对文件映射来说，内核会负责页面调入、脏页回写等工作

常见场景：

- 大文件高效读写
- 共享内存 IPC
- 动态库和文件映射
- 大块匿名内存分配

一句话：

- `mmap` 的核心是“把文件/匿名内存映射成一段可直接访问的虚拟地址空间”

## 22.4 DMA 是什么

- DMA = `Direct Memory Access`，直接内存访问
- 它的核心不是“怎么映射内存”，而是“谁来搬数据”
- CPU 先配置源地址、目标地址、传输长度
- 后续由 DMA 控制器直接完成外设和内存之间，或者内存和内存之间的数据搬运

好处：

- 降低 CPU 参与重复搬运的开销
- 更适合大块、连续、流式的数据传输
- CPU 可以在搬运时并行做别的计算或调度

常见场景：

- 网卡把数据写入内存
- 摄像头把图像帧写入 buffer
- 串口 / SPI / I2C 持续收发
- GPU / NPU / 存储设备做数据传输

注意：

- DMA 不是“完全不要 CPU”
- CPU 仍然负责初始化、同步、处理中断和错误恢复

### DMA 能不能并行搬

- 一般可以，但取决于硬件是否有多个 DMA channel 或 copy engine
- 即使逻辑上能并行，也会受到总线仲裁和内存带宽限制
- 多个 DMA、CPU、外设一起访存时，常见瓶颈往往不是控制器数量，而是共享带宽

一句话：

- DMA 解决的是“谁来搬数据”，而且通常支持一定程度的并行搬运，但真实性能受带宽限制

## 22.2 内存池

- 内存池是预先申请一大块内存，再按需切分给小对象使用的策略
- 目的是减少频繁 `malloc/new` 带来的分配开销和内存碎片

适合场景：

- 小对象很多
- 生命周期模式比较稳定
- 高频申请和释放

优点：

- 分配释放更快
- 减少碎片
- 便于统一管理

缺点：

- 实现更复杂
- 预留过多会浪费内存

## 22.1 32 位系统能访问 4GB 以上内存吗

- 普通 32 位虚拟地址空间理论上线是 `2^32 = 4GB`
- 所以单个 32 位地址本身只能直接表示 4GB 范围
- 但通过 PAE 等扩展技术，32 位系统在某些场景下可以管理 4GB 以上物理内存

面试稳妥说法：

- “单进程虚拟地址空间通常仍受 32 位限制”
- “系统层面通过扩展机制可能管理超过 4GB 的物理内存”

## 23. 快表（TLB）

- 快表本质上是页表项的高速缓存
- 用来加速虚拟地址到物理地址的转换

### 有快表和没快表的区别

- 没快表
  - 地址转换时更频繁访问内存中的页表
- 有快表
  - 命中时可更快得到页表映射
  - 显著减少地址转换开销

## 24. 进程调度算法

- 先来先服务（FCFS）
- 短作业优先（SJF）
- 优先级调度
- 时间片轮转
- 多级反馈队列

面试常答：

- 轮转强调公平性
- 短作业优先平均周转时间通常更好
- 多级反馈队列是综合型高频答案

## 25. 多进程和多线程怎么选

- 多线程
  - 地址空间共享
  - 通信成本低
  - 切换通常更轻
  - 适合同一进程内高相关任务协作
- 多进程
  - 隔离性更好
  - 一个进程崩溃不一定拖垮其他进程
  - 更适合安全隔离、服务拆分、分布式扩展

## 工程上常见选择

- 任务强相关、共享数据多：偏多线程
- 需要强隔离、稳定性、跨机扩展：偏多进程
- 实际项目里常是“多进程 + 多线程”混合

## 25.1 进程、线程、协程的区别

- 进程
  - 资源分配基本单位
  - 地址空间独立，隔离性最好
- 线程
  - CPU 调度基本单位
  - 同一进程内共享地址空间，通信成本低
- 协程
  - 用户态更轻量的执行单元
  - 切换通常由用户态代码控制，切换成本更低

工程上可这样答：

- 强隔离、服务拆分：偏进程
- 同进程协作和共享数据：偏线程
- 大量 I/O 等待、希望减少内核态切换：可考虑协程

### 为什么引入协程

- 降低线程切换和调度开销
- 提高高并发 I/O 场景下的资源利用率
- 用较少线程承载大量任务

注意：

- 协程不是操作系统调度的“抢占式线程”
- 它通常依赖语言运行时或框架调度

## 25.2 线程池是如何实现的

核心思想：

- 预先创建一批工作线程
- 主线程不断把任务放入任务队列
- 工作线程从任务队列取任务执行
- 线程复用，避免频繁创建和销毁线程

典型组成：

- 任务队列
- 工作线程集合
- 互斥锁 / 条件变量
- 拒绝策略或关闭逻辑

优点：

- 降低线程创建销毁成本
- 控制并发度
- 提高吞吐和稳定性

### 为什么要创建线程池

- 频繁创建和销毁线程代价不低
- 线程池可以复用线程，降低资源消耗
- 也方便统一做限流、排队、拒绝策略和监控

### 线程池线程数量怎么确定

经验答法：

- CPU 密集型：
  - 线程数通常接近 CPU 核数，常见答 `CPU 核数` 或 `CPU 核数 + 1`
- I/O 密集型：
  - 线程数通常可以更多
  - 常见经验是 `2 * CPU 核数` 起步，再结合等待时间调参

更稳妥的说法：

- 最终要结合任务性质、CPU 利用率、阻塞比例、上下文切换成本压测确定

### 核心线程和普通线程

- 核心线程
  - 通常长期保留在线程池里
- 非核心线程
  - 常在任务高峰期临时扩容使用
  - 空闲一段时间后可回收

## 26. 虚拟化技术

- 时间复用
  - 多个执行流分时使用 CPU
- 空间复用
  - 多个执行流复用内存、磁盘等资源的逻辑抽象

常见理解：

- 多进程/多线程体现了时间复用
- 虚拟内存体现了空间复用

## 27. 服务器高并发的常见思路

- 静态资源与动态业务分离
- 客户端缓存 / CDN
- 负载均衡
- 集群与分布式拆分
- 反向代理
- 异步化、消息队列、削峰填谷
- 连接复用与 I/O 多路复用
- 缓存（本地缓存 / Redis 等）

### 多进程、多线程和 I/O 复用的区别

- 多进程
  - 强隔离
  - 稳定性好
  - 开销较大
- 多线程
  - 共享内存方便
  - 切换成本相对更低
  - 但同步复杂
- I/O 复用
  - 用少量线程管理大量连接
  - 更适合高并发网络 I/O 场景

一句话：

- 多进程/多线程偏“执行单元组织方式”
- I/O 复用偏“等待多个 I/O 事件的机制”

### 零拷贝

- 零拷贝的目标是减少数据在用户态和内核态之间的拷贝次数，以及减少上下文切换
- 常见做法包括 `sendfile`、`mmap + write` 等

好处：

- 降低 CPU 开销
- 降低内存带宽消耗
- 提高文件传输和网络发送效率

注意：

- “零拷贝”通常不是字面意义上的完全零次复制
- 更准确是“尽量避免不必要的用户态参与拷贝”

### 重点：DMA、零拷贝、mmap 的区别

这一组特别容易混，建议直接这样记：

- `DMA`：谁来搬数据
- `零拷贝`：尽量少搬甚至不搬
- `mmap`：怎么把数据映射到进程地址空间

更完整地说：

- `DMA` 是硬件搬运机制，重点是“搬运工作由专门控制器做，不是 CPU 一条条 copy”
- 零拷贝是一类性能优化目标，重点是“减少不必要的数据副本和用户态参与”
- `mmap` 是操作系统提供的内存映射接口，重点是“把文件或设备缓冲区映射到虚拟地址空间中”

三者关系：

- `DMA` 可以作为零拷贝的一部分
- `mmap` 也可以作为零拷贝的一部分
- 零拷贝经常会组合使用 `DMA + mmap + sendfile + 共享 buffer`
- 但三者不是同义词，也不在同一层

### 一套面试里很稳的例子

比如摄像头采集一帧图像给用户程序：

- 摄像头通过 `DMA` 把数据写入内存 buffer
- 驱动通过 `mmap` 把这块 buffer 映射到用户态
- 用户程序直接访问这块数据，而不是再 `read` 一份新副本

这里：

- `DMA` 负责搬运
- `mmap` 负责映射
- 最终效果接近零拷贝

### 面试速记

> DMA 解决“谁来搬”，mmap 解决“怎么访问”，零拷贝解决“能不能少拷几次”。

### Jetson Orin Nano（UMA）视角下怎么理解

- Jetson Orin Nano 是 `UMA` 架构，CPU 和 GPU 共享同一套物理内存
- 所以它不像离散显卡那样，要频繁在“主存”和“独立显存”之间来回拷贝
- 但这不代表数据搬运没有成本，因为 CPU、GPU、DMA、外设都在抢同一套内存带宽

放到这个平台上看：

- `DMA` 仍然负责让外设或存储异步把数据搬进内存，减少 CPU 参与
- `零拷贝` 的价值会更大，因为它减少的是全系统共享带宽上的冗余 copy
- `mmap` 或共享 buffer 的意义，是尽量让不同模块直接访问同一份数据，而不是再复制一份

一句话：

> 在 Jetson Orin Nano 这类 UMA 平台上，零拷贝特别重要，因为最怕的不是“算一下”，而是“多搬一下”。

### 同步 / 异步，阻塞 / 非阻塞

- 同步 / 异步
  - 更偏向“结果完成后，调用方怎么获知”
- 阻塞 / 非阻塞
  - 更偏向“调用发出后，当前线程会不会立刻被挂起”

面试稳妥说法：

- 同步：调用方通常要主动等待结果
- 异步：结果完成后通过通知、回调等方式告诉调用方
- 阻塞：当前线程会被挂起等待
- 非阻塞：调用立即返回，结果稍后再看

### epoll、select、poll 的区别

- `select`
  - 有 fd 数量上限
  - 每次都要拷贝和扫描 fd 集合
- `poll`
  - 解决了 `select` 的位图上限问题
  - 但本质上仍需线性扫描
- `epoll`
  - 把注册和事件通知分离
  - 活跃连接很多时通常更高效

为什么 `epoll` 常被认为更高效：

- 不必每次都重新拷贝整个关注集合
- 不必像 `select/poll` 那样每轮线性扫描全部 fd
- 更适合大量连接、少量活跃的场景

### 多路 I/O 复用技术有哪些

- `select`
- `poll`
- `epoll`

## 30. 一个程序从开始运行到结束的完整过程

如果面试官问“从代码到运行，再到退出”，可以按这条线答：

- 编写源码
- 预处理
  - 展开宏
  - 处理条件编译
  - 展开头文件
  - 去掉注释
- 编译
  - 词法分析
  - 语法分析
  - 语义分析
  - 优化
  - 生成汇编
- 汇编
  - 把汇编代码转成目标文件
- 链接
  - 把多个目标文件和库链接成可执行文件
  - 可能是静态链接，也可能是动态链接
- 装载
  - 操作系统创建进程
  - 映射代码段、数据段、共享库
  - 建立虚拟地址空间
- 运行
  - 从入口点开始执行
  - 发生系统调用、缺页中断、调度切换等
- 结束
  - 正常返回或异常终止
  - 操作系统回收进程资源

一句话速记：

- 源码先经过“预处理、编译、汇编、链接”变成可执行文件
- 运行时再由操作系统“装载、调度、回收”

## 31. GDB 调试

- 常见命令和 `core dump` 调试我单独整理到：
  - `[[操作系统/gdb]]`

面试里可先简答：

- 常用有 `break`、`run`、`next`、`step`、`continue`、`print`、`bt`
- 条件断点就是“满足指定条件时才停下”
- 多进程/多线程调试要结合 `follow-fork-mode`、`info inferiors`、`info threads`

## 32. 软件测试常见缩写

- `UT`
  - Unit Test，单元测试
- `IT`
  - Integration Test，集成测试
- `ST`
  - System Test，系统测试
- `UAT`
  - User Acceptance Test，用户验收测试

## 22. 栈和堆上建对象哪个快

- 一般来说栈上更快

原因：

- 栈分配通常由编译器和栈指针直接调整完成
- 堆分配往往需要进入分配器查找合适空闲块并维护元数据

## 注意

- 这是一般规律，不是说“堆上一定非常慢”
- 真实差异还取决于分配器实现和访问模式
