## 5. BERT、T5 与常见架构区别

### 5.1 BERT

- BERT 是纯 Encoder 架构。
- 主要特点是双向注意力，适合做理解类任务。
- 常见任务：
  - 分类
  - 命名实体识别
  - 文本匹配
  - 抽取式问答

### 5.2 T5

- T5 是 Encoder-Decoder 架构。
- 它把很多 NLP 任务都统一成文本到文本任务。
- 更适合生成类任务，比如：
  - 翻译
  - 摘要
  - 改写
  - 问答生成

### 5.3 Decoder-Only 和 Encoder-Decoder

- Decoder-Only
  - 更适合自回归生成
  - 结构简单
  - 很适合大语言模型
- Encoder-Decoder
  - 更适合输入输出长度不同的序列到序列任务
  - 更容易建模输入和输出之间的映射

面试简答：

- BERT 偏理解，T5 偏生成。
- Decoder-Only 更适合通用大模型生成，Encoder-Decoder 更适合翻译和摘要这类 seq2seq 任务。

## 6. LoRA、Adapter 和参数高效微调

### 6.1 LoRA 原理

- LoRA 的核心思想是冻结原始大权重 `W`，只训练一个低秩增量：
- `W' = W + BA`
- 其中：
  - `A` 和 `B` 是低秩矩阵
  - rank 通常远小于原矩阵维度

### 6.2 为什么这么做

- 大模型微调时，全量更新参数成本太高。
- 低秩分解假设任务相关更新主要落在一个低维子空间里。
- 这样可以大幅减少可训练参数、显存和通信开销。

### 6.3 LoRA 初始化

- 常见做法：
  - `A` 用较小随机值初始化
  - `B` 初始化为 0

### 6.4 为什么这样初始化

- 训练开始时 `BA` 近似为 0，所以 `W'` 一开始几乎等于原始 `W`。
- 这样不会破坏预训练模型的初始能力。
- `A` 用小随机值可以避免训练初期更新过猛，更稳定。
- `B` 为 0 可以保证刚开始 LoRA 分支几乎不引入扰动。

### 6.5 LoRA 和 Adapter 哪个更好

- LoRA
  - 参数更省
  - 推理时更容易融合回线性层
  - 对算子图侵入更小
- Adapter
  - 在层间显式插入小瓶颈网络
  - 表达能力更强一些
  - 参数和计算开销通常略高

面试简答：

- 如果更看重参数效率和部署方便，LoRA 通常更合适。
- 如果任务更复杂、希望适配能力更强，Adapter 可能更好。

## 7. Llama、GQA 和 MLA

### 7.1 为什么要 GQA

- GQA 是 Grouped Query Attention。
- 它让多个 Q 头共享同一组 K/V 头。
- 这样做的目的主要是减少 KV Cache 体积和带宽压力。

### 7.2 GQA 的直观理解

- Q 头数可以很多。
- K/V 头数更少。
- 一个 K/V 头服务一组 Q 头。
- 所以精度通常接近 MHA，但显存和访存成本更低。

### 7.3 Llama 系列常见演进点

- 参数规模不断扩大
- 上下文长度不断提升
- 注意力机制更偏向 GQA
- 分词器和训练数据持续更新

### 7.4 MLA

- MLA 可以理解为更节省 KV 表示和访存的一类注意力方案。
- 它的核心目标也是降低长上下文下的存储和带宽开销。
- 面试里如果被追问，可以先从“减少 KV 存储和读取成本”这个方向讲。

## 8. 幻觉、思维链和可靠性

### 8.1 思维链

- 思维链就是让模型把复杂问题拆成多步中间推理步骤再回答。
- 对复杂数学、逻辑、多跳推理任务经常有帮助。

### 8.2 幻觉怎么缓解

- RAG
- RLHF
- 更好的监督数据
- 多模态或外部知识辅助
- 降低采样温度
- 在高可靠任务中使用更严格的检索和校验

### 8.3 生成参数的影响

- 温度低一些，通常更稳定，幻觉更少。
- 温度高一些，多样性更强，但风险也更高。
- beam search 更大有时能提高稳定性，但也会增加延迟和重复风险。

## 9. 从 Hugging Face 下载的模型通常包含什么

这里的 “hg” 一般就是指 Hugging Face / HF 模型目录。

### 9.1 最常见的几类文件

一个 Hugging Face 模型目录，通常会包含这几类内容：

| 类别 | 常见文件 | 作用 |
| --- | --- | --- |
| 模型结构配置 | `config.json` | 定义层数、hidden size、head 数、rope 参数、vocab size 等 |
| 权重文件 | `model.safetensors`、`model-00001-of-000xx.safetensors`、`pytorch_model.bin` | 存模型参数 |
| 分片索引 | `model.safetensors.index.json`、`pytorch_model.bin.index.json` | 告诉你每层权重在哪个分片文件里 |
| tokenizer 文件 | `tokenizer.json`、`tokenizer.model`、`vocab.json`、`merges.txt` | 定义分词规则和词表 |
| tokenizer 配置 | `tokenizer_config.json`、`special_tokens_map.json` | 定义 BOS/EOS/PAD 等特殊 token 行为 |
| 生成配置 | `generation_config.json` | 默认采样参数，如 `bos_token_id`、`eos_token_id`、`max_length` |
| 预处理配置 | `preprocessor_config.json`、`processor_config.json` | 多模态模型常见，定义图像/音频预处理 |
| 说明文件 | `README.md`、`model_card` | 模型介绍、用法、许可、训练来源 |

### 9.2 对纯文本大模型来说，最关键的是什么

如果是普通 LLM，真正最关键的通常就三部分：

1. `config.json`
2. 权重文件：`*.safetensors` 或 `*.bin`
3. tokenizer 相关文件

可以理解成：

- `config.json` 决定“模型长什么样”
- 权重文件决定“模型学到了什么”
- tokenizer 文件决定“文本怎么变 token”

### 9.3 为什么有时候会有很多个权重文件

因为模型太大，单文件不方便存储和下载，所以会切成多个 shard，例如：

```text
model-00001-of-00008.safetensors
model-00002-of-00008.safetensors
...
model.safetensors.index.json
```

这时候：

- 分片文件里是真正的参数
- `index.json` 负责记录每个参数张量落在哪个分片里

### 9.4 为什么现在更常见 `safetensors`

因为它相对 `pytorch_model.bin` 更安全，也更适合部署：

- 不依赖 pickle 反序列化执行
- 读取更直接
- 更适合推理框架加载

所以现在很多新模型默认优先发：

- `*.safetensors`

### 9.5 如果面试里问一句“HF 下载的模型包含什么”

可以直接答：

> Hugging Face 下载下来的模型目录通常包含模型配置、权重文件、tokenizer 文件，以及一些生成和预处理配置。最核心的是 `config.json`、权重文件和 tokenizer 相关文件；如果模型很大，权重还会被切成多个分片，并配一个 `index.json` 做映射。

## 10. Qwen3 的词表一般是怎么来的？不同大小模型词表一样大吗

### 10.1 词表一般怎么来

像 Qwen3 这类大模型，词表通常不是手工写出来的，而是：

1. 先收集大规模多语种训练语料
2. 在语料上训练 tokenizer
3. tokenizer 根据统计规律把常见字词、子词、符号、空格模式等压缩成一套 token vocabulary

也就是说，词表本质上是：

> 在大规模语料上学出来的一套“文本切分单位”。

常见做法更接近：

- BPE
- SentencePiece
- Unigram
- 或者它们的工程变体

对中文、英文、数字、标点、代码、空格模式都会一起考虑。

### 10.2 为什么不是按“一个汉字一个 token”来做

因为真实语料里需要同时兼顾：

- 中文
- 英文
- 数字
- 标点
- 代码
- 多语种混合

如果词表设计太粗：

- token 数太多
- 序列变长
- 训练和推理成本更高

如果词表设计太细：

- 词表过大
- embedding 和 LM head 成本上升

所以词表大小本质上是在平衡：

- 压缩率
- 泛化能力
- 参数量
- 推理效率

### 10.3 不同大小的模型词表一样大吗

很多同一系列模型里：

- 7B
- 14B
- 32B

往往会共用同一套 tokenizer 和词表。

原因很简单：

- 这样同系列模型输入输出语义一致
- 方便蒸馏、微调、部署和模型切换
- 也方便共享数据处理流水线

所以：

> 同一模型家族里，不同参数规模的模型，词表通常是一样大的。

### 10.4 但也不是绝对

如果出现下面这些情况，词表也可能变化：

- 新版本 tokenizer 重训了
- 增加了多语种能力
- 增强了代码能力
- 增加了特殊 token
- 从旧架构迁移到新架构

所以更准确的说法是：

- 同一代、同一家族、同一 tokenizer 方案下，通常词表一致
- 跨代版本、跨用途版本时，词表可能变化

### 10.5 面试可直接答

> Qwen3 这类模型的词表一般是在大规模训练语料上通过 tokenizer 训练得到的，不是人工枚举的。它会综合中文、英文、数字、标点、代码等统计规律，形成一套 token vocabulary。对于同一系列不同大小的模型，通常会共用同一套词表和 tokenizer，这样输入输出语义一致、部署和训练链路也更统一；但如果 tokenizer 版本升级或模型用途变化，词表也可能调整。
