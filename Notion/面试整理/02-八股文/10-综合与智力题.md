# 综合与智力题

## 1. C/C++ 语言小题

### `int (*s[10])(int)` 中 `s` 是什么

- `s` 是一个数组
- 数组里有 10 个元素
- 每个元素都是“函数指针”
- 这些函数指针指向形如 `int func(int)` 的函数

一句话：

- `s` 是“函数指针数组”

### `strcpy` 函数

函数原型常见为：

```c
char *strcpy(char *dest, const char *src);
```

特点：

- 按 `'\0'` 结尾复制
- 不检查目标缓冲区是否足够大
- 返回 `dest`，方便链式调用

### `strcpy` 和 `memcpy` 的区别

- `strcpy`
  - 面向 C 风格字符串
  - 以 `'\0'` 为结束标记
- `memcpy`
  - 面向任意原始内存块
  - 需要显式指定拷贝字节数

### `strcpy` 和 `strncpy` 的区别

- `strcpy`
  - 不限制长度
  - 更容易越界
- `strncpy`
  - 会限制最多拷贝的字节数
  - 但如果源串太长，结果未必自动补 `'\0'`

稳妥回答：

- `strncpy` 相对更可控
- 但不能简单说“绝对安全”，因为它也容易留下未终止字符串的问题

### 实现一个内存安全的 `memcpy`

面试里常见考点不是“照着写库函数”，而是你是否知道：

- 参数判空
- 是否返回目标地址
- 长度如何处理
- 关键点：若源和目标区域重叠，不能直接按普通 `memcpy` 思路写

稳妥回答：

- 真正“内存安全”的重叠拷贝应更接近 `memmove` 语义
- 可根据地址关系决定正向拷贝还是逆向拷贝

### `copy()` 和 `copy_backward()`

- `copy`
  - 从前往后复制
- `copy_backward`
  - 从后往前复制

适用场景：

- 当区间可能重叠，且目标区间在源区间右侧时，常考虑逆向拷贝

### `static_cast` 比 C 风格强转好在哪

- 可读性更强
- 更容易全局搜索和审查
- 语义更明确，能看出程序员意图
- 类型系统约束更强，很多不合理转换会直接报错

### 一致性哈希

- 一致性哈希的核心目标是：
  - 在增删节点时，尽量减少已有 key 的迁移
- 常见做法：
  - 把节点和数据都映射到哈希环上
  - 数据沿顺时针找到第一个节点归属

常见应用：

- 分布式缓存
- 分库分表
- 负载均衡

### 用 C++ 设计一个不能被继承的类

现代 C++ 最直接方式：

```cpp
class A final {
};
```

面试补一句：

- C++11 之后优先答 `final`
- 老写法才会通过私有构造/虚继承等技巧限制继承

### 不用额外空间交换两个数

常见答法：

- 加减法
- 异或法

更稳妥补充：

- 实际工程里直接用 `std::swap` 最清晰
- 加减法可能溢出
- 异或法可读性较差，也要注意别对同一变量自操作

## 2. 智力与经典思维题

### 100 层楼和 2 个玻璃球

核心思路：

- 不是简单二分，因为球碎了后代价很大
- 应让第一次尝试的楼层差逐步递减
- 例如第一次扔 `14` 层，之后加 `13`、`12`、`11` ...
- 这样最坏情况下总次数最小

### 36 匹马，6 个跑道，找最快前 3

高频结论：

- 最少通常答 `8` 次

### 为什么二叉树前序 + 后序不能唯一确定一棵树

- 因为缺少“根左右子树边界”的唯一划分信息
- 尤其当某个节点只有一个孩子时，无法判断它是左孩子还是右孩子

一句话：

- 前序和后序都能知道“根”，但不够确定子树结构

## 3. 海量数据题

### 10 亿个数排序

常见思路：

- 外排序
- 先分块读入内存排序
- 写回多个有序临时文件
- 最后做多路归并

### 100 亿个数据找最大的 1000 个

高频思路：

- 分治 / 分区
- 每个分块内部维护一个大小为 1000 的小顶堆
- 最后再把各块结果汇总，继续用小顶堆筛出全局前 1000

### 10 亿搜索词日志，求 Top 10 热词

常见思路：

1. 先按哈希把大文件拆成多个小文件
2. 保证同一个关键词一定落在同一小文件里
3. 每个小文件内部做词频统计
4. 每个小文件求局部 Top K
5. 最后汇总做全局 Top 10

### 10G 整数找中位数，内存限制 2G

常见思路有两类：

- 如果能分桶：
  - 先按值域分桶统计
  - 定位中位数所在桶
  - 再对该桶细化统计
- 如果数据可多轮扫描：
  - 可借鉴快速选择 / 分治思想

面试重点：

- 不一定非要写代码
- 要说清楚“值域换空间”或“多轮扫描换内存”

## 4. 系统设计题

### 微信抢红包思路

高频关注点：

- 库存扣减
- 防超发
- 幂等
- 并发控制
- 金额分配算法
- 入账与对账

稳妥回答框架：

1. 红包主数据放缓存，减少数据库压力
2. 抢红包先做资格判断：
   - 是否还有红包
   - 用户是否已抢过
3. 金额分配可采用“二倍均值法”等近似随机方案
4. 扣减个数和金额要保证原子性
5. 抢到后异步入库、异步入账
6. 最后通过对账系统兜底一致性

注意：

- 不能简单说“微信群人数有限，所以没有高并发”
- 大群、热点活动、集中时刻一样可能出现高并发

### 秒杀系统

常见问题：

- 瞬时流量过大
- 库存超卖
- 热点 key
- 数据库被打垮
- 黄牛刷单

高频设计思路：

- 前置限流
- 静态化页面 / CDN
- 请求削峰：消息队列
- 库存预扣放缓存
- 原子扣减避免超卖
- 异步下单
- 防重复提交
- 熔断降级

一句话：

- 秒杀系统的核心不是“算得快”，而是“扛得住高峰，还不能超卖”

## 5. 分布式与认证题

### 分布式多个机器生成 ID，如何保证不重复

常见思路：

- 数据库自增 ID
- Redis 自增
- UUID
- 雪花算法（Snowflake）

面试高频回答：

- 真正工程里更常答“时间戳 + 机器号 + 序列号”
- 也就是类似 Snowflake 的方案

核心点：

- 同一时刻同一机器上靠序列号区分
- 不同机器靠机器号区分
- 不同时刻靠时间戳区分

### 扫码登录如何实现

典型角色：

- PC 端
- 手机端
- 服务端

典型流程：

1. PC 端请求服务端生成二维码标识
2. 服务端返回二维码 ID，并记录状态、过期时间
3. PC 展示二维码
4. 手机扫码后把二维码 ID 和自己的登录态发给服务端
5. 服务端校验手机登录态并确认授权
6. 服务端更新二维码状态，并为 PC 端生成登录凭证
7. PC 端通过轮询或长连接得知扫码成功，完成登录

面试补一句：

- 核心不是“二维码图片”，而是“二维码背后的临时登录凭证和状态流转”

### 单点登录（SSO）

- 单点登录指在多个系统之间，只登录一次就能访问多个受信系统

核心思路：

- 统一认证中心负责登录
- 各业务系统信任认证中心签发的票据 / token
- 用户首次访问业务系统时，如未登录则跳转到认证中心
- 认证成功后，认证中心回传票据，业务系统完成本地会话建立

### 账号密码登录

常见流程：

1. 用户提交账号密码
2. 服务端校验
3. 校验成功后签发会话标识或 token
4. 客户端后续请求携带该标识
5. 服务端校验标识后确定登录态

### APP 认证机制

相比 Web 常见多了设备维度：

- 首次登录通常需要账号密码
- 成功后服务端签发 token / refresh token
- 客户端后续请求带 token
- 常结合设备信息做风控、绑定和长期登录控制

稳妥回答：

- 不要简单说“APP token 永不过期”
- 更合理的说法是：
  - 访问 token 往往较短期
  - refresh token 较长期
  - 结合设备信息实现更平滑的长期登录体验

### 分布式集群中如何保证线程安全

严格说，这类题更常考“分布式一致性下如何保证共享资源安全访问”。

常见答法：

- 分布式锁
  - Redis 锁
  - ZooKeeper 锁
- 数据库乐观锁 / 悲观锁
- 幂等设计
- 消息队列串行化

面试补一句：

- 单机线程安全和分布式场景不是一个层级的问题
- 分布式下关键在“多个节点之间如何协调对共享资源的访问”

## 6. 推荐与缓存题

### 商家推荐怎么做

常见回答框架：

- 基础排序信号：
  - 销量
  - 点击率
  - 转化率
  - 好评率
  - 用户评分
  - 距离 / 类目匹配
- 可加入人工规则和风控规则
- 最终按综合得分排序

如果问“第二天怎么更新推荐商家”：

- 不能简单让前一天的头部永远吃满流量
- 可保留一部分高质量老商家
- 同时给一部分新商家探索曝光机会

一句话：

- 这是“效果最大化”和“生态公平性”的平衡问题

### 如何设计一个本地缓存

高频考虑点：

- 存什么
- 存多少
- 淘汰策略
- 一致性
- 线程安全
- 过期策略
- 命中率统计
- 异常兜底

常见设计要点：

- 数据结构
  - `unordered_map + 双向链表` 常用于 LRU
- 淘汰策略
  - LRU / LFU / TTL
- 容量控制
  - 条目数限制
  - 内存大小限制
- 过期机制
  - 被动过期
  - 定时清理
- 并发控制
  - 锁 / 分段锁 / 原子操作

## 7. 经典智力题补充

### 老鼠与毒药

这类题本质考二进制编码。

高频思路：

- 每只老鼠对应一个二进制位
- 每瓶药编号转成二进制
- 某位为 1 的瓶子让对应老鼠喝
- 通过死亡老鼠组合反推出毒药编号

### 烧绳子

核心思路：

- 绳子燃烧速度不均匀
- 但“整根烧完所需总时间”固定
- 因此通过两头同时点燃，可把总时间折半

### 24 小时里时针分针秒针重合几次

这类题常见问法有歧义，面试里最好先澄清：

- 是问时针和分针重合次数
- 还是三针完全重合次数

如果只问时针和分针：

- 12 小时内重合 11 次
- 24 小时内重合 22 次

### 100 个奴隶猜帽子颜色

高频结论：

- 通过约定“奇偶校验”策略
- 可以保证至少 `99` 人一定活
- 第一人承担不确定性

### rand(5) 生成 rand(7)

常见思路：

- 先构造更大的等概率样本空间
- 再做拒绝采样

例如：

- 用两次 `rand5()` 生成 `0~24`
- 取其中 `0~20`
- 再对 `7` 取模映射到 `1~7`

### 空瓶换饮料

高频思路：

- 模拟
- 或找规律
- 关键在于是否允许“借瓶子”这一条件，不同条件答案不同

### 高楼扔鸡蛋

核心不是背结论，而是背思路：

- 1 个鸡蛋只能线性试
- 2 个鸡蛋用“递减步长”实现最坏情况平衡
- 多个鸡蛋可以用 DP / 决策最优化来做

### N 个强盗分金币

这类题本质是逆向归纳。

高频套路：

- 从人数最少的情况开始反推
- 每一步都分析“如果当前人死了，后面会怎么分”
- 当前人只需要买通刚好足够多的最便宜支持者

### 先手必胜问题

典型套路：

- 找不变量
- 找必败态和必胜态
- 倒推状态转移

比如：

- 取石子
- Nim 博弈
- 巧克力掰块

### 海量数据处理

这类题常见套路就三件事：

- 分治拆文件
- 哈希分桶
- 堆 / 排序 / 归并

面试里优先讲清：

1. 为什么内存放不下
2. 怎么把大问题拆成小问题
3. 小问题如何统计
4. 最后如何合并结果
