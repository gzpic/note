# 项目部署问题

## 1. 部署问题先讲什么

面试里讲部署，建议先按这条主线说：

1. 明确目标指标
2. 选模型和精度
3. 选硬件和推理框架
4. 做模型转换和编译
5. 跑 profiling 找瓶颈
6. 做图优化、算子优化和内存优化
7. 最后做稳定性和长时间压测

## 2. 部署时必须先定的指标

### 2.1 时延类

- TTFT
- Prefill latency
- 单 token decode latency
- 端到端总时延

### 2.2 吞吐类

- Prefill tokens/s
- Decode tokens/s
- QPS

### 2.3 资源占用类

- 峰值显存
- 稳态显存
- KV Cache 占用
- 模型权重占用
- workspace / 临时 buffer 占用
- GPU 利用率
- CPU 利用率

### 2.4 工程稳定性类

- 功耗和温度
- 长时间运行是否抖动
- 模型加载时间
- 输出正确率和退化情况

## 3. 模型、精度、硬件怎么一起选

- 先看业务目标，是追求低延迟、低成本，还是高吞吐。
- 再看模型大小、上下文长度、并发量。
- 再决定精度方案，比如 FP16、BF16、INT8、W4A16。
- 最后结合硬件能力选择框架，比如 TensorRT-LLM、vLLM、自己框架或厂商工具链。

面试里常见回答：

- 如果显存紧张，优先考虑量化和 KV Cache 压缩。
- 如果时延敏感，优先考虑编译优化、融合和 decode 路径优化。
- 如果吞吐敏感，优先考虑 batch、调度和并发能力。

## 4. 模型转换和编译流程

- 原始模型通常先从 PyTorch 等格式导出。
- 然后转成中间表示，例如 ONNX。
- 再由后端 parser 解析成目标推理引擎内部图。
- 接着做图优化、算子融合、内存规划、tactic 选择。
- 最终生成目标引擎文件或可执行运行时。

### 4.1 ONNX 需要知道什么

- ONNX 本质是中间图表示。
- 图里描述节点、拓扑关系、输入输出和静态权重。
- parser 会把 ONNX 算子映射成后端支持的 layer 或 kernel。
- 如果后端不支持某些算子，就需要 plugin、自定义 kernel 或图改写。

## 5. 计算图优化的本质

- 减少中间 tensor
- 减少显存回写
- 减少 kernel launch
- 让更多算子走到底层高性能实现

### 5.1 常见图优化

- MatMul + Add + 激活融合
- QKV 合并
- Norm + Linear 融合
- Q/DQ 与计算算子融合
- 链式 elementwise 融合

## 6. 部署时常见瓶颈

### 6.1 显存放不下

- 权重太大
- KV Cache 太大
- workspace 超预算
- 动态 shape 下 buffer 复用差

常见解决：

- 量化
- 分页 KV Cache
- 减少上下文长度
- 更积极的 memory plan

### 6.2 TTFT 高

- 模型加载慢
- 首次编译或首次 build 代价大
- Prefill 算力吃不满
- CPU 侧准备和 launch 开销高

### 6.3 Decode 慢

- KV Cache 读取效率低
- 小 kernel 太多
- L2 命中差
- 调度策略不合理

## 7. 模型量化怎么讲

- 量化的核心收益是压缩显存和提升吞吐。
- 但量化不是只改 bit 数，还要考虑：
  - outlier 处理
  - 标定方式
  - kernel 支持
  - 数据布局
  - 精度损失

### 7.1 面试高频点

- PTQ 和 QAT 的区别
- 权重量化和激活量化的区别
- KV Cache 量化有什么收益和风险
- 为什么低精度不一定一定更快

## 8. 部署落地时的工程关注点

- 是否支持动态 shape
- 是否支持 continuous batching
- 是否支持 paged KV Cache
- 是否支持多 stream / overlap
- 是否方便接 profiling 工具
- 是否方便做 plugin 或自研算子

## 9. 可直接背的部署答案

- 我做部署时会先把目标拆成时延、吞吐和资源三个维度。
- 然后根据模型大小、并发和上下文长度选精度、框架和硬件。
- 接着做模型转换、图优化和编译，优先解决显存和关键路径问题。
- 最后通过 profiling 把瓶颈拆成 prefill、decode、KV Cache、算子和调度几个层面，再决定是调框架参数、换库还是自研算子。
