# 项目部署问题

## 1. 部署问题先讲什么

面试里讲部署，建议先按这条主线说：

1. 明确目标指标
2. 选模型和精度
3. 选硬件和推理框架
4. 做模型转换和编译
5. 跑 profiling 找瓶颈
6. 做图优化、算子优化和内存优化
7. 最后做稳定性和长时间压测

## 2. 部署时必须先定的指标

### 2.1 时延类

- TTFT
- Prefill latency
- 单 token decode latency
- 端到端总时延

#### 2.1.1 这些指标分别是什么意思

- `TTFT`：`Time To First Token`，用户发出请求后，到收到第一个 token 的端到端时间
- `Prefill latency`：模型处理整段输入 prompt、建立上下文和 KV Cache 的耗时
- `单 token decode latency`：进入生成阶段后，每生成 1 个 token 的平均耗时
- `端到端总时延`：从请求进入到完整输出结束的总耗时

可以直接这样记：

- `Prefill latency` 更像“先把输入读进去要多久”
- `单 token decode latency` 更像“后面每吐一个 token 要多久”
- `TTFT` 更像“用户等到第一个字出来要多久”

#### 2.1.2 Prefill latency 和 TTFT 的区别

- `Prefill latency` 更偏模型内部的 prompt 处理耗时
- `TTFT` 是用户感知到的首 token 总等待时间

所以一般有：

> `TTFT >= Prefill latency`

因为 `TTFT` 往往还包含：

- 请求排队
- tokenizer / 前处理
- 调度和数据搬运
- 首次 build / warmup / graph capture
- 采样和结果回传

一句话：

- `Prefill latency` 是模型在“读 prompt”
- `TTFT` 是用户在“等第一个 token”

### 2.2 吞吐类

- Prefill tokens/s
- Decode tokens/s
- QPS

#### 2.2.1 吞吐指标怎么理解

- `Prefill tokens/s`：prompt 阶段每秒能处理多少输入 token
- `Decode tokens/s`：生成阶段每秒能生成多少输出 token
- `QPS`：系统每秒能处理多少请求

通常：

- `Prefill tokens/s` 更反映计算吞吐
- `Decode tokens/s` 更反映访存和 KV Cache 路径效率

#### 2.2.2 吞吐和时延的区别

- 时延看的是“单个请求要等多久”
- 吞吐看的是“单位时间内系统总共能干多少活”

可以直接记：

- 时延更偏单请求体验
- 吞吐更偏系统总体处理能力

常见对应关系：

- `TTFT`、`Prefill latency`、`单 token decode latency` 属于时延指标
- `Prefill tokens/s`、`Decode tokens/s`、`QPS` 属于吞吐指标

不要混着说：

- 时延低，不一定吞吐高
- 吞吐高，也不一定单请求体验好

例如：

- 单请求跑得很快，说明时延可能不错
- 但如果系统不能并发处理很多请求，吞吐仍然可能一般

#### 2.2.3 吞吐小且时延大通常意味着什么

这通常不是“只在意单请求”或者“只在意并发”的问题，而是：

- 单请求本身就慢
- 系统整体资源利用率也不高

常见原因：

- GPU 没吃满，但 CPU、调度、同步开销很重
- kernel 太碎，launch 太多
- 既有计算瓶颈，又有访存瓶颈
- batch 组织差，并发调度差
- 数据搬运、排队、前处理占比高
- 框架路径不对，很多高性能能力没启用

一句话：

- 吞吐小且时延大，通常说明系统既不“快”，也不“满”

### 2.3 资源占用类

- 峰值显存
- 稳态显存
- KV Cache 占用
- 模型权重占用
- workspace / 临时 buffer 占用
- GPU 利用率
- CPU 利用率

#### 2.3.1 资源占用指标分别是什么意思

- `峰值显存`：推理过程中某一时刻达到的最大显存占用
- `稳态显存`：系统跑稳之后，持续阶段的显存占用
- `KV Cache 占用`：历史 token 的 K/V 缓存占用的显存或内存
- `模型权重占用`：模型参数本身占用的显存或内存
- `workspace / 临时 buffer 占用`：算子执行过程中的临时工作区和中间 buffer 占用
- `GPU 利用率`：GPU 计算单元被使用的程度
- `CPU 利用率`：CPU 在前处理、调度、后处理、数据搬运等环节的繁忙程度

可以直接这样记：

- 显存类指标看“能不能放下”
- 利用率类指标看“硬件有没有吃满”

#### 2.3.2 资源占用和时延、吞吐的区别

- 资源占用关注“系统用了多少资源”
- 时延关注“一个请求要等多久”
- 吞吐关注“单位时间能处理多少任务”

它们之间有关系，但不是一回事：

- 资源占用高，不一定时延高
- 资源占用低，也不一定吞吐高
- GPU 利用率高，不一定用户体验好

例如：

- 显存占用很高，但如果能稳定跑，时延不一定差
- GPU 利用率不高，往往说明吞吐还有提升空间

#### 2.3.3 资源占用高通常意味着什么

- `峰值显存` 高，通常说明某些阶段中间 tensor、workspace 或 KV Cache 冲得很高
- `稳态显存` 高，通常说明权重、KV Cache 或长期 buffer 占用重
- `GPU 利用率` 低，通常说明计算没吃满，可能有调度、同步或访存问题
- `CPU 利用率` 高，通常说明前后处理、调度或数据搬运压力大

### 2.4 工程稳定性类

- 功耗和温度
- 长时间运行是否抖动
- 模型加载时间
- 输出正确率和退化情况

#### 2.4.1 工程稳定性指标分别是什么意思

- `功耗和温度`：系统运行时的能耗水平和热状态
- `长时间运行是否抖动`：长时间压测后，时延、吞吐、输出质量是否出现波动
- `模型加载时间`：模型从磁盘加载、反序列化、初始化完成所需时间
- `输出正确率和退化情况`：部署后模型输出是否与预期一致，是否因为量化、图改写或算子替换导致质量下降

可以直接这样记：

- 时延、吞吐、资源看“跑得快不快、省不省”
- 工程稳定性看“能不能长期稳定地跑”

#### 2.4.2 工程稳定性和前面三类指标的区别

- 时延关注单次请求体验
- 吞吐关注系统处理能力
- 资源占用关注消耗多少硬件资源
- 工程稳定性关注系统是否能长期稳定、可重复、可上线

也就是说：

- 一个系统即使现在跑得快，如果温度压不住、长时间会抖、输出会漂，也不能算真正可部署

#### 2.4.3 工程稳定性差通常意味着什么

- 温度高、功耗高，通常说明硬件接近热限制或能耗压力大
- 长时间抖动，通常说明调度、内存、缓存、热降频或资源竞争有问题
- 模型加载时间长，通常说明模型文件太大、序列化格式不合适或初始化流程重
- 输出正确率下降，通常说明量化、图优化、插件替换或精度设置带来了精度风险

## 3. 模型、精度、硬件怎么一起选

- 先看业务目标，是追求低延迟、低成本，还是高吞吐。
- 再看模型大小、上下文长度、并发量。
- 再决定精度方案，比如 FP16、BF16、INT8、W4A16。
- 最后结合硬件能力选择框架，比如 TensorRT-LLM、vLLM、自己框架或厂商工具链。

面试里常见回答：

- 如果显存紧张，优先考虑量化和 KV Cache 压缩。
- 如果时延敏感，优先考虑编译优化、融合和 decode 路径优化。
- 如果吞吐敏感，优先考虑 batch、调度和并发能力。

## 4. 模型转换和编译流程

- 原始模型通常先从 PyTorch 等格式导出。
- 然后转成中间表示，例如 ONNX。
- 再由后端 parser 解析成目标推理引擎内部图。
- 接着做图优化、算子融合、内存规划、tactic 选择。
- 最终生成目标引擎文件或可执行运行时。

### 4.1 ONNX 需要知道什么

- ONNX 本质是中间图表示。
- 图里描述节点、拓扑关系、输入输出和静态权重。
- parser 会把 ONNX 算子映射成后端支持的 layer 或 kernel。
- 如果后端不支持某些算子，就需要 plugin、自定义 kernel 或图改写。

## 5. 计算图优化的本质

- 减少中间 tensor
- 减少显存回写
- 减少 kernel launch
- 让更多算子走到底层高性能实现

### 5.1 常见图优化

- MatMul + Add + 激活融合
- QKV 合并
- Norm + Linear 融合
- Q/DQ 与计算算子融合
- 链式 elementwise 融合

## 6. 部署时常见瓶颈

### 6.1 显存放不下

- 权重太大
- KV Cache 太大
- workspace 超预算
- 动态 shape 下 buffer 复用差

常见解决：

- 量化
- 分页 KV Cache
- 减少上下文长度
- 更积极的 memory plan

### 6.1.1 资源占用类常见问题

#### 峰值显存太高

常见原因：

- 某些阶段中间 tensor 很大
- workspace 申请过大
- prefill 阶段 batch 或 prompt 太长
- 动态 shape 让 buffer 复用变差

定位方法：

- 分阶段统计显存占用
- 区分是加载期、prefill 期还是 decode 期冲高
- 看是权重、KV Cache 还是 workspace 占大头

处理方法：

- 量化权重
- 降低 batch 或上下文长度
- 优化 memory plan
- 减少中间 tensor 和临时 buffer

#### 稳态显存太高

常见原因：

- 模型权重本身太大
- KV Cache 长期累积
- 长生命周期 buffer 太多
 
定位方法：

- 统计稳态阶段的显存构成
- 看权重、KV Cache、长期 buffer 分别占多少

处理方法：

- 用更低精度权重，比如 `INT8`、`W4A16`
- 做 KV Cache 量化或分页
- 控制上下文长度和并发数

#### GPU 利用率低

常见原因：

- batch 太小
- kernel 太碎
- CPU 调度或同步拖慢 GPU
- 数据搬运和计算没有重叠
- decode 阶段受带宽限制，算力单元吃不满

定位方法：

- 看 GPU 时间线上是否有空洞
- 看热点 kernel 是否过小过碎
- 结合带宽、SM 利用率、Tensor Core 利用率一起看

处理方法：

- 增强 batch 和并发组织
- 做算子融合
- 减少同步点和 launch 开销
- 做 copy / compute overlap

#### CPU 利用率高

常见原因：

- tokenizer 和后处理重
- 调度逻辑复杂
- 数据搬运频繁
- CPU 在做大量 launch 和同步控制

定位方法：

- 分开统计前处理、调度、后处理时间
- 看 CPU 热点是否集中在 tokenizer、调度线程或 memcpy

处理方法：

- 优化前后处理
- 减少 CPU 参与的数据搬运
- 简化调度路径
- 做异步 pipeline，让 CPU 和 GPU 更好并行

### 6.1.2 工程稳定性类常见问题

#### 功耗和温度过高

常见原因：

- GPU 或 CPU 长时间满载
- 算子路径低效，导致做了过多无效工作
- 带宽压力大，系统一直高功耗运行
- 散热条件有限，尤其是端侧设备

定位方法：

- 长时间记录温度、频率、功耗曲线
- 看性能下降是否和温度升高同步发生
- 对比不同 batch、上下文长度、精度配置下的功耗变化

处理方法：

- 降低模型精度或模型规模
- 优化 kernel、融合和数据路径，减少无效计算
- 控制 batch、上下文长度和并发度
- 必要时做功耗模式或频率策略调整

#### 长时间运行出现抖动

常见原因：

- 热降频
- 内存碎片化或缓存命中率波动
- 调度策略不稳，batch 形成不稳定
- 后台任务、I/O、日志等周期性干扰

定位方法：

- 做长时间压测，记录 TTFT、decode latency、tokens/s 的时间序列
- 同时记录温度、频率、显存和 CPU/GPU 利用率
- 看抖动是周期性还是随运行时间逐步恶化

处理方法：

- 稳定 batch 和调度策略
- 降低热压力，避免降频
- 优化内存管理，减少碎片和频繁分配
- 清理非必要后台工作和高频日志

#### 模型加载时间过长

常见原因：

- 模型文件大
- 权重格式不利于快速加载
- 初始化、反序列化、build、warmup 流程重
- 磁盘 I/O 慢

定位方法：

- 分段统计：读文件、反序列化、build、权重搬运、warmup
- 看是 I/O 慢还是初始化慢

处理方法：

- 压缩模型体积
- 使用更适合部署的权重格式
- 预构建 engine，减少运行时 build
- 提前 warmup 或做懒初始化优化

#### 输出正确率下降或结果退化

常见原因：

- 量化精度损失
- 激活或 KV Cache 量化过激进
- 图优化或算子融合改写了数值路径
- 自定义 plugin / kernel 存在实现误差

定位方法：

- 和基线浮点模型做对比评测
- 对比关键层输出和最终任务指标
- 看退化是否集中在特定输入、长上下文或某些任务类型

处理方法：

- 对敏感层恢复更高精度
- 重新做校准或更换量化方案
- 回退高风险优化项逐个验证
- 加强回归测试和 golden output 对比

### 6.2 TTFT 高

- 模型加载慢
- 首次编译或首次 build 代价大
- Prefill 算力吃不满
- CPU 侧准备和 launch 开销高

### 6.3 Decode 慢

- KV Cache 读取效率低
- 小 kernel 太多
- L2 命中差
- 调度策略不合理

## 6.4 指标太长时怎么排查

一个通用原则：

- `TTFT` 长，不一定是模型算得慢，也可能是排队、前处理、调度、warmup 慢
- `Prefill latency` 长，通常先怀疑 prompt 太长、GEMM / Attention 算力没吃满
- `单 token decode latency` 长，通常先怀疑 KV Cache 访问、带宽和碎小 kernel
- `tokens/s` 低，本质上是 prefill 或 decode 路径效率不高

### 6.4.0 常用定位工具和分工

#### 常用工具

- 代码里埋点 / 时间戳：先把请求拆成排队、tokenize、prefill、decode、sample、回传
- `nsys`：看整体 timeline，查 CPU/GPU 协同、kernel launch gap、memcpy、同步点
- `ncu`：看单个 kernel 为什么慢，查 occupancy、stall、L2/DRAM、Tensor Core 利用率
- `perf`：看 CPU 热点、上下文切换、cache miss、调度开销
- `nvidia-smi` 或平台监控工具：看显存、GPU 利用率、功耗、温度、频率
- 框架自带 profiler / trace：看请求分段耗时、batch 形成、队列长度、调度行为

#### 一套稳妥的使用顺序

1. 先做业务埋点，确认慢在哪个阶段
2. 再用 `nsys` 看整条链路是不是有空洞、同步或 launch gap
3. 如果 GPU kernel 本身慢，再用 `ncu` 下钻
4. 如果怀疑 CPU 前处理、调度或提交问题，再用 `perf`
5. 最后结合 `nvidia-smi` 或板端监控看显存、温度、频率和功耗

一句话：

- `埋点` 负责拆阶段
- `nsys` 负责看全局时间线
- `ncu` 负责看 kernel 内部
- `perf` 负责看 CPU 侧热点

### 6.4.1 TTFT 太长

常见原因：

- 请求排队严重
- tokenizer / 前处理慢
- 首次建图、首次编译、首次加载、warmup 慢
- prompt 太长，导致 prefill 本身很重
- CPU 侧调度、launch 开销大
- 数据搬运慢

定位方法：

- 把链路拆成排队、tokenize、prefill、sample、回传几个阶段分别打点
- 对比首请求和稳态请求，判断是不是 warmup 问题
- 用 profiler 看 CPU 侧是否空转、GPU 是否有长时间空闲

定位工具：

- 先用代码埋点 / trace 看时间到底花在排队、tokenize、prefill 还是回传
- 用 `nsys` 看首请求是否有明显 build、memcpy、同步和 launch gap
- 用 `perf` 看 CPU 是否卡在 tokenizer、调度线程或提交逻辑
- 用平台监控工具看首轮是否伴随频率变化或资源初始化开销

处理方法：

- 做 warmup，避免首请求触发编译或 graph capture
- 优化 tokenizer 和前处理
- 缩短 prompt，或者做 prefix cache
- 减少 CPU 侧碎开销，优化调度和 launch
- 减少不必要的数据拷贝和同步

### 6.4.2 Prefill latency 太长

常见原因：

- 输入 prompt 太长
- GEMM / Attention 没跑满硬件
- 算子融合不够，kernel launch 太多
- 图优化、layout、tactic 选择不好
- 没有走 Tensor Core 或低精度高性能路径

定位方法：

- 单独测不同 prompt 长度下的 prefill 时间
- 看 GPU 利用率、Tensor Core 利用率、热点 kernel
- 看瓶颈是在 GEMM、Attention 还是 launch 太碎

定位工具：

- 用代码埋点单独统计 prefill 阶段耗时
- 用 `nsys` 看 prefill 阶段是不是 kernel 太碎、有没有明显 launch gap
- 用 `ncu` 看 GEMM / Attention kernel 的 Tensor Core 利用率、occupancy、stall 原因
- 用 `nvidia-smi` 或板端监控看 GPU 利用率和显存变化

处理方法：

- 控制输入长度，做 prompt 裁剪或 prefix cache
- 用更合适的精度，比如 `FP16`、`BF16`、`INT8`
- 做图优化和算子融合
- 调整 batch、shape 和 engine 配置
- Attention 路径换更高效实现

### 6.4.3 单 token decode latency 太长

常见原因：

- KV Cache 读写效率低
- decode 阶段 kernel 太小太碎
- 访存带宽成为瓶颈
- KV Cache 布局不好，cache miss 多
- prefill 和 decode 互相抢资源

定位方法：

- 分开测 prefill 和 decode，不要混着看
- 测不同上下文长度下 decode latency 的变化
- 看带宽利用率、L2 命中率、dram 访问
- 看热点是不是 attention / KV cache gather

定位工具：

- 用代码埋点只统计 decode 阶段单步耗时
- 用 `nsys` 看 decode 阶段是否存在小 kernel 过多、stream 不重叠、同步点过多
- 用 `ncu` 看 attention / KV cache gather kernel 的 L2、DRAM、stall 和 occupancy
- 用平台监控工具看显存带宽压力、频率和功耗变化

处理方法：

- 优化 KV Cache 布局，比如 paged KV 或更连续的布局
- 减少小 kernel，尽量融合
- 用更适合 decode 的 attention 实现
- 做权重或 KV Cache 量化，减轻带宽压力
- 优化调度，避免 prefill 和 decode 相互干扰

### 6.4.4 tokens/s 太低

常见原因：

- prefill 计算路径没打满
- decode 访存路径受限
- batch 和并发调度不合理
- continuous batching / paged attention 没发挥出来
- CPU、GPU、I/O 之间没有重叠起来
- 数据搬运和同步开销过大

定位方法：

- 分开看 `Prefill tokens/s` 和 `Decode tokens/s`
- 看是单请求就慢，还是并发上来才掉
- 看 GPU、CPU、内存带宽利用率
- 看 batch 是否形成得起来
- 看请求排队和调度是否有空洞

定位工具：

- 用框架 trace 或代码埋点看 batch 形成、队列长度、请求等待时间
- 用 `nsys` 看 CPU/GPU 是否有空洞，copy 和 compute 是否重叠
- 用 `perf` 看 CPU 是否花太多时间在调度、前后处理或 memcpy
- 用 `nvidia-smi` 或系统监控工具看 GPU 利用率、显存和功耗

处理方法：

- prefill 看计算优化
- decode 看 KV Cache 和带宽优化
- 多请求场景优化 batch 和调度
- 做 copy / compute overlap

### 6.4.5 吞吐慢的常见原因和处理方案

常见原因可以分成 4 类：

#### 6.4.5.1 计算路径没打满

表现：

- `Prefill tokens/s` 低
- GPU 利用率不高
- Tensor Core 利用率低

处理：

- 调整 batch 和 shape
- 用更合适的精度，比如 `FP16`、`BF16`、`INT8`
- 做图优化和算子融合
- 让更多算子走高性能 kernel

定位工具：

- `nsys` 先看 GPU 是否有明显空洞
- `ncu` 再看 Tensor Core 利用率、occupancy 和指令类型

#### 6.4.5.2 访存路径受限

表现：

- `Decode tokens/s` 低
- dram 带宽高但算力单元不满
- KV Cache 访问成为热点

处理：

- 优化 KV Cache 布局
- 减少不必要的数据搬运
- 做权重或 KV Cache 量化，减带宽压力
- 用更适合 decode 的 attention 实现

定位工具：

- `ncu` 看 DRAM 吞吐、L2 命中率、memory stall
- `nsys` 看 memcpy 和 kernel 是否挤在关键路径上

#### 6.4.5.3 调度和并发组织差

表现：

- 单请求还行，但并发一上来吞吐不升反降
- batch 形成不稳定
- 请求之间相互干扰明显

处理：

- 优化 continuous batching
- 优化请求调度和队列策略
- 拆分 prefill / decode 资源
- 减少无效等待和同步点

定位工具：

- 框架 trace 看队列长度、batch 形成和调度决策
- `nsys` 看时间线上是否有等待空洞和同步点
- `perf` 看调度线程是否成为 CPU 热点

#### 6.4.5.4 CPU / 前后处理拖后腿

表现：

- GPU 看起来没满，但整体 `QPS` 不高
- tokenizer、后处理、拷贝占比较大

处理：

- 优化 tokenizer 和前后处理
- 减少 CPU 参与的数据搬运
- 做异步化和 pipeline 化
- 尽量让 CPU、GPU、I/O 重叠执行

定位工具：

- `perf` 看 tokenizer、memcpy、调度线程和系统调用热点
- `nsys` 看 CPU 提交和 GPU 执行之间是否脱节

### 6.4.6 一套很稳的定位顺序

1. 先把链路拆成：排队 / tokenize / prefill / decode / sample / 回传
2. 再区分是首包慢还是稳态慢
3. 再区分是计算瓶颈还是访存瓶颈
4. 最后决定是调框架参数、换精度、做图优化，还是改 kernel / KV Cache / 调度

### 6.4.7 不同异常对应先看什么工具

| 异常 | 第一工具 | 第二工具 | 重点看什么 |
| --- | --- | --- | --- |
| TTFT 高 | 埋点 / trace | `nsys` | 排队、warmup、同步、launch gap |
| Prefill 慢 | `nsys` | `ncu` | GEMM、Attention、Tensor Core、kernel 粒度 |
| Decode 慢 | `nsys` | `ncu` | KV Cache、L2、DRAM、小 kernel |
| 吞吐低 | trace / 埋点 | `nsys` / `perf` | batch 形成、队列等待、CPU/GPU overlap |
| GPU 利用率低 | `nsys` | `ncu` | 空洞、occupancy、依赖链 |
| CPU 利用率高 | `perf` | `nsys` | tokenizer、调度、memcpy、提交开销 |
| 显存异常 | 监控工具 / 日志 | `nsys` | 权重、KV Cache、workspace、峰值位置 |
| 长时间抖动 | 长压测日志 | 监控工具 | 温度、频率、时延曲线、资源波动 |

## 7. 模型量化怎么讲

- 量化的核心收益是压缩显存和提升吞吐。
- 但量化不是只改 bit 数，还要考虑：
  - outlier 处理
  - 标定方式
  - kernel 支持
  - 数据布局
  - 精度损失

### 7.1 面试高频点

- PTQ 和 QAT 的区别
- 权重量化和激活量化的区别
- KV Cache 量化有什么收益和风险
- 为什么低精度不一定一定更快

## 8. 部署落地时的工程关注点

- 是否支持动态 shape
- 是否支持 continuous batching
- 是否支持 paged KV Cache
- 是否支持多 stream / overlap
- 是否方便接 profiling 工具
- 是否方便做 plugin 或自研算子

## 9. 可直接背的部署答案

- 我做部署时会先把目标拆成时延、吞吐和资源三个维度。
- 然后根据模型大小、并发和上下文长度选精度、框架和硬件。
- 接着做模型转换、图优化和编译，优先解决显存和关键路径问题。
- 最后通过 profiling 把瓶颈拆成 prefill、decode、KV Cache、算子和调度几个层面，再决定是调框架参数、换库还是自研算子。

## 10. 这几个指标的面试问答

### 10.1 Prefill latency 是什么

- 它表示模型处理整段输入 prompt、建立上下文和 KV Cache 的耗时
- 更偏 prompt 阶段的大矩阵计算

### 10.2 单 token decode latency 是什么

- 它表示生成阶段每生成 1 个 token 的平均耗时
- 更偏 KV Cache 访问和访存效率

### 10.3 Prefill latency 和 TTFT 的区别

- `Prefill latency` 是 prompt 处理的模型计算时间
- `TTFT` 是用户从发请求到看到第一个 token 的端到端时间
- 所以 `TTFT` 一般大于等于 `Prefill latency`

### 10.4 如果这些指标太长，你会怎么排查

可以直接这样答：

> 我会先把 TTFT、prefill、decode 分开测，因为它们对应的瓶颈不一样。TTFT 长先查排队、前处理、warmup 和首轮初始化；prefill 长重点看 GEMM、attention 和图优化；decode 长重点看 KV Cache 访问、带宽和碎小 kernel。定位上先做分阶段打点，再结合 profiler 看 GPU 利用率、热点 kernel、带宽和 cache 命中率；处理上再针对性做 warmup、量化、融合、KV Cache 优化和调度优化。

### 10.5 吞吐和时延的区别

- 时延关注单个请求多久返回
- 吞吐关注单位时间系统能处理多少 token 或多少请求
- 时延偏用户体验，吞吐偏系统能力

### 10.6 吞吐小且时延大说明什么

- 这通常说明系统既没有把单请求做好，也没有把整体资源吃满
- 常见是调度、同步、kernel 碎片化、带宽瓶颈和前后处理开销叠在一起

### 10.7 吞吐慢一般怎么处理

- 如果是 prefill 吞吐低，优先看计算路径、精度、融合和 batch
- 如果是 decode 吞吐低，优先看 KV Cache、带宽和 attention 实现
- 如果是 QPS 低但 GPU 不满，优先看调度、前后处理和 CPU/GPU overlap

### 10.8 资源占用类指标怎么看

- 峰值显存看最坏情况下能不能放下
- 稳态显存看长期运行是否稳
- KV Cache 和权重占用看大头是谁
- GPU/CPU 利用率看系统有没有吃满

### 10.9 资源占用高一般怎么排查

- 先拆显存构成：权重、KV Cache、workspace、临时 buffer
- 再看时间维度：是加载期高、prefill 高，还是稳态一直高
- 最后结合 GPU/CPU 利用率判断是“资源不够”还是“资源没吃满”

### 10.10 工程稳定性为什么要单独看

- 因为部署不是只跑一两次 benchmark
- 真正上线要看能不能稳定、持续、可重复地运行
- 所以除了时延、吞吐、资源，还必须单独看温度、功耗、长时间抖动、加载时间和输出退化

### 10.11 工程稳定性差一般怎么排查

- 先做长时间压测，记录时延、吞吐、温度、频率、资源曲线
- 再拆加载期、稳态运行期和长上下文场景分别看
- 输出质量问题则和基线模型做回归对比，确认是不是量化、图优化或插件带来的误差

### 10.12 定位问题时常用什么工具

- 我一般先做业务埋点，把请求拆成排队、tokenize、prefill、decode、sample、回传
- 再用 `nsys` 看整体时间线，确认是 CPU 提交慢、GPU 执行慢，还是同步和 memcpy 挤在关键路径
- 如果是关键 kernel 慢，再用 `ncu` 看 occupancy、stall、L2/DRAM 和 Tensor Core 利用率
- 如果怀疑 CPU 侧问题，再用 `perf` 看 tokenizer、调度线程、memcpy 和系统调用热点
