# 推理框架问题

## 1. vLLM 调度

从 `[[ARM/大模型基础/优秀推理框架/vllm new/调度机制/scheduler]]` 提炼：

- 核心约束：
  - `token_budget`
  - `max_num_seqs`
  - `block_manager`
- 两类调度策略：
  - 默认调度：Prefill First
  - Chunked Prefill：Decode First
- Chunked Prefill 的价值：
  - 保证正在生成请求的 ITL 稳定
  - 长 prompt 不会把整个 step 的预算一次性吃光
- 本质哲学：
  - 显存是第一生产力
  - step 级 continuous batching 才是高吞吐关键

## 4. PD 分离

从 `[[ARM/大模型基础/大模型推理核心/pd分离原理/pd分离原理]]` 提炼：

- L1：阶段级分离
- L2：批次级分离
- L3：请求级/Token 级分离
- 关键收益：
  - Prefill 不阻塞 Decode
  - Decode 可跨请求聚合
  - KV 生命周期独立
  - GPU 利用率更高

## 6. PyTorch 多卡训练

### 6.1 DP

- 单进程多卡
- 通常把梯度收集到主卡再更新
- 实现简单，但扩展性和效率一般

### 6.2 DDP

- 多进程多卡
- 每个进程一张卡
- 梯度通过 AllReduce 同步
- 是更常见的标准方案

### 6.3 FSDP

- 参数、梯度、优化器状态做分片
- 前向和反向时按需 AllGather
- 算完再释放非本 rank 分片
- 更省显存，但通信和实现复杂度更高

## 7. TensorRT、ONNX 和端侧框架

### 7.1 TensorRT 动态 shape

- 需要提前配置优化 profile
- 为不同输入维度范围准备 shape 区间
- build 时会针对 profile 做优化

### 7.2 ONNX 优缺点

- 优点：
  - 跨框架
  - 跨硬件
  - 作为中间表示很方便
- 缺点：
  - 算子支持可能不全
  - 后端优化能力差异大
  - 某些复杂模型转换麻烦

### 7.3 NCNN / TFLite

- NCNN 更专注移动端推理
- 对 ARM 和 Neon 做了很多针对性优化
- 功能更聚焦，所以很多时候效率更高

## 8. 卷积实现、图优化和算子融合

### 8.1 卷积常见实现

- Im2Col + GEMM
  - 通用性强
  - 访存较规整
  - 但会有额外展开成本
- Winograd
  - 适合部分小卷积场景

### 8.2 图优化常见项

- 常量折叠
- 算子融合
- 图剪枝
- 访存优化
- 调度优化

### 8.3 Conv + BN 融合

- 把 BN 吸收到卷积权重和偏置里
- 这样推理时就不需要单独跑 BN
- 好处是减少 kernel 和中间内存访问

## 9. 量化、算子融合和端侧部署

### 9.1 量化为什么可能加速

- 权重更小，加载更快
- 访存更省
- 低精度计算吞吐更高
- 某些硬件对 INT8 / FP8 有专门支持

### 9.2 量化不一定一定加速

- 硬件可能不支持
- Q / DQ 有开销
- 有些算子仍要回退高精度
- 图中频繁切换精度会抵消收益

### 9.3 哪些算子更适合量化

- GEMM
- Conv
- ReLU
- 部分逐元素算子

### 9.4 哪些算子量化更难

- Softmax
- LayerNorm
- GELU / Sigmoid / Tanh
- Attention 中某些归一化步骤

### 9.5 算子融合为什么快

- 减少中间 tensor
- 减少显存回写
- 减少 kernel launch

### 9.6 融合越多越好吗

- 不一定。
- 过度融合可能导致：
  - 寄存器压力太大
  - occupancy 下降
  - 数值处理更复杂
  - 底层硬件专用路径反而失效

## 10. 端侧部署为什么难

- 算力有限
- 显存或内存容量有限
- 功耗和温度敏感
- 需要适配不同硬件后端
- 模型压缩后还要兼顾精度

## 11. 硬件代际与 PD 分离

### 11.1 H100 / A100 / V100

- 主要差异看：
  - Tensor Core 代际
  - 支持的精度类型
  - 显存带宽
  - NVLink 能力

### 11.2 PD 分离为什么重要

- Prefill 更偏计算密集
- Decode 更偏访存密集
- 放在一起会资源互相争抢
- 分离后可以分别优化时延和吞吐

## 12. 推理系统高频概念

### 12.1 Orca

- Orca 的代表性思想是更细粒度地做调度和 continuous batching。
- 它强调不要把一个请求从头到尾独占执行，而是按 step / token 维度更细地混合执行。
- 这样能提升整体吞吐和资源利用率。

### 12.2 Sarathi-Serve

- 关键词可以记成：
  - 拆分
  - 重排
  - 平衡吞吐和时延
- 它的核心思路是通过更合理地拆 prompt 和重排执行顺序，缓解 Prefill 和 Decode 的资源冲突。

### 12.3 Paged Attention / Page Attention

- 核心思想是把 KV Cache 分成页或 block 管理，而不是要求一整段连续大内存。
- 好处：
  - 降低碎片
  - 更方便做复用和回收
  - 支持更高并发

## 13. Prefix Cache 和 Suffix Cache

### 13.1 Prefix Cache

- 如果多个请求共享相同前缀，可以直接复用前缀部分的 KV Cache。
- 这样最直接的收益是省 prefill。

### 13.2 Suffix Cache

- suffix cache 没有 prefix cache 常见。
- 如果被问到，可以这样答：
- prefix cache 复用的是“开头相同”的部分，天然更容易命中。
- suffix cache 复用的是“结尾相同”的部分，在自回归生成里复用场景少得多，实现也更复杂。
- 所以工程上主流还是 prefix cache。

## 14. 推理系统的调度口头题

- continuous batching 是什么
  - 不是等一整个 batch 结束再处理下一批，而是每个 step 都动态把新请求加进来
- 为什么 paged KV Cache 很重要
  - 因为长上下文和高并发下，显存管理本身就是核心问题
- 为什么 prefix cache 值钱
  - 因为共享 prompt 在很多业务里很多，直接省掉重复 prefill

## 15. 反问环节可以怎么问

- 你们线上更关注 TTFT、TPOT 还是整体吞吐
- 当前推理瓶颈更偏算子、调度还是显存管理
- 框架侧主要是基于 TensorRT-LLM、vLLM，还是自研
- 团队里更希望候选人补的是 kernel、系统调度还是部署适配能力
