# 推理算子问题

## 1. 大模型推理主流程

### 1.1 Prefill

- Embedding
- RMSNorm
- QKV GEMM
- RoPE
- KV Cache Write
- Attention
- Output Projection
- Residual
- RMSNorm
- SwiGLU / FFN
- FFN Down Projection
- Residual
- Final Norm / LM Head

### 1.2 Decode

- 流程和 prefill 类似
- 但每次只处理一个或少量 token
- attention 会读取历史 KV Cache
- 性能瓶颈会更偏向 memory bound 和 cache 命中率

## 2. Prefill 和 Decode 的关键差异

- Prefill 更像大矩阵批处理，核心看 GEMM、FlashAttention、kernel fusion。
- Decode 每步计算量小，但需要频繁读 KV Cache，核心看访存模式、L2 命中率和 launch 开销。
- Prefill 更容易把 Tensor Core 吃满。
- Decode 更容易出现 GPU 利用率不高但延迟仍高的情况，因为它常常受小 kernel、带宽和调度限制。

面试直接回答模板：

- Prefill 主要是算力问题，Decode 主要是访存和调度问题。

## 3. 推理时最重要的几类算子

### 3.1 GEMM

- 是最核心的算子，QKV 投影、输出投影、FFN 都离不开 GEMM。
- 优化重点：
  - Tensor Core
  - tile 切分
  - shared memory 搬运
  - 合理布局
  - bias / activation 融合

### 3.2 Reduce

- 典型场景：
  - RMSNorm
  - Softmax
  - TopK / TopP 里的部分步骤
- 优化重点：
  - warp 级归约
  - block 级归约
  - 尽量减少全局同步

### 3.3 Elementwise

- 典型场景：
  - residual add
  - SiLU
  - GLU 里的 mul
  - RoPE
- 单个算子 FLOPs 不高，但数量多，通常要靠融合优化。

### 3.4 Load / Store / Gather

- Embedding lookup、KV Cache 读写都强依赖访存。
- 这类算子的瓶颈通常是连续性、对齐和缓存命中，而不是算力。

## 4. RMSNorm

- RMSNorm 一般是 memory-bound。
- 它的计算本身不复杂，问题在于要读一整段向量、做归约、再写回结果。

### 4.1 常见优化顺序

- kernel 融合
- 半精度
- warp 级归约
- 向量化加载
- 合理使用 shared memory

### 4.2 面试回答点

- 如果 hidden size 不大，warp 级归约通常就很有效。
- 如果 hidden size 很大，常需要 block 级归约和分阶段 reduction。
- 如果 RMSNorm 和后面的 linear 连在一起，优先考虑做融合，减少一次读写。

## 5. SwiGLU

- SwiGLU 的核心不是公式本身，而是它通常伴随两次 up projection 和一次 down projection。
- 工程里经常把 gate 和 up 权重拼接，一次 GEMM 同时算出两路输出。
- 后续再做激活和逐元素乘法。

### 5.1 常见优化

- Gate / Up 合并 GEMM
- 激活融合到 epilogue
- 向量化读写
- 中间 buffer 复用
- 量化场景下走低精度路径

### 5.2 面试表述

- SwiGLU 优化的核心不是单独优化激活函数，而是减少中间 tensor 和多次访存。

## 6. Softmax 和 Online Softmax

- 标准 softmax 需要先求最大值，再求指数和，再归一化。
- 如果直接物化整张 score 矩阵，显存和带宽开销都很大。
- Online softmax 用 running max 和 running denominator 增量更新，可边扫边算。

### 6.1 价值

- 避免完整写回 attention score
- 显著降低中间存储需求
- 是 FlashAttention 的核心思想之一

### 6.2 面试说法

- FlashAttention 的本质不是“算法结果变了”，而是通过 tile 化和 online softmax 改了计算顺序，从而减少了显存读写。

## 7. TopK / TopP / Prefix Sum

- TopK 主要是选最大的 K 个值。
- TopP 是先做概率累计，直到累计概率超过阈值。
- Prefix Sum 在 sampling 场景里经常出现。

### 7.1 工程上的理解

- 这类算子通常不是全链路最大头，但会影响最后一步采样延迟。
- batch 很小的时候，甚至放 CPU 做也可能够用。
- 如果要放 GPU，自定义 kernel 重点在并行选择和归约，而不是复杂数学计算。

## 8. KV Cache 读写

### 8.1 写入

- Prefill 和 Decode 都会写 KV Cache。
- 要求地址尽量连续、append 高效、避免频繁 memcpy。

### 8.2 读取

- Decode 时每生成一个 token 都要读取历史 KV。
- 这一步很容易受制于带宽和 L2 命中。

### 8.3 面试重点

- Decode attention 通常比 prefill 更 memory-bound。
- 如果 KV Cache 布局不好，哪怕算子逻辑没问题，延迟也会很差。

## 9. 自研算子和直接用库的边界

### 9.1 适合直接用库

- 大部分标准 GEMM
- 标准 attention
- 常见 conv / matmul / activation

### 9.2 适合自研

- 小 batch、小矩阵、固定 shape 的特化场景
- 融合很重的业务算子
- 框架现有实现不适合目标硬件
- 采样、KV Cache、特殊布局变换这类工程定制算子

### 9.3 面试怎么答

- 我的判断标准通常是通用库是否已经覆盖、当前瓶颈是否真的在这个算子，以及自研带来的维护成本值不值得。

## 10. 常见融合策略

### 10.1 Prefill 阶段

- RMSNorm + QKV
- Attention 内部融合 softmax 和 matmul
- FFN 内部融合 gate、activation、mul

### 10.2 Decode 阶段

- 更偏局部融合
- 例如 norm + projection、KV write 周边处理
- 一般不建议跨 attention 大边界做过重融合

原因：

- Decode 本身算子粒度小，但数据依赖强
- 过重融合可能导致实现复杂、寄存器压力上升、收益反而不稳定

## 11. 面试高频追问

### 11.1 为什么 RMSNorm 常常比 GEMM 更值得优化

- 因为 GEMM 往往已有成熟库支持，Tensor Core 利用率高。
- 而 RMSNorm 这种 memory-bound 小算子在整条链路里很多，且容易产生额外 launch 和访存。

### 11.2 为什么 Decode 比 Prefill 更难优化

- 因为 Decode 每步计算量小，难以把大矩阵优势发挥出来。
- 它更依赖缓存命中、连续内存、低 launch 开销和调度细节。

### 11.3 为什么融合有时候反而变慢

- 融合后寄存器使用太多
- occupancy 降低
- shared memory 压力变大
- kernel 太复杂导致调度不理想

## 12. 可直接背的总结

- 大模型推理算子优化我会先按 Prefill 和 Decode 分开看。
- Prefill 重点看 GEMM、Attention 和融合，目标是把 Tensor Core 和吞吐吃满。
- Decode 重点看 KV Cache 访问、memory bound 算子和 launch 开销，目标是降低单 token 延迟。
- 真正常见的高收益点通常是 RMSNorm、Softmax、KV Cache 和 FFN 里的融合，而不是单独盯某个公式。

## 13. 如果让我新写一个算子，我会怎么做

这个问题面试里很常见，建议按下面这个顺序回答。

### 13.1 第一步：先定义问题

- 算子输入输出是什么
- shape 是什么
- 数据类型是什么
- 是训练用还是推理用
- 主要场景是 prefill 还是 decode
- 目标是低延迟还是高吞吐

先把这些说清楚，否则后面很多参数都没法定。

### 13.2 第二步：先判断要不要自己写

- 先看 cuBLAS、CUTLASS、TensorRT-LLM 或框架现成实现能不能满足。
- 如果通用库已经够快，一般没必要自研。
- 真正适合自己写的场景通常是：
  - shape 很特殊
  - batch 很小
  - 固定 shape
  - 需要和上下游做深度融合
  - 现有库在目标硬件上利用率不高

### 13.3 第三步：判断瓶颈类型

- 是 compute-bound 还是 memory-bound
- 是大矩阵还是小矩阵
- 是单次执行还是高频重复调用
- 数据复用高不高

这一步会直接决定后面是优先做 tiling、融合，还是先做访存和 launch 优化。

## 14. 用 GEMM 举例，新算子怎么落地

假设要写的是 `C = A x B`。

### 14.1 先确定输入条件

- `M`
  - A 的行数，也是 C 的行数
- `N`
  - B 的列数，也是 C 的列数
- `K`
  - A 的列数，也是 B 的行数
- 数据类型
  - FP32、FP16、BF16、INT8
- layout
  - row-major 还是 col-major
- 是否有 bias
- 是否要融合激活
- 是否 batch GEMM
- shape 是否固定

### 14.2 实现步骤

1. 先写一个功能正确的 reference 版本
- 先不追求极致性能，先保证结果对
- 可以先写最简单的 global memory 直接算版本

2. 建 profiling 基线
- 测正确性
- 测耗时
- 测吞吐
- 看是不是已经接近库实现

3. 设计并行划分
- 一个 block 算 C 的一个 tile
- 一个 warp 算 tile 里的一个子块
- 每个线程负责若干个输出元素

4. 设计内存搬运
- A、B 分块搬到 shared memory
- 尽量让 global load 连续且对齐
- 看是否适合向量化加载

5. 设计计算内核
- shared memory 中做分块乘加
- 尽量复用 A、B tile
- 能走 Tensor Core 就尽量走 Tensor Core

6. 做写回和可选融合
- 结果写回 C
- 如果后面还有 bias、activation，可考虑直接融合

7. 用 `nsys` 和 `ncu` 迭代调优
- 看 launch gap
- 看 occupancy
- 看 shared memory 和寄存器压力
- 看 Tensor Core 使用率

## 15. GEMM 里要定哪些参数，怎么定

### 15.1 Block Tile 大小

常见问题：

- 一个 block 负责多大的 `M x N` 输出块

怎么定：

- 先看 shared memory 能否放下 A tile 和 B tile
- 再看这个 tile 是否能让 global memory 搬运足够连续
- 再看是否能让一个 SM 上同时驻留足够多 block

经验上：

- tile 大一些，数据复用更好
- 但 tile 太大，会吃掉太多 shared memory 和寄存器，occupancy 会掉

### 15.2 Warp Tile 大小

常见问题：

- 一个 warp 负责 tile 中的哪一块

怎么定：

- 要和 block tile 对齐
- 要让 warp 内线程分工均匀
- 要尽量匹配 Tensor Core 或 mma 指令的最小计算单元

### 15.3 K 方向分块大小

常见问题：

- 每次沿 K 方向取多长的一段做累加

怎么定：

- 主要看 shared memory 容量
- 也看数据类型和向量化宽度
- K tile 大一点，复用会更好
- 但同步和资源消耗也会上升

### 15.4 BlockDim

常见问题：

- 一个 block 开多少线程

怎么定：

- 通常选 128、256 这类 32 的倍数
- 要和 warp tile 划分对应
- 要避免线程太少导致并行度不够
- 也要避免线程太多导致寄存器和 shared memory 压力太大

### 15.5 每线程负责多少个输出元素

怎么定：

- 每线程多算几个值，可以提高寄存器复用和计算密度
- 但会增加寄存器压力
- 所以要在计算复用和 occupancy 之间平衡

### 15.6 是否使用 shared memory

怎么定：

- 如果数据复用明显，几乎一定要用
- 如果是很小、很碎的场景，要看 shared memory 和同步成本值不值

### 15.7 是否使用 Tensor Core

怎么定：

- 看数据类型是否支持
- 看矩阵维度是否适配
- 看目标硬件 SM 架构
- 看 layout 和对齐是否满足要求

### 15.8 是否做 double buffering

怎么定：

- 如果搬运和计算都比较重，可以一边搬下一块一边算当前块
- 但实现会更复杂，也更吃 shared memory

### 15.9 是否融合 bias / activation

怎么定：

- 如果 GEMM 后面紧跟 bias、SiLU、GELU，这类融合通常收益不错
- 但如果融合导致寄存器暴涨，可能反而变慢

## 16. 这些参数本质上是怎么确定的

不是拍脑袋定，而是按下面这个逻辑：

1. 先看硬件约束
- shared memory 上限
- 寄存器上限
- warp 大小
- Tensor Core 指令形态

2. 再看问题规模
- `M/N/K` 大小
- batch 大小
- 固定 shape 还是动态 shape

3. 再看优化目标
- 是追求单次延迟还是总体吞吐

4. 最后靠 profiling 和实验收敛
- 先给几组候选 tile
- 实测吞吐、occupancy、stall、寄存器和 shared memory 使用
- 保留效果最好的组合

## 17. 面试里可以直接这样回答

- 如果让我从 0 写一个新算子，我不会一上来就写代码，而是先定义输入输出、shape、数据类型和目标场景，然后判断有没有必要自研。
- 如果以 GEMM 为例，我会先写正确版本，再做 block、warp、K 方向的 tiling，确定 blockDim、shared memory、每线程负责的输出块和是否走 Tensor Core。
- 这些参数不是凭感觉定的，核心看三件事：硬件资源约束、问题规模，以及 profiling 结果。
- 最后我会在正确性、吞吐、单次延迟和实现复杂度之间找一个工程上最合适的点。
