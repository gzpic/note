# 问题定位方法

## 1. 我通常怎么分层定位

建议固定排查顺序，不要一上来就盯某个 kernel：

1. 是整体慢，还是局部慢
2. 是 Prefill 慢，还是 Decode 慢
3. 是 CPU 提交慢，还是 GPU 执行慢
4. 是算力瓶颈，还是带宽瓶颈
5. 是显存容量问题，还是显存访问模式问题

面试里可以直接说：

- 我习惯先从宏观时间线确认瓶颈层级，再下钻到单个算子和单个 kernel。

## 2. 工具分工

### 2.1 `perf`

- 看 CPU 热点
- 看函数级采样
- 看上下文切换
- 看 cache miss、branch miss 等系统层行为

### 2.2 `nsys`

- 看整体 timeline
- 看 CPU/GPU 协同
- 看 kernel launch gap
- 看 memcpy、kernel、同步点的相对位置
- 看 stream overlap 是否做起来了

### 2.3 `ncu`

- 看单个 kernel 为什么慢
- 看 occupancy
- 看 stall 原因
- 看寄存器压力
- 看 shared memory 使用
- 看 DRAM / L2 吞吐
- 看 Tensor Core 利用率

## 3. 这些工具分别擅长什么

- `perf` 适合查 CPU 侧热点和调度问题。
- `nsys` 适合先回答“整体慢在哪”。
- `ncu` 适合回答“这个 kernel 为什么慢”。

不要混用结论：

- `nsys` 不擅长解释单个 kernel 内部细节。
- `ncu` 不适合看整机宏观时序。
- `perf` 不适合直接看 GPU kernel 内部执行。

## 4. 常见定位思路

### 4.1 TTFT 高

优先排查：

- 模型加载时间
- 首次 build / compile 开销
- Prefill 计算量
- CPU 侧输入准备和 launch 开销
- 是否有不必要的同步

### 4.2 Decode 延迟高或抖动

优先排查：

- KV Cache 读取模式
- L2 命中情况
- 小 kernel 是否太多
- 调度是否频繁抢占
- 显存回收和 block manager 是否有抖动

### 4.3 GPU 利用率低

优先排查：

- batch 太小
- kernel 粒度太碎
- launch 开销过大
- 依赖链太长
- occupancy 太低

### 4.4 OOM 或显存异常

优先排查：

- 历史 KV 是否释放
- 内存碎片
- workspace 预算是否超了
- 动态 shape 是否导致额外 buffer
- 调度器是否把并发打太高

## 5. 如何判断是算力瓶颈还是带宽瓶颈

### 5.1 更像带宽瓶颈

- DRAM 吞吐高
- L2 / global load 很重
- Tensor Core 利用率不高
- stall 以 memory dependency、long scoreboard 为主

### 5.2 更像算力瓶颈

- Tensor Core 或 FMA 利用率高
- DRAM 吞吐没有顶满
- 指令主要集中在 mma / fma
- 算子本身 FLOPs 很大

### 5.3 两边都不高

- 往往不是纯算力或纯带宽问题
- 更可能是并行度不足、launch gap、线程分歧或依赖链太长

## 6. `ncu` 看指标时的判断逻辑

- `gld_efficiency` 低
  - 优先怀疑不连续访问和对齐问题
- local memory 高
  - 优先怀疑寄存器 spill
- eligible warps 低
  - 可运行 warp 不够，隐藏不了延迟
- control 指令占比高
  - 分支和循环结构可能有问题
- shared memory 使用高但收益不明显
  - 可能存在 bank conflict 或同步成本过高

## 7. 大模型场景下的固定排查模板

### 7.1 Prefill 慢

- 先看 GEMM 是否走到高性能路径
- 再看 attention 是否用了合适实现
- 再看 norm、rope、elementwise 是否小 kernel 太多
- 最后看有没有融合空间

### 7.2 Decode 慢

- 先看 KV Cache 读写
- 再看小矩阵 GEMM 和 attention kernel
- 再看 launch 开销和 graph / stream 优化空间
- 最后看调度策略和 batch 组织方式

## 8. 面试里可直接背的定位答案

- 我定位问题通常分三层：先用 `nsys` 看整条链路慢在哪，再用 `ncu` 看关键 kernel 慢在哪里，最后如果怀疑 CPU 侧准备或调度问题，再用 `perf` 看热点。
- 如果是 Prefill 慢，我会重点看 GEMM、Attention 和融合。
- 如果是 Decode 慢，我会重点看 KV Cache、L2 命中率、小 kernel 数量和 launch gap。
- 如果 GPU 利用率不高，我不会直接下结论说算力不足，而是先判断是不是 batch 太小、occupancy 太低或者依赖链太长。
