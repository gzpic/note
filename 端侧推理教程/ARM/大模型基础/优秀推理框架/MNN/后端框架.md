## 🧩 二、CUDARuntime：底层 GPU 驱动封装层（设备级）

`CUDARuntime` 是 MNN 中与 **CUDA Driver API** 直接交互的类，相当于“GPU 环境控制器”。

### 📦 功能概述

| 功能 | 说明 |
| --- | --- |
| 设备上下文初始化 | 在构造函数中根据 `device_id` 调用 `cudaSetDevice()`、`cudaGetDeviceProperties()` 等 |
| 内存分配与释放 | `alloc()`、`free()` 封装了 `cudaMalloc`、`cudaFree` |
| 数据拷贝 | `memcpy()` 封装了 `cudaMemcpy`，支持 H2D / D2H / D2D |
| 同步控制 | `device_sync()` 用 `cudaDeviceSynchronize()` 保证 GPU 执行完毕 |
| 能力检测 | `isSupportedFP16()`、`isSupportedDotInt8()` 等检测 GPU 特性 |
| kernel 调度辅助 | 提供线程块数、共享内存、warp 配置（`blocks_num()` / `smemPerBlock()`） |
| cache/tuning 支持 | 记录 cutlass tuning 的 block/warp shape 信息 |

---

### 📘 核心成员

```c++
cudaDeviceProp mProp;      // CUDA 设备属性
int mDeviceId;             // 当前使用的 GPU ID
size_t mThreadPerBlock;    // 默认每个 block 的线程数
bool mIsSupportedFP16;     // 硬件特性
bool mSupportDotInt8;
std::map<...> mTunedBlockWarpShape;  // kernel tuning cache


```

> 🔹 注意：
> 这个类中不持有 Tensor，也不调度 Execution。它只负责提供**最原始的 CUDA 操作接口**。
> 
> 所以它是整个 CUDA 执行栈中**最底层的支撑层**。

---

## ⚙️ 三、CUDABackend：中层算子执行与资源调度层

`CUDABackend` 则是建立在 `CUDARuntime` 基础上的 **执行管理器**。

它不直接调用 CUDA API，而是通过内部的 `CUDARuntime` 实例执行。

---

### 📦 功能概述

| 功能 | 说明 |
| --- | --- |
| Tensor 内存分配 | `onAcquire()` 从 `BufferAllocator` 申请 GPU 内存 |
| 创建算子执行体 | `onCreate()` 根据 op 类型创建 `Execution`（例如卷积、GEMM 等） |
| 计算前后控制 | `onResizeBegin()` / `onResizeEnd()` / `onExecuteBegin()` / `onExecuteEnd()` 控制整个计算周期 |
| 设备同步 | `onSync()` 调用 `cudaDeviceSynchronize()` |
| 数据拷贝 | `onCopyBuffer()` 负责 Host↔Device Tensor 传输 |
| kernel 注册管理 | 每种算子注册自己的 `Creator`，实现算子→Execution 的映射 |

---

### 📘 核心成员

```c++
std::shared_ptr<CUDARuntime> mCUDARuntime;  // 指向底层 GPU 环境
std::shared_ptr<BufferAllocator> mBufferPool;  // 内存池
CPUResizeCache mCache;  // 缓存输入输出尺寸映射
const Runtime* mRuntime;  // 上层 RuntimeWrapper 引用


```

> 🔹 它是“上接 RuntimeWrapper、下调 Runtime”的中层核心。
> 在一次模型推理时，`CUDABackend` 创建并执行所有算子的 `Execution` 对象。

---

## 🧠 四、CUDARuntimeWrapper：Runtime 接口层（工厂）

在 `CUDABackend` 之上，还有一个 `CUDARuntimeWrapper`，它继承自 MNN 的抽象 `Runtime` 基类。

这是系统用于统一管理不同后端（CPU、CUDA、OpenCL、Vulkan等）的入口。

### 📦 功能概述

| 功能 | 说明 |
| --- | --- |
| 工厂模式 | `onCreate()`：创建对应的 `CUDABackend` |
| 垃圾回收 | `onGabageCollect()`：清理 buffer pool |
| 缓存管理 | `onGetCache()` / `onSetCache()` |
| 性能统计 | `onGetMemoryInMB()` 返回当前 GPU 内存使用情况 |
| 默认策略 | 精度、内存、功耗三种模式的默认配置 |

---

### 📘 核心成员

```c++
std::shared_ptr<CUDARuntime> mCUDARuntime;    // 指向底层 GPU 上下文
std::shared_ptr<EagerBufferAllocator> mBufferPool;
bool mIsCreateError;
BackendConfig::PrecisionMode mDefaultPrecision;
BackendConfig::MemoryMode mDefaultMemory;


```

> 🔹 它相当于“RuntimeManager”中的注册表项。
> 每当 Session 想要一个 CUDA 后端时，就会调用 `CUDARuntimeWrapper::onCreate()` 生成一个新的 `CUDABackend`。

---

## 🔩 五、三者的调用链总结

| 顺序 | 调用方 | 被调用者 | 作用 |
| --- | --- | --- | --- |
| 1️⃣ | Session / Executor | `CUDARuntimeWrapper::onCreate()` | 请求创建一个后端 |
| 2️⃣ | CUDARuntimeWrapper | `new CUDABackend()` | 创建 CUDA Backend |
| 3️⃣ | CUDABackend | `mCUDARuntime->alloc()` / `memcpy()` / `device_sync()` | 调用底层 CUDA 环境 |
| 4️⃣ | Execution | `cuLaunchKernel()` | 实际执行 CUDA kernel |

完整过程就是：

```plain text
Session → RuntimeWrapper → CUDABackend → CUDARuntime → CUDA Driver → GPU SM 执行


```

## 🔬 六、层次功能总结表

| 模块 | 类名 | 定义位置 | 核心功能 | 生命周期 |
| --- | --- | --- | --- | --- |
| Runtime Wrapper | `CUDARuntimeWrapper` | `CUDABackend.hpp` | 统一管理、生成 Backend、缓存资源 | Session 全局 |
| Backend | `CUDABackend` | `CUDABackend.hpp` | 管理算子、分配显存、执行算子 | 推理级 |
| GPU Context | `CUDARuntime` | `CUDARuntime.hpp` | CUDA 环境封装、设备管理、显存操作 | 全局或设备级 |
| Kernel | 各 `Execution` 类 | 各 op 源文件 | 实际算子执行（cuLaunchKernel） | 单 op 执行周期 |

# 🟦 **Step 0：读取输入 Tensor**

```c++
query_input_tensor = inputs[0]; // [B, Lq, Hq, D]
key_input_tensor   = inputs[1]; // [B, L_new, H_kv, D]
value_input_tensor = inputs[2]; // [B, L_new, H_kv, D]
mask_input_tensor  = inputs[3]; // (optional)


```

Mask 的特殊处理：

> 如果 mask 是 1×1×1×1 且值为 0/–0，视为 无 mask（某些模型 export 会带无效 mask）

---

# 🟦 **Step 1：准备 GPU Stream**

```c++
cudaStream_t stream = 0;


```

MNN 对 CUDA 不使用显式 stream，统一用默认 stream。

---

# 🟦 **Step 2：构造 CPU 端的 AttentionKernelParam param_cpu**

填充核心字段：

```plain text
param_cpu.batch
param_cpu.query_seq_len
param_cpu.head_num
param_cpu.kv_head_num
param_cpu.group = head_num / kv_head_num
param_cpu.head_dim
param_cpu.scale = 1/sqrt(D)
param_cpu.current_kv_seq_len_new = mNewKvSeqLen


```

在 decode/prefill 的 Attention 中，这是 GPU 端所有 kernel 的 **全局元数据指挥中心**。

---

# 🟦 **Step 3：处理 KV Cache（最关键部分）**

两种情况：

---

# **① 如果启用 KV Cache（mIsKVCacheEnabled = true）**

这里分两类情况：

---

## **Case A：没有 meta（普通 KV Cache 增长模式）**

```plain text
past_kv_len = mCache->mPastLength
current_total_kv_len = past + new


```

然后：

```plain text
reallocKVCache_gpu(current_total_kv_len)


```

（扩容 KVCache，使能放下 old + new）

---

## **Case B：有 meta（启用 remove/reserve/add 的动态 KV 管理）**

例如 sliding window：

```plain text
past_len_after_remove = mPastLength - meta->remove
reserve 区段重新compact
new_kv 追加 add tokens


```

实际代码：

```plain text
required_total_kv_len = (previous - remove + reserve_total) + addreallocKVCache_gpu(required_total_kv_len, meta)


```

**这个分支将旧KV按 reserve 重新排列，非常强大。
类似 vLLM 的 PagedAttention。**

---

# 🟦 **Step 4：把当前 step 的 K/V 写入 KV Cache**

调用 kernel：

```plain text
copy_kv_to_cache_kernel<T>


```

将：

```plain text
K: [B, L_new, Hkv, D]
V: [B, L_new, Hkv, D]


```

写入：

```plain text
KCache: [L_total_alloc, B, Hkv, D]
VCache: [B, Hkv, D, L_total_alloc]


```

写入起点：

```plain text
dest_seq_idx = past_kv_len + l


```

这是你的 K/V 缓存能 **Prefill + Decode** 的根基。

---

# 🟦 **Step 5：计算 Q 分片大小（L_q_piece）**

```c++
max_q_seq_piece_len = UP_DIV(mQuerySeqLen, mQseqSplitNum)


```

例如：

```plain text
L_q = 4096
mQseqSplitNum = 4
→ L_q_piece = 1024


```

然后分片循环：

```c++
for i in range(mQseqSplitNum):
    q_seq_offset = i * L_q_piece
    current_piece_actual_len = min(L_q_piece, L_q - offset)


```

Q 分片是 **长序列 attention 的内存优化关键**。

---

# 🟦 **Step 6：每个 Q 片段启动 Attention**

每片执行：

```plain text
QK → Softmax → QKV


```

下面分 3 步讲每个 kernel 的职责。

---

# 🟩 **Step 6.1：把当前片段 param_cpu 拷贝到 GPU**

因为 q_seq_piece_len 每片不同：

```plain text
current_iteration_param_cpu.q_seq_piece_len = current_piece_actual_len
cudaMemcpyAsync(mParam_gpu, ...)


```

GPU Kernel 之后将用 mParam_gpu 的数据。

---

# 🟩 **Step 6.2：QK Kernel**

```c++
qk_kernel<T><<<>>>(
    query_input, key_cache,
    mTempQK, mask, mParam_gpu,
    q_seq_offset, mHasMask, mIsAddMask
);


```

输出：

```plain text
TempQK: [B, Hq, L_q_piece, L_k_total]


```

功能：

- 点积 Q·K^T
- 乘以 scale
- 根据 mask 做 additive mask 或 zero mask
- GQA：h_kv = h_q / group

**这是 Attention 中最耗时的步骤之一。**

---

# 🟩 **Step 6.3：Softmax Kernel（MNN 最优版本）**

不是你写的 slow softmax，而是：

```plain text
if axis <= 32 → SOFTMAX_WARP_32
else → SOFTMAX_AXIS_REDUCE


```

输出：

```plain text
mTempSoftmax: [B, Hq, L_q_piece, L_k_total]


```

Warp 级 softmax 是非常快的。

---

# 🟩 **Step 6.4：QKV Kernel（attention of value）**

```c++
qkv_kernel<T><<<>>>(
    mTempSoftmax, value_cache,
    final_output_tensor,
    mParam_gpu,
    q_seq_offset
);


```

输出写入最终 tensor：

```plain text
output[b, q_offset + q, h, d]


```

注意：

- ValueCache layout 是 [B, H_kv, D, L]
→ 访问连续内存，速度更快（MNN 这点比很多框架都优秀）

---

# 🟦 **Step 7：更新 KV Cache 的 past length**

如果启用 KV Cache：

```plain text
mCache->mPastLength += mNewKvSeqLen;


```

如果启用 meta：

```plain text
pastLength = remove 后的 + reserve 后的 + add 新的


```

---

# 🟦 **Step 8：返回最终结果**

```c++
return MNN::NO_ERROR;


```

---

# 🧠 **最终总流程图（你可以直接放文档里的版本）**

```plain text
Input Q, K_new, V_new
        │
        ▼
  (1) 构造 KernelParam
        │
        ▼
  (2) 预处理 KVCache
      - remove
      - reserve
      - compact
      - realloc
        │
        ▼
  (3) 追加新 KV 到 Cache
        │
        ▼
  (4) Q 分片 for piece in pieces:
        │
        ├── QK kernel → TempQK
        │
        ├── Softmax kernel → TempSoftmax
        │
        └── QKV kernel → 写入 final output
        │
        ▼
  (5) 更新 past length
        │
        ▼
  输出 Attention 结果


```

---

# 🏆 **一句话总结：**

> onExecute 的核心：
通过 KV Cache 管理 + Q 分片，在显存可控的情况下执行完整 Attention，并支持动态长序列（remove/reserve/add）。

这是一个 **完整、工业级、可扩展到上万 token 的 Attention 引擎**。
