1. 抢占的时机

decode/prefill 显存不足的时候，不抢占会怎么样？OOM

抢占（Preemption）是 vLLM 调度器在显存资源紧张时的**自救机制**。

简单来说，当 GPU 显存（KV Cache）被写满，无法容纳下一个要生成的 Token 时，调度器必须做出决断：**暂停某些正在运行的请求，把它们占用的显存释放出来，给剩下的请求腾出空间继续跑。**

如果没有抢占机制，系统就会直接 OOM（Out Of Memory）崩溃。

---

### 1. 什么时候触发抢占？

抢占逻辑主要发生在 `_schedule_running` 函数中，也就是处理 **Decode（生成）阶段** 的时候。

在代码中，触发流程如下：

2. **尝试调度**：调度器遍历 `running` 队列，想让所有正在跑的请求都生成下一个 Token。
3. **检查显存**：调用 `self._can_append_slots(seq_group)`。这会问 `BlockManager`：“还能给这个请求分配一个新的物理块吗？”
4. **发现不足**：如果 `BlockManager` 说“不行，没物理块了”，循环条件 `while not self._can_append_slots(...)` 成立。
5. **触发抢占**：进入 `while` 循环内部，开始踢人。

Python

`# 代码位置: Scheduler._schedule_running

# 只要显存不够给当前 seq_group 分配新 slot
while not self._can_append_slots(seq_group, enable_chunking):
    # ... (中间省略了 budget 归还逻辑) ...
    
    # 1. 选择受害者 (Victim)
    if running_queue:
        # 优先踢掉队列末尾的请求（通常是优先级最低或最晚加入的）
        victim_seq_group = running_queue.pop()
    else:
        # 如果队列空了，说明当前这个请求自己都跑不了，只能踢掉自己
        victim_seq_group = seq_group
    
    # 2. 执行抢占
    self._preempt(victim_seq_group, blocks_to_swap_out)`

### 2. 抢占的两种模式

vLLM 支持两种抢占模式，由 `PreemptionMode` 枚举定义。在代码的 `_preempt` 方法中可以看到决策逻辑：

### 模式一：Swapping（交换 / CPU Offload）

这是最经典的模式。类似于操作系统的 Swap 分区。

- **机制**：将该请求在 GPU 上的 KV Cache 数据拷贝到 CPU 内存（Host Memory）中，然后释放 GPU 块。
- **状态变化**：`RUNNING` -> `SWAPPED`。
- **优点**：保留了生成进度，不需要重算。
- **缺点**：PCIe 传输速度比 GPU 显存慢，换入换出有延迟。
- **适用场景**：Beam Search（因为重算状态太复杂）或用户显式配置。

### 模式二：Recomputation（重计算）

- **机制**：直接丢弃该请求的所有 KV Cache，释放显存。下次轮到它时，把它当成一个新的 Prefill 请求，利用 **Prefix Caching**（如果开启）或从头开始计算。
- **状态变化**：`RUNNING` -> `WAITING` (通常重置为初始状态)。
- **优点**：不需要 CPU 内存做 Swap，实现简单。如果 Prefix Caching 命中率高，代价也不大。
- **缺点**：浪费了之前的 GPU 计算资源。
- **适用场景**：单序列生成（`n=1`），且代码注释中提到默认倾向于 Recompute（因为 Swap 开销大）。

> 代码细节中的“默认策略”：
在你提供的源码 _preempt 方法中，有一个有趣的逻辑：
> `if self.user_specified_preemption_mode is None:
>     if seq_group.get_max_num_running_seqs() == 1:
>         preemption_mode = PreemptionMode.RECOMPUTE # 单序列默认重算
>     else:
>         preemption_mode = PreemptionMode.SWAP      # Beam Search 等必须 Swap`
> 
> vLLM 认为对于简单的请求，重算可能比通过 PCIe 搬运数据更快（特别是如果 Prompt 很短）。

### 3. 抢占后的“复活” (Resume)

被抢占的请求不会被丢弃，它们只是暂时退出了舞台。

### 对于 Swapped 的请求：

在后续的 Step 中，如果显存充裕了，`_schedule_swapped` 函数会被调用：

6. 检查是否有足够的 GPU 显存。
7. 执行 `_swap_in`：申请 GPU 块，通过 `blocks_to_swap_in` 列表告诉 Worker 将 CPU 数据搬回 GPU。
8. 状态变回 `RUNNING`，继续生成。

### 对于 Recomputed 的请求：

它们被放回了 `waiting` 队列头部（通常有较高优先级）：

9. 被 `_schedule_prefills` 重新捕获。
10. 重新经历 Prefill 阶段（可能利用 Prefix Cache 加速）。
11. 状态变回 `RUNNING`。

### 4. 总结：抢占机制的数据流

我们可以用一个简单的流程图来概括这个过程：

12. **正常运行**: 请求 A, B, C 在 GPU 上 `RUNNING`。
13. **OOM 危机**: 轮到请求 D 需要新 Token，但 `BlockManager` 报 Full。
14. **选择受害者**: 调度器从 `running` 队列末尾选中请求 C。
15. **执行抢占**:
    - **Swap 模式**: C 的 KV Blocks (GPU) -> C 的 CPU Blocks。C 状态 -> `SWAPPED`。
    - **Recompute 模式**: 释放 C 的 KV Blocks。C 状态 -> `WAITING`。
16. **空间释放**: 请求 C 占用的 GPU 块变成 Free 状态。
17. **继续调度**: 请求 D 拿到刚才释放的块，成功分配 Slot，继续运行。
18. **稍后恢复**: 等 D 跑完释放了空间，C 被重新调度回 GPU。

这就是为什么你在使用 vLLM 时，即使并发请求量超过了显存物理上限，程序也不会崩，只是推理延迟（Latency）会增加的原因。
