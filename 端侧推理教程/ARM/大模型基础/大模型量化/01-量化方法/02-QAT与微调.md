# QAT与微调

## 1. QAT 必须在预训练时做吗

不是。

QAT 的本质是：

> 在可训练阶段把量化误差引入前向传播，并通过反向传播让模型适应这种误差。

因此它只要求：

- 权重可更新
- 有训练 / 微调过程

并不要求一定发生在预训练阶段。

## 2. 为什么很多人误以为 QAT 必须在预训练做

原因主要来自传统 CNN 经验：

- 从零训练时就插入量化模拟
- 所以容易把 QAT 和 pretrain 绑定起来

但 LLM 的现实完全不同：

- 预训练成本太高
- 部署目标往往在训练后期才明确
- 更常见的是微调后再部署

## 3. LLM 中 QAT 的现实位置

| 阶段 | 能否做 QAT | 工程现实 |
| --- | --- | --- |
| 预训练 | 能 | 很少这样做 |
| 全量微调 | 能 | 成本很高 |
| SFT | 能 | 有，但不算主流 |
| LoRA / Adapter 微调 | 能 | 最常见 |
| 只对部分 block 再训练 | 能 | 很实用 |

## 4. 为什么 LLM 很少在预训练时做 QAT

主要是因为不划算：

- 训练成本已经极高
- FakeQuant 会增加数值不稳定性
- 会拖慢训练
- 预训练阶段并不一定知道最终部署精度和硬件

一句话：

> 预训练的目标是学泛化能力，不是绑定某个量化格式。

## 5. 微调阶段做 QAT 为什么更合理

微调阶段的优势很明显：

- 数据量小
- 训练成本低得多
- 可以针对具体部署方案优化
- 可以只修复量化带来的精度损失

这才是 QAT 在 LLM 里最实际的入口。

## 6. 常见的 QAT 微调模式

### 6.1 LoRA-QAT

最常见的现实方案：

- Base model 冻结
- 只训练 LoRA / Adapter
- 前向中插入 FakeQuant

优点：

- 成本低
- 精度提升明显
- 更贴近工业实践

### 6.2 Block-level QAT

只对关键模块做 QAT，例如：

- Attention block
- MLP block
- 某些 projection

这种方式很适合推理框架 / 算子工程视角，因为它更接近“只修最敏感的部分”。

### 6.3 全模型 QAT

理论上可以，现实里不太建议：

- 显存压力大
- 训练复杂
- 收益不一定匹配成本

## 7. QAT 与 PTQ 的关系

| 维度 | PTQ | QAT |
| --- | --- | --- |
| 是否改权重 | 否 | 是 |
| 成本 | 低 | 高 |
| 激活量化 | 难 | 更容易 |
| 低 bit 精度 | 一般 | 更好 |
| 工程门槛 | 低 | 高 |

一句话：

> PTQ 是先试的方案，QAT 是精度兜底方案。

## 8. 工程建议

如果目标是 Jetson / TensorRT / 自研推理框架，比较合理的路线通常是：

1. 先做 PTQ，例如 SmoothQuant / AWQ
2. 如果精度不够，再做 LoRA-QAT 或局部 QAT
3. 不要一上来就做全模型 QAT

## 9. 面试速记版

- QAT 不是“预训练技术”，而是“带量化误差的再训练技术”
- 在 LLM 里，QAT 更常发生在微调阶段，而不是预训练阶段
- 工程上最常见的是 LoRA-QAT 和 block-level QAT
