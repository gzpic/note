# 推理时量化由谁决定

## 1. 结论

推理阶段到底使用什么量化方案，不是模型单独决定，也不是推理框架单独决定，而是三方共同作用的结果：

- 模型给出可量化边界
- 推理框架决定如何落地
- 硬件决定哪些路径真正高效

一句话：

> 推理量化方案 = 模型约束 ∩ 框架能力 ∩ 硬件支持

## 2. 模型决定什么

模型主要决定：

- 哪些部分容易量化
- 哪些部分对精度更敏感
- 权重分布和激活分布是否友好
- 是否已有量化 checkpoint

例如：

- 某些模型更适合 W8A8
- 某些模型 KV cache 更敏感
- 某些模型 weight-only 路线更稳

## 3. 推理框架决定什么

推理框架真正掌控的是工程实现细节：

- per-tensor / per-channel / per-group 怎么选
- scale 怎么存
- dequant 在哪里做
- KV cache 是否量化
- prefill 和 decode 是否用不同量化策略

这些都是性能差异的真正来源。

## 4. 硬件决定什么

即使模型和框架都支持，硬件不友好也没有意义。

例如在 Jetson Orin Nano 上：

- `FP16` 很重要
- `INT8` 现实可用
- 更激进的精度要看 unpack / dequant 是否划算

所以工程上不能只看“比特更低”，而要看“总执行路径是否更省”。

## 5. 工程判断公式

可以把是否采用某种量化方案理解成：

```text
量化是否值得 =
模型数值稳定性
AND 框架是否有高效 kernel
AND 是否真的减少了总访存与总开销
```

## 6. 面试速记版

- 模型决定可量化上限
- 框架决定工程实现
- 硬件决定是否真的快
- 真正的性能差异不在“是否量化”，而在“怎么量化、怎么跑”
