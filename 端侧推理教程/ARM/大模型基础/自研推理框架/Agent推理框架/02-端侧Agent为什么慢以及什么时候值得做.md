# 端侧 Agent 为什么慢以及什么时候值得做

## 1. 结论

端侧 Agent 大概率会比云端 Agent 更慢，也通常比普通本地聊天更慢。

但端侧做 Agent 的目标，不只是追求绝对速度，而是追求：

- 本地可用
- 低抖动
- 隐私
- 离线能力
- 成本可控

## 2. 为什么端侧 Agent 会更慢

### 2.1 Agent 天生就比普通聊天慢

普通问答通常是：

- 1 次 prefill
- 1 段 decode

Agent 则常常变成：

- 推理
- 判断
- 调工具
- 回写 observation
- 再推理

所以：

- token 更多
- 轮次更多
- context 更长

### 2.2 端侧算力本来就弱

和云端相比，端侧设备通常有：

- 更小显存
- 更低带宽
- 更弱算力
- 更强的资源竞争

所以 Agent 这种“多轮、多上下文、多工具”的模式，会把端侧劣势放大。

## 3. 那为什么还要做端侧 Agent

因为有些场景里，价值不在“全球最快”，而在“本地就能完成任务”。

典型价值有：

- 省掉网络往返
- 本地工具调用更直接
- 离线也能工作
- 隐私更强
- 响应更稳定

## 4. 哪些场景适合端侧 Agent

端侧 Agent 更适合：

- 智能家居
- 机器人
- 车载助手
- 工业终端
- 本地知识库检索
- 摄像头 / 传感器协同

这些场景里，tool 本来就在本地，网络反而是额外开销。

## 5. 哪些场景不适合端侧 Agent

通常不太适合端侧的是：

- 超长复杂推理
- 深度搜索
- 大量网页工具编排
- 重度多工具协同
- 超长文档分析

这些通常还是云端更合适。

## 6. 端侧 Agent 最重要的设计目标

端侧做 Agent，重点不是追求“最强 Agent”，而是：

> 做一个够用、稳定、可控的 Agent。

因此更现实的路线通常是：

- 小模型
- 少量本地工具
- 短上下文
- 轻量 memory
- 尽量少轮次推理

## 7. 智能家居为什么常常更适合端侧

因为这类任务通常：

- 上下文短
- 工具简单
- 工具就在本地
- 不需要长 reasoning

例如：

- 开灯
- 调空调
- 控制窗帘

这些任务很多时候本地不仅更隐私，而且整体链路反而更快、更稳定。

## 8. 一句话总结

端侧 Agent 通常会更慢，但它的意义本来就不只是快，而是：

> 用本地可控、低依赖、低泄露的方式完成任务。
