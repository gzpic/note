# 为什么要做冗余消除

## 1. 核心原因

原始计算图通常不是最简图。

模型在导出、转换、常量折叠、子图替换之后，经常会留下很多：

- 对最终输出没有贡献的节点
- 数学上等价但工程上多余的节点
- 只会增加调度和访存开销的中间结构

所以需要做冗余消除。

一句话：

> 冗余消除的目标，是删除对最终结果没有实际贡献、但会消耗执行成本的图结构。

## 2. 冗余通常从哪里来

### 2.1 训练图转推理图的残留

训练阶段需要，推理阶段不需要的算子，典型如：

- `Dropout`
- `Identity`
- 一些训练辅助节点

例如：

```text
x → MatMul → Dropout → Add → y
```

推理时 `Dropout` 等价于直通，就可以删掉。

### 2.2 导出器自动插入的无用节点

从 PyTorch / Hugging Face 模型导出到 ONNX 时，经常会出现：

- 连续 `Identity`
- 多余 `Reshape`
- 多余 `Transpose`
- 不必要的 `Cast`

例如：

```text
x → Identity → Identity → MatMul
```

这些节点通常都没有实际意义。

### 2.3 常量折叠后留下的死节点

当一段常量子图被提前算完后，原来的中间节点可能就失去作用了。

例如：

```text
Const → Add → Const
```

被折叠成一个常量后，原来那段子图就该删掉。

### 2.4 不再被使用的分支

例如某个中间输出已经不再连接到最终输出，或者某条分支根本没有被消费，那么整段子图都可以删除。

这类通常叫 dead node / dead subgraph。

## 3. 如果不做冗余消除会怎样

### 3.1 增加额外计算

哪怕是 `Identity`、`Reshape`、`Cast` 这种小算子，也可能真的带来执行成本。

### 3.2 增加显存和中间 tensor

多余节点意味着：

- 多余中间张量
- 多余生命周期管理
- 多余访存

### 3.3 增加 kernel launch

在端侧尤其明显。

例如 Jetson Orin Nano 这类平台上，很多“小节点”的代价不只是算力，而是 launch latency 和调度开销。

### 3.4 阻碍后续融合

比如原本可以融合成：

```text
MatMul + Add + Gelu
```

但中间如果夹了：

```text
Identity / Cast / Reshape
```

就可能破坏 pattern match，导致融合失败。

## 4. 常见的冗余消除类型

常见 pass 包括：

- Dead node elimination
- Identity elimination
- Redundant reshape elimination
- Redundant cast elimination
- Duplicate subgraph elimination

这些优化经常和常量折叠一起出现。

## 5. 为什么它通常放在优化前面

因为很多更强的优化，都依赖图先变干净。

常见顺序通常类似：

1. 冗余消除
2. 常量折叠
3. 模式融合
4. layout 优化
5. 内存优化

先把垃圾节点清掉，后面的融合和子图下沉才更容易命中。

## 6. 在 LLM 里的一个直观例子

Transformer 图里经常出现：

- `Reshape`
- `Transpose`
- 再 `Reshape`

有些导出后的组合前后其实互相抵消，或者对最终布局没有实质影响。

如果不清理：

- 节点数变多
- launch 变多
- 融合机会变少

## 7. 面试速记版

- 原始 ONNX 图通常不是最简图
- 冗余来自导出残留、常量折叠后死节点、无用分支等
- 冗余消除能减少节点、launch、访存，并给后续融合铺路
