**一、总体设计哲学（C++ 版不变量）**

**1️⃣ Runner 是「物理执行器」，不是逻辑体**

**Runner 永远不知道：**

- 当前是第几轮对话
- token 的语义位置
- 谁的请求
- cache 能不能复用
- 需不需要回收

**Runner 只接受：**

- 物理指针
- 物理 block 表
- slot mapping
- 预先算好的 mask

📌 **任何“判断 / 决策 / 策略”必须在 Scheduler / KV Manager 完成**

**2️⃣ 绝对无状态（Stateless + Idempotent）**

- 每次 run_*() 都是**独立的**
- 任意一次调用都可：
    - 被打断
    - 被丢弃
    - 被重试

**禁止 Runner 内部存在：**

- step 计数器
- ring head / tail
- radix tree 节点
- reference count

**3️⃣ 物理与语义彻底解耦（核心）**

| **维度** | **谁负责** |
| --- | --- |
| 语义时间（token idx） | Scheduler |
| 前缀匹配 | Radix Tree |
| block 回收策略 | KV Manager |
| block 物理位置 | Physical Layer |
| kernel 执行 | Runner |

Runner **只看物理地址，不看“历史”**。

**二、物理层（Physical Layer / Anchor）**

**一旦初始化完成，终身不可变**

**4️⃣ 一次性资源占有（One-time Reservation）**

Runner 启动阶段必须完成：

- KV Cache（Int8 / FP16）
- KV Scales
- Static Input Buffers
- Pinned Output Buffer
- CUDA Streams
- CUDA Graph Executable

**运行期红线：**

❌ 禁止 malloc / new❌ 禁止 cudaMalloc❌ 禁止 Tensor resize

**5️⃣ KV Cache 是「物理池」，不是容器**

- 连续内存
- 固定 Block Size（16 / 32）
- Block ID → 物理偏移 = O(1)

**重要：**

Ring Buffer / Radix Tree**都是“地址分配策略”，不属于 Runner**

**6️⃣ Zero-Copy 是强约束，不是优化项**

- Logits 直接写入 pinned host memory
- CPU 通过指针读取
- **不允许 cudaMemcpy D2H**

这是 UMA 平台的**硬约束**。

**三、三态 Executor（逻辑层）**

Runner 内部逻辑**只分三种执行路径**。

**7️⃣ Prefill Executor（吞吐优先）**

**目标：**

最大化 FLOPS，容忍延迟

**特征：**

- 动态 kernel enqueue
- 标准 stream（非高优）
- 无 CUDA Graph

**StreamLLM 关键点：**

**Compute-All, Store-Sparse**

- Attention 看到完整上下文
- KV Cache 只写入 mask 允许的 block

📌 Prefill 不参与 Ring 回收

**8️⃣ Decode Executor（延迟优先）**

**目标：**μs 级稳定延迟

**特征：**

- CUDA Graph Replay
- 高优先级 Stream
- 固定 kernel 数 / 顺序 / grid

**Graph 不变量：**

- Batch = 1
- Block Size 固定
- 指针地址固定
- shape 不变

**运行期唯一变化：**

- input buffer 内容
- block_tables 内容

**9️⃣ Agent Executor（复用 / Tiered）**

**职责：**

- 把“前缀匹配”翻译成“物理数据是否在 GPU”
- 不在 → DMA
- 在 → suffix prefill

**Agent Runner 永远不做：**

- prefix 匹配
- cache 淘汰决策

它只是 **Proxy + DMA 发射器**。

**四、调度与执行的边界红线**

**🔴 Runner 绝不跨 Step Boundary**

- block 覆写
- radix tree 变更
- ring 回收

**只能发生在：**

上一个 decode kernel 完全结束之后

**🔴 Runner 不拥有任何线程模型**

- 不起线程池
- 不轮询
- 不 sleep

所有并发都交给：

- CUDA Stream
- DMA
- Scheduler

**五、CUDA Graph 相关要点（C++ 关键）**

**10️⃣ Decode Graph 必须“单一职责”**

- 只服务 decode
- 不混 prefill
- 不混 agent

Graph 越小越稳定。

**11️⃣ Graph Capture 的时机是“系统态”**

- 所有 allocator 已稳定
- 所有 Tensor 地址固定
- 无 Python / PyTorch 干扰

📌 这是 C++ 比 Python 强的根本原因。

**六、Tiered Storage（Runner 视角）**

Runner 对 Tiered Cache 的**唯一认知**：

| **状态** | **行为** |
| --- | --- |
| 在 GPU | 直接用 |
| 不在 GPU | 发 async load |
| load 未完成 | 返回 PENDING |

**不做：**

- 等待
- 阻塞
- fallback

**七、Orin 平台专属要点**

**12️⃣ Decode Stream 必须 High Priority**

防止：

- Camera
- ISP
- NVENC
- TTS

抢占 decode kernel。

**13️⃣ 频率锁定是 Runner 生命周期的一部分**

Runner init → 锁频Runner destroy → 解锁

目的：**延迟确定性，不是极限性能**。

**14️⃣ 内存对齐是“功能正确性”**

- Block Size = 16 / 32
- L2 Cache Line 对齐
- Warp 对齐

不是优化，是**避免隐藏退化**。
