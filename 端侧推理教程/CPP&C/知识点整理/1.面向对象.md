# 1.1 C++ 语言的特点

## 核心特点

- C++ 在 C 的基础上发展而来，既保留了接近底层、效率高的特点，也引入了面向对象、泛型编程和现代类型系统。
- C++ 支持多范式编程：
  - 过程式
  - 面向对象
  - 泛型编程
  - 函数式风格
- C++ 兼顾抽象能力和性能，常用于对性能和资源控制要求高的场景，例如系统软件、游戏引擎、数据库、音视频、推理框架等。

## 常见面试回答点

- 面向对象能力强：支持封装、继承、多态。
- 泛型能力强：模板机制强，标准库 STL 复用度高。
- 性能高：编译为原生机器码，通常接近 C 的性能。
- 资源控制细：可以直接控制对象生命周期、内存布局、构造析构过程。
- 类型系统更强：支持 `const`、引用、函数重载、模板、`constexpr`、类型转换运算符等。
- 语言持续演进：C++11/14/17/20/23 不断增强现代开发能力。

## 需要纠正的表述

- “C++ 只比汇编慢 10%~20%” 不是通用结论，过于绝对。实际性能取决于编译器优化、代码写法、内存访问模式和具体场景。
- “C++ 更安全” 也不能一概而论。C++ 提供了更多安全工具，例如 RAII、智能指针、`const`、类型转换、异常机制，但如果滥用裸指针或手动管理资源，仍然可能出错。

## Modern C++ 常见新增特性

- C++11：`nullptr`、`auto`、Lambda、右值引用、移动语义、智能指针、`thread`
- C++14/17：泛型 Lambda、结构化绑定、`if constexpr`、内联变量、`std::optional`
- C++20：概念、协程、范围库、模块

# 1.2 C 语言的特点

## 核心特点

- 语法简洁，接近硬件。
- 可移植性强，适合跨平台移植。
- 编译后运行效率高，额外运行时负担小。
- 可以直接操作地址、寄存器映射区和位级数据。
- 适合作为系统编程和嵌入式开发基础语言。

## 常见应用场景

- 操作系统
- 驱动程序
- MCU 裸机开发
- RTOS 内核与底层模块
- 对性能和内存占用敏感的库

## 注意

- C 语言“能直接操作底层寄存器”本质上通常是通过地址映射、指针和 `volatile` 完成，而不是语言层面天然提供寄存器对象。
- C 高效，但类型系统和抽象能力相对弱，代码规模大时更依赖程序员规范。

# 1.3 为什么嵌入式里 C 用得多

## 主要原因

- 离硬件近：可以方便操作寄存器、内存映射 IO、中断控制器。
- 性能和体积可控：适合 Flash/RAM 紧张的环境。
- 编译器和工具链成熟：几乎所有 MCU/SoC 平台都有成熟 C 编译器。
- 可读性与开发效率优于纯汇编。
- 便于跨芯片、跨平台移植。

## C 和汇编的分工

- 启动代码、上下文切换、部分中断入口、特殊寄存器操作常用汇编。
- 外设驱动逻辑、协议栈、业务流程、算法部分通常用 C 实现。

## 一句话总结

- 汇编负责“最贴硬件、最敏感”的部分，C 负责“可维护、可移植、能表达复杂逻辑”的部分。

# 1.4 面向对象的四大特征

## 1. 抽象

- 从具体事物中提炼出稳定的共性，忽略不重要细节。
- 目的是降低复杂度，先看“是什么、能做什么”，再看“怎么做”。

例如：

- 把“汽车、公交车、卡车”抽象成“车辆”。

## 2. 封装

- 把数据和操作数据的方法放在一起，对外暴露必要接口，隐藏内部实现细节。
- 目的是保护对象状态、降低耦合、方便后续修改实现。

C++ 体现方式：

- `public / protected / private`
- 类对外暴露成员函数，对内隐藏成员数据

## 3. 继承

- 子类复用父类的成员和接口，并可扩展自己的行为。
- 体现“is-a”关系。

例如：

- `Dog` 继承 `Animal`

## 4. 多态

- 同一接口，在不同对象上表现出不同实现。

分类：

- 编译期多态：函数重载、模板
- 运行期多态：继承 + 虚函数 + 父类指针/引用指向子类对象

# 1.5 面向对象的五大原则

## 1. 单一职责原则

- 一个类只负责一类职责，变化原因应该只有一个。

## 2. 开放封闭原则

- 对扩展开放，对修改关闭。
- 新需求尽量通过新增代码完成，而不是频繁修改稳定代码。

## 3. 里氏替换原则

- 子类必须能够替换父类而不破坏程序正确性。
- 子类不能违背父类对外承诺的语义。

## 4. 依赖倒置原则

- 高层模块不依赖低层模块，二者都应依赖抽象。
- 面向接口编程，而不是面向具体实现编程。

## 5. 接口隔离原则

- 不应强迫使用者依赖它不需要的接口。
- 大接口要拆成小而专的接口。

# 1.6 面向过程 vs 面向对象

## 面向过程

特点：

- 以函数和步骤为中心
- 强调“先做什么，再做什么”

优点：

- 结构直接
- 性能开销通常更低
- 对简单问题实现快

缺点：

- 代码规模变大后，可维护性和扩展性较差
- 数据和行为分离，修改时容易牵一发而动全身

## 面向对象

特点：

- 以对象为中心
- 强调“谁来做这件事”

优点：

- 易维护
- 易复用
- 易扩展

缺点：

- 抽象不合理时反而会增加复杂度
- 某些场景下会引入额外间接层和运行期开销

## 一句话理解

- 面向过程重“步骤”，面向对象重“职责和协作”。

# 1.7 静态绑定和动态绑定

## 静态类型与动态类型

- 静态类型：变量在声明时的类型，编译期确定。
- 动态类型：指针或引用实际指向对象的真实类型，运行期确定。

例如：

```cpp
Base* p = new Derived();
```

- `p` 的静态类型是 `Base*`
- `*p` 指向对象的动态类型是 `Derived`

## 静态绑定

- 函数调用在编译期确定。
- 常见于普通成员函数、重载函数、模板实例化。

## 动态绑定

- 函数调用在运行期根据对象真实类型决定。
- 常见于虚函数调用。

例如：

```cpp
struct Base {
    virtual void show() { std::cout << "Base\n"; }
};

struct Derived : Base {
    void show() override { std::cout << "Derived\n"; }
};
```

通过 `Base*` 或 `Base&` 调用 `show()` 时，若函数是虚函数，则发生动态绑定。

# 1.8 重载、重写、隐藏

## 重载（Overload）

- 同一作用域内
- 函数名相同
- 参数列表不同
- 与返回类型无关

例如：

```cpp
void print(int);
void print(double);
```

## 重写（Override）

- 发生在继承体系中
- 子类重写父类虚函数
- 函数名、参数列表、`const` 限定等需要匹配

例如：

```cpp
struct Base {
    virtual void f(int);
};

struct Derived : Base {
    void f(int) override;
};
```

## 隐藏（Hiding）

- 子类定义了与父类同名函数时，父类同名函数可能被隐藏
- 即使参数不同，也会先发生名字隐藏

例如：

```cpp
struct Base {
    void f(int);
    void f(double);
};

struct Derived : Base {
    void f(char);
};
```

此时 `Derived` 中的 `f` 会把 `Base` 中的同名重载集隐藏掉。若想重新引入，可写：

```cpp
using Base::f;
```

## 面试一句话区分

- 重载：同一层里“同名不同参”
- 重写：子类改写父类虚函数
- 隐藏：子类同名成员把父类同名成员遮住

# 1.9 C 语言如何“模拟”重载

C 本身不支持 C++ 式函数重载，因为同名函数的符号不能仅靠参数列表区分。

常见替代思路：

- 通过不同函数名区分，例如 `print_int`、`print_double`
- 通过宏分发
- 通过可变参数
- 在 GCC/Clang 下使用扩展能力

## 面试回答建议

- 不要说 “C 可以真正实现重载”
- 更准确地说法是：C 只能通过命名规则、宏或编译器扩展模拟类似效果

# 1.10 构造函数有哪些，分别做什么

常见可回答为：

- 默认构造函数
  - 无参数，负责默认初始化对象
- 普通构造函数
  - 也常叫带参构造函数，用给定参数初始化对象
- 拷贝构造函数
  - 用同类型已有对象初始化新对象
- 移动构造函数
  - 从右值对象“接管资源”
- 转换构造函数
  - 单参数构造函数可参与隐式类型转换，通常建议配合 `explicit`

## 示例

```cpp
struct A {
    A() = default;
    A(int x) : value(x) {}
    A(const A& other) = default;
    A(A&& other) noexcept = default;

    int value = 0;
};
```

# 1.11 类默认会生成哪些特殊成员函数

面试里常答“六个特殊成员函数”：

- 默认构造函数
- 析构函数
- 拷贝构造函数
- 拷贝赋值运算符
- 移动构造函数
- 移动赋值运算符

## 注意

- 是否“真的被自动生成、能否使用、是否被删除”受类成员、基类以及你是否手写了某些函数影响。
- 老面试题有时只说“四个”：默认构造、析构、拷贝构造、拷贝赋值，这是 C++11 之前更常见的说法。

# 1.12 只定义析构函数，会自动生成哪些函数

面试里不要再简单背“只要写了析构函数，就一定还会自动生成默认构造和拷贝构造”。

更稳妥的回答是：

- 编译器是否生成默认构造函数，取决于你是否声明了其他构造函数。
- 用户自定义析构函数会影响移动操作的自动生成。
- 拷贝构造和拷贝赋值在很多情况下仍可能被声明出来，但是否可用还要看成员和基类。

## 面试简化答法

- 如果用户没有声明构造函数，编译器通常会生成默认构造函数。
- 如果只手写析构函数，拷贝相关函数通常仍可能存在，但移动构造/移动赋值往往不会按你期待那样自动生成。
- 现代 C++ 建议直接使用 `=default` 或 `=delete` 明确表达意图。

# 1.13 如何禁止拷贝或禁止实例化

## 禁止拷贝

现代 C++ 推荐：

```cpp
struct NonCopyable {
    NonCopyable() = default;
    NonCopyable(const NonCopyable&) = delete;
    NonCopyable& operator=(const NonCopyable&) = delete;
};
```

旧写法是把拷贝构造和拷贝赋值放到 `private`，但不如 `= delete` 直观。

## 禁止实例化

常见方法：

- 做成抽象类，包含纯虚函数
- 把构造函数设为 `private` 或 `protected`
- 只提供静态接口，把构造析构都禁掉

# 1.14 成员初始化列表与成员初始化顺序

## 两种初始化方式

- 构造函数体内赋值
- 成员初始化列表初始化

例如：

```cpp
struct A {
    A(int x) : value(x) {}
    int value;
};
```

## 哪些情况必须用初始化列表

- 引用成员
- `const` 成员
- 成员对象没有默认构造函数时
- 需要显式调用基类带参构造函数时

## 为什么初始化列表通常更好

- 对类类型成员，初始化列表通常是“直接构造”
- 构造函数体内赋值往往意味着“先默认构造，再赋值”
- 因此初始化列表更自然，也可能更高效

## 初始化顺序由谁决定

不是由初始化列表里写的顺序决定，而是由成员声明顺序决定。

例如：

```cpp
struct A {
    A() : b(2), a(1) {}
    int a;
    int b;
};
```

实际初始化顺序仍然是先 `a`，后 `b`。

# 1.15 构造函数执行顺序

单继承或普通类中，一般顺序是：

1. 虚基类
2. 普通基类
3. 成员对象
4. 自身构造函数体

## 多继承时

- 基类构造顺序按继承列表中的声明顺序
- 成员对象构造顺序按成员声明顺序
- 析构顺序与构造顺序相反

## 多继承的优缺点

- 优点
  - 可以复用多个基类接口或能力
- 缺点
  - 容易带来名字冲突、二义性和菱形继承问题
  - 设计和维护复杂度更高

## 作为开发者怎么看多继承

- 业务代码里通常更偏向“组合优于继承”
- 多继承更适合接口型继承、Mixin 等相对可控场景

# 1.16 向上转型和向下转型

## 向上转型

- 子类转父类
- 通常安全
- 可隐式完成

```cpp
Derived d;
Base* p = &d;
```

## 向下转型

- 父类转子类
- 不一定安全

### `dynamic_cast`

- 适用于带虚函数的多态类型
- 运行期检查，更安全

```cpp
Base* p = new Derived();
Derived* d = dynamic_cast<Derived*>(p);
```

### `static_cast`

- 编译期转换，不做运行期检查
- 若实际对象不是目标子类，可能出错

## 纠正一点

- 向上转型通常不需要 `dynamic_cast`
- 题目若问“安全的向下转型”，优先答 `dynamic_cast`

# 1.17 C++ 中的多态

## 静态多态

- 函数重载
- 运算符重载
- 模板

编译期决定调用目标。

## 动态多态

需要同时满足：

- 继承
- 虚函数
- 基类指针或引用指向派生类对象

运行期通过虚表机制决定调用哪个实现。

# 1.18 虚析构、为什么不能虚构造

## 为什么要虚析构

如果一个类会被当作基类使用，并且可能通过基类指针删除派生类对象，那么析构函数应声明为虚函数。

```cpp
struct Base {
    virtual ~Base() = default;
};
```

否则：

- `Base* p = new Derived();`
- `delete p;`

可能只调用基类析构，导致派生类资源未释放。

## 为什么构造函数不能是虚函数

- 构造对象时，对象的真实类型本来就已经明确
- 调用虚函数需要对象里已经有可用的 `vptr`
- 但构造阶段对象本身还在建立过程中

所以 C++ 不支持虚构造函数。

## 构造/析构函数里能不能调用虚函数

- 语法上可以
- 但通常不会得到我们直觉中的多态效果

原因是：

- 构造期间，对象只被当作当前正在构造的那一层类型
- 析构期间，更派生层部分通常已经销毁
- 因此调用虚函数时，通常只会落到当前层级对应的版本

## 纯虚析构

- 纯虚析构函数可以存在
- 但仍然必须提供定义

```cpp
struct Base {
    virtual ~Base() = 0;
};

inline Base::~Base() {}
```

# 1.19 虚表、`vptr`、写入时机

## 虚表里有什么

- 本类虚函数入口地址
- 在某些实现里还可能包含 RTTI 等相关信息

## 虚表何时形成

- 类的虚表通常在编译期由编译器生成

## `vptr` 何时写入对象

- 在对象构造过程中，由编译器生成的构造代码负责写入 `vptr`

## 面试稳妥说法

- “虚表整体是编译器生成的”
- “对象中的 `vptr` 在构造阶段被设置”

不要把实现细节说得过死，例如“一定在构造函数之前某一固定指令点完成二次写入”。

# 1.20 构造函数里调用虚函数为什么没有多态效果

语法上允许，但通常达不到我们直觉中的“派生类多态效果”。

原因是：

- 执行基类构造函数时，派生类部分尚未构造完成
- 此时对象会被视为当前正在构造的那一层类型
- 因此基类构造函数中调用虚函数，通常只会调用基类版本

析构阶段同理：

- 析构进入基类阶段后，派生类部分已经被销毁
- 再调用虚函数也不会分派到派生类版本

# 1.21 引用能否实现动态绑定

可以，但前提仍然是：

- 绑定的是基类引用
- 调用的是虚函数
- 实际引用到的是派生类对象

例如：

```cpp
Base& ref = derived;
ref.show();  // 若 show 是虚函数，则发生动态绑定
```

本质上，动态绑定取决于“虚函数 + 对象真实类型”，不是取决于“指针还是引用”。

# 1.22 `struct` 和 `class` 的区别

在 C++ 里，`struct` 和 `class` 本质上都可以定义类类型，主要区别有两点：

- `struct` 默认访问权限是 `public`
- `class` 默认访问权限是 `private`

以及：

- `struct` 默认继承权限是 `public`
- `class` 默认继承权限是 `private`

## 其他说明

- 在 C++ 中，`struct` 也可以有成员函数、构造函数、继承、模板等。
- 工程习惯上：
  - `struct` 常用于数据聚合
  - `class` 常用于强调封装和行为

# 1.23 C++ 的 `struct` 和 C 的 `struct` 有什么区别

C 里的 `struct`：

- 主要是数据聚合
- 不能直接定义成员函数
- 没有构造函数、析构函数、访问控制、继承、多态等机制

C++ 里的 `struct`：

- 可以有成员函数、构造析构、继承、模板、访问控制
- 只是默认访问权限与 `class` 不同

## 一个面试常见点

- C 里使用结构体变量通常要写 `struct Node n;`
- C++ 里定义完 `struct Node { ... };` 后，可直接写 `Node n;`

# 1.24 `this` 指针

## 什么是 `this`

- `this` 是非静态成员函数中的隐式参数
- 指向当前调用该成员函数的对象

例如：

```cpp
class A {
public:
    void set(int x) { this->value = x; }
private:
    int value = 0;
};
```

## 特点

- 只能在非静态成员函数中使用
- 静态成员函数没有 `this`
- `this` 的类型通常可理解为 `ClassType* const`
- 在 `const` 成员函数中，`this` 可理解为 `const ClassType* const`

## 补充

- `this` 不属于对象内存中的“普通成员”
- 它是编译器为非静态成员函数隐式传入的参数
- 常见用途：
  - 区分成员变量和同名局部变量
  - 返回当前对象本身，例如 `return *this;`

## 普通成员函数 vs 静态成员函数

| 对比项 | 普通成员函数 | 静态成员函数 |
| --- | --- | --- |
| 是否有 `this` | 有 | 没有 |
| 是否属于对象 | 是 | 否，属于类 |
| 是否能访问普通成员变量 | 能 | 不能直接访问 |
| 是否能访问静态成员变量 | 能 | 能 |
| 调用方式 | 对象调用 | 类名调用 |

## 一个容易答错的点

- 不是“指针不能重绑定且必须初始化”
- 正好相反：
  - 指针可以改指向，也可以先定义后赋值，只是未初始化指针危险
  - 引用必须初始化，且绑定后不能改绑

# 1.25 在成员函数里调用 `delete this` 会怎样

可以编译，但风险很高。

## 可能问题

- 当前对象必须确实是通过 `new` 创建的，否则行为未定义
- `delete this` 后，对象生命周期结束，后续再访问对象就是未定义行为
- 若函数后续还继续访问成员、调用虚函数、返回后外部继续使用该对象，都会有风险

## 什么时候勉强可用

- 极少数受控设计里，对象明确由堆分配并自管理生命周期

## 面试建议

- 直接回答：能写，但非常危险，除非你百分之百确认对象来自堆且后续绝不会再被访问

## 如果在析构函数里调用 `delete this`

- 这通常会导致无限递归
- 因为 `delete this` 会再次触发当前对象析构函数
- 析构函数里又执行 `delete this`，于是反复递归
- 最终可能导致栈溢出或程序崩溃

所以：

- 析构函数里绝对不应该再 `delete this`

# 1.26 类/对象的“加载过程”应如何理解

“类的加载过程”不是标准 C++ 精确定义术语，面试中一般是想问对象创建过程。

可以回答为：

1. 为对象分配内存
2. 若有虚函数，构造阶段设置 `vptr`
3. 先构造基类部分
4. 再构造成员对象
5. 最后执行当前类构造函数体

如果题目问的是“类成员/静态成员何时出现”：

- 静态数据成员不属于对象，生命周期独立于对象
- 普通成员要等对象创建时才真正占据对象内存

# 1.27 类对象大小受哪些因素影响

主要受以下因素影响：

- 非静态数据成员大小
- 内存对齐和填充字节
- 是否有虚函数，从而引入 `vptr`
- 是否继承基类，基类子对象也占空间

## 不影响对象大小的内容

- 普通成员函数
- 静态成员函数
- 静态数据成员本身

# 1.28 普通函数和虚函数的区别

## 普通函数

- 调用目标通常在编译期确定
- 不依赖虚表
- 没有运行期多态分派开销

## 虚函数

- 通过虚表机制支持运行期多态
- 需要对象中存在 `vptr`
- 通常会多一次间接调用成本

## 面试简化答法

- 普通函数是静态绑定为主
- 虚函数支持动态绑定

# 1.29 常对象

## 什么是常对象

- 被 `const` 修饰的对象就是常对象

```cpp
const A a;
```

## 特点

- 不能修改其普通成员
- 只能调用 `const` 成员函数
- `mutable` 成员是例外

# 1.30 析构函数能抛出异常吗

不建议抛出，通常应避免。

原因：

- 析构函数常用于释放资源
- 若异常传播过程中又触发另一个析构函数抛异常，程序可能直接 `std::terminate`

## 面试稳妥回答

- 析构函数理论上可以写出抛异常代码
- 但工程实践里应尽量保证析构函数不抛异常
- 若析构里有失败风险，通常应在析构前显式提供单独的 `close/release` 接口处理错误

## 构造函数能抛异常吗

- 可以
- 若构造过程中抛异常，对象会被视为未构造完成
- 已经成功构造的基类和成员对象会按规则自动析构
- 但当前对象不会以“完整对象”的方式再额外执行一次析构

# 1.31 继承时访问权限规则

基类成员本身有：

- `public`
- `protected`
- `private`

子类对基类成员的访问首先遵循：

- 可访问基类的 `public` 和 `protected`
- 不可直接访问基类的 `private`

## 不同继承方式的效果

### `public` 继承

- 基类 `public` 到子类里仍是 `public`
- 基类 `protected` 到子类里仍是 `protected`

### `protected` 继承

- 基类 `public` 到子类里变成 `protected`
- 基类 `protected` 仍是 `protected`

### `private` 继承

- 基类 `public` 到子类里变成 `private`
- 基类 `protected` 到子类里也变成 `private`

## 一句话

- `private` 成员永远不能被子类直接访问
- 继承方式主要影响“基类可继承成员在子类中的对外可见性”

# 1.32 继承是否破坏封装

面试里可以这样答：

- 从严格意义上说，继承会让子类接触到基类的实现细节，尤其是 `protected` 成员，因此会削弱一部分封装
- 但继承本身是语言提供的受控机制，并不等于完全破坏封装

## 更稳妥表述

- 继承在提高复用性的同时，会增加耦合
- 如果滥用继承，确实容易伤害封装性
- 因此工程上常强调“组合优于继承”

# 1.33 常函数

常函数就是成员函数后面带 `const`：

```cpp
class A {
public:
    int get() const { return value; }
private:
    int value = 0;
};
```

## 作用

- 承诺该函数不修改对象的非静态成员状态
- 让 `const` 对象也能调用该函数
- 改善接口语义和可读性

# 1.34 虚继承 / 菱形继承

## 什么是菱形继承

```cpp
struct A {};
struct B : A {};
struct C : A {};
struct D : B, C {};
```

此时 `D` 中会有两份 `A` 子对象。

## 带来的问题

- 空间浪费
- 访问 `A` 成员时产生二义性

## 解决办法：虚继承

```cpp
struct A {};
struct B : virtual A {};
struct C : virtual A {};
struct D : B, C {};
```

这样最底层派生类中只保留一份共享的虚基类子对象。

## 一句话

- 虚继承是为了解决菱形继承中的重复基类子对象和二义性问题

# 1.35 虚函数、纯虚函数与抽象类

## 虚函数

- 基类中用 `virtual` 声明
- 派生类可重写
- 通过基类指针/引用调用时支持动态绑定

## 纯虚函数

```cpp
virtual void show() = 0;
```

- 表示基类只给接口，不给默认实现要求
- 含有纯虚函数的类通常是抽象类

## 抽象类

- 含有至少一个纯虚函数
- 不能直接实例化

## 派生类要不要实现纯虚函数

- 如果派生类想变成可实例化的具体类，就必须实现继承来的纯虚函数
- 否则派生类仍然是抽象类

## 纯虚函数能有定义吗

- 可以
- 只是声明处仍要写 `= 0`
- 纯虚析构函数就是最常见例子

# 1.36 虚函数实现原理

常见实现可概括为：

- 含虚函数的类通常有一张虚函数表
- 对象内部通常有一个 `vptr` 指向该表
- 调用虚函数时，通过 `vptr` 找到实际应调用的函数地址

## 注意

- 这是主流实现思路
- 具体内存布局属于实现细节，不应背得过死

# 1.37 虚函数和纯虚函数的区别

- 虚函数可以有实现
- 纯虚函数用 `= 0` 声明，强调接口性质
- 含纯虚函数的类是抽象类，不能直接实例化
- 纯虚函数在派生类中若不实现，派生类也仍然抽象

# 1.38 虚函数开销

## 空间开销

- 类通常需要虚表
- 对象通常会多一个 `vptr`

## 时间开销

- 调用虚函数通常会多一次间接寻址

## 面试回答建议

- 虚函数有开销，但通常不是性能瓶颈
- 真正要看热点路径和实际 profiling

# 1.39 手动调用虚表函数

理论上在特定编译器实现下，可以通过对象里的 `vptr` 找到虚表，再强行取函数指针调用。

但要注意：

- 这是严重依赖编译器实现的非可移植技巧
- 不属于标准 C++ 正式用法
- 面试里最多说明“理论上可以，工程上不应这么做”

# 1.40 拷贝构造函数的参数为什么通常是 `const T&`

典型写法：

```cpp
ClassName(const ClassName& other);
```

原因：

- 如果按值传递，会先为了传参调用一次拷贝构造
- 这会再次触发拷贝构造本身，形成无限递归
- 用引用可以避免额外拷贝
- 用 `const` 可以接受常对象和临时对象

# 1.41 什么情况下会调用拷贝构造函数

常见场景：

- 用一个已有对象初始化另一个对象
- 值传递方式传入类对象
- 函数按值返回对象时，在未发生拷贝省略或移动时

例如：

```cpp
A a;
A b = a;
```

## 注意

- 现代 C++ 中经常发生 RVO/NRVO 或移动构造
- 因此“理论上会调用”不等于“运行时一定实际发生”

# 1.42 拷贝赋值和移动赋值

## 拷贝赋值

```cpp
A& operator=(const A& other);
```

- 用已有对象的内容复制给另一个已存在对象

## 移动赋值

```cpp
A& operator=(A&& other) noexcept;
```

- 把右值对象持有的资源转移给当前对象
- 常用于避免深拷贝

## 和构造的区别

- 构造：创建新对象时发生
- 赋值：对象已经存在，再给它新值

# 1.42.1 构造函数、拷贝构造函数和赋值运算符的区别

- 构造函数
  - 对象还不存在，负责把对象创建出来
- 拷贝构造函数
  - 对象还不存在，但要用另一个同类型对象初始化它
- 赋值运算符
  - 对象已经存在，把别的对象内容赋给它

## 一句话

- 初始化新对象：看构造/拷贝构造
- 给已有对象重新赋值：看赋值运算符

# 1.43 类成员访问权限

类成员访问权限有三种：

- `public`
- `protected`
- `private`

## 常见理解

- `public`
  - 类外可访问
- `protected`
  - 类外不可直接访问，但子类可访问
- `private`
  - 只有本类内部可直接访问

# 1.44 如何理解抽象类

- 含有至少一个纯虚函数的类通常是抽象类
- 抽象类不能直接实例化
- 主要用途是作为接口基类，约束派生类实现行为

## 可以做什么

- 定义指针和引用
- 作为多态基类使用

# 1.45 什么是多态，除了虚函数还有什么方式

## 多态的本质

- 同一接口，不同对象表现不同实现

## 运行期多态

- 继承 + 虚函数 + 基类指针/引用

## 编译期多态

- 函数重载
- 运算符重载
- 模板

所以除了虚函数，模板和重载也能实现多态，只不过属于编译期多态。

# 1.46 虚基类能否被实例化

虚基类和抽象类不是一回事。

- 虚基类：强调它在继承体系中被“虚继承”
- 抽象类：强调它含纯虚函数，不能实例化

因此：

- 虚基类如果不是抽象类，就可以实例化

# 1.47 哪些函数不能声明为虚函数

常见面试回答：

- 普通非成员函数
- 静态成员函数
- 构造函数
- 友元函数

## 关于内联函数

- 内联成员函数可以同时是虚函数
- 只是当通过多态方式调用时，运行期动态绑定通常使它难以像普通内联那样展开

所以“内联函数不能是虚函数”这个说法不准确。

# 1.47.1 什么情况下编译器会合成默认构造函数

更准确地说，编译器在一些需要初始化语义的场景下会隐式声明默认构造函数，例如：

- 成员对象需要默认构造
- 基类需要默认构造
- 类里有虚函数
- 类里有虚基类

## 注意

- 合成默认构造函数主要是为了完成必要初始化
- 它不会神奇地把所有内置类型成员都自动初始化成你期望的值

# 1.47.2 什么情况下会合成拷贝构造函数

- 用户没有显式声明拷贝构造函数时，编译器可能会隐式声明一个
- 它通常按成员和基类逐个拷贝

## 面试稳妥说法

- 现代 C++ 下特殊成员函数的生成/删除规则比较细
- 面试里答“编译器会按成员和基类需要来隐式生成，但是否可用还受类定义影响”会更稳

# 1.48 友元

友元用于打破类的访问限制，让指定函数或类可以访问该类的 `private` / `protected` 成员。

## 友元函数

```cpp
class A {
    friend void show(const A&);
private:
    int x = 0;
};
```

## 友元类

```cpp
class B;

class A {
    friend class B;
};
```

## 特点

- 友元不是成员函数
- 友元关系不具有传递性
- 友元关系不具有继承性

## 面试回答建议

- 友元提高了灵活性
- 但会破坏封装，应谨慎使用

# 1.49 空类

- 空类大小通常是 1 字节
- 目的是保证不同对象拥有不同地址

## 注意

- 若空类带虚函数，通常还会因为 `vptr` 带来额外大小
