# 2.1 什么是模板

- 模板是 C++ 实现泛型编程的核心机制。
- 它让我们可以把“类型”参数化，写出可复用的类或函数。

常见形式：

- 函数模板
- 类模板

例如：

```cpp
template <typename T>
T my_max(const T& a, const T& b) {
    return a < b ? b : a;
}
```

# 2.1.1 类模板和模板类的区别

- 类模板：模板本身，是带类型参数的“类蓝图”
- 模板类：类模板在某个具体类型下实例化后的结果

例如：

```cpp
template <typename T>
class Box {};
```

这里 `Box` 是类模板。

而：

```cpp
Box<int> b;
```

这里的 `Box<int>` 对应具体实例，可理解为模板类实例。

# 2.1.2 函数模板和类模板的区别

- 函数模板
  - 常可根据实参自动推导模板参数
  - 也可以显式指定模板参数
- 类模板
  - 使用时通常需要显式写出模板实参

## 面试稳妥说法

- 老面试题常答“函数模板支持自动推导，类模板通常不行”
- 现代 C++ 里也有类模板实参推导（CTAD）这类特例，所以不要说得太绝对

# 2.2 模板类/模板函数是在什么时候实现的

更准确地说法是：

- 模板本身在看到定义时并不会立刻为所有类型生成代码
- 当编译器发现某个具体类型被使用时，才会进行模板实例化

## 两种常见实例化方式

- 隐式实例化
  - 编译器根据使用点自动推导并生成实例
- 显式实例化
  - 程序员明确指定要为某种类型生成实例

例如：

```cpp
template class std::vector<int>;
```

## 底层实现可怎么答

- 编译器不会生成“一份能处理任意类型的万能函数”
- 而是：
  - 先检查模板本身语法
  - 在具体使用点按实际类型实例化
  - 生成对应类型版本的代码

# 2.3 模板具体化 / 特化

当通用模板对某些类型不合适时，可以为特定类型提供专门实现。

## 常见形式

- 全特化
- 偏特化（类模板常见）

例如：

```cpp
template <typename T>
struct Printer {
    void print() {}
};

template <>
struct Printer<int> {
    void print() {}
};
```

## 一句话理解

- 实例化：把模板“套上具体类型”
- 特化：给某些特殊类型“开小灶”

# 2.4 模板的优缺点

## 优点

- 代码复用高
- 类型安全
- 零运行时抽象开销

## 缺点

- 编译错误信息可能复杂
- 可能导致编译时间变长
- 模板实例过多时可能增大二进制体积

# 2.5 面试常见补充

## 为什么模板通常放在头文件里

- 因为使用点实例化时，编译器需要看到完整模板定义

## 模板和宏的区别

- 模板有类型系统支持
- 宏只是文本替换

## 模板和多态的区别

- 模板多态偏编译期
- 虚函数多态偏运行期

## 类模板为什么常分文件编写会出问题

- 模板只有在实例化时才真正生成代码
- 如果模板定义只写在 `.cpp` 里，而使用点看不到完整定义，编译器往往无法实例化

## 常见做法

- 直接把模板声明和定义都放在头文件中
- 或者：
  - 头文件放声明
  - 实现写到 `.tpp/.ipp` 中
  - 最后再被头文件 `#include`
- 少数场景也可配合显式实例化
  - 在 `.cpp` 中写模板定义
  - 再显式实例化出确实要用的具体类型
  - 但这种方式灵活性较差，工程里不如头文件方案常见

## 一句话

- 模板通常要让使用点“看见完整定义”
