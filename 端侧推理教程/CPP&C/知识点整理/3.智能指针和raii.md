# 3.1 智能指针的作用

- 自动管理资源生命周期，减少忘记释放带来的内存泄漏。
- 配合 RAII，让资源和对象生命周期绑定。
- 降低悬空指针风险，但不能彻底消灭所有非法访问。

## 注意

- 智能指针主要解决“谁负责释放资源”的问题。
- 它不能自动解决数组越界、数据竞争、非法裸指针访问等所有内存问题。

# 3.2 野指针、悬空指针、空指针

## 野指针

- 指向未知地址的指针。
- 常见来源：
  - 未初始化的指针
  - 指针被错误赋值为非法地址
  - 越界计算后得到的无效地址

## 悬空指针

- 原本指向有效对象，但对象生命周期已经结束。
- 常见来源：
  - `delete`/`free` 后继续使用
  - 返回局部变量地址
  - 容器扩容后继续使用失效迭代器/指针

## 空指针

- 值为空的指针，不指向任何有效对象。
- C++ 中优先使用 `nullptr`，不要再用 `NULL` 或字面量 `0` 代替空指针。

## 区别总结

- 野指针：一开始就不可靠
- 悬空指针：曾经可靠，后来失效
- 空指针：明确表示“不指向任何对象”

# 3.3 如何避免野指针和悬空指针

- 定义后立即初始化，能设为 `nullptr` 就先设为 `nullptr`
- 动态分配后检查结果是否符合预期
- 释放后及时置空
- 不返回局部对象地址
- 注意容器扩容、删除元素后的迭代器失效问题
- 优先使用 RAII 和智能指针，而不是手写 `new/delete`

# 3.4 RAII

RAII（Resource Acquisition Is Initialization）的核心思想是：

- 资源在对象构造时获取
- 资源在对象析构时释放

资源不只包括内存，还包括：

- 文件句柄
- 互斥锁
- socket
- 数据库连接

这样可以保证即使函数中途返回或抛异常，资源也能正确释放。

# 3.5 `unique_ptr`

## 特点

- 独占所有权
- 同一时刻只能有一个 `unique_ptr` 拥有对象
- 不能复制，只能移动

## 典型用法

```cpp
auto p = std::make_unique<int>(42);
auto q = std::move(p);
```

## 面试回答点

- 适合明确单一所有者的场景
- 默认开销小，语义清晰
- 优先级通常高于 `shared_ptr`

## 补充

- `unique_ptr` 可管理 `new[]` 分配的数组形式
- 不能复制，只能移动

# 3.6 `shared_ptr`

## 特点

- 多个智能指针共享同一对象所有权
- 通过控制块维护引用计数
- 最后一个拥有者销毁时释放对象

## 控制块通常包含

- 强引用计数
- 弱引用计数
- 删除器等元数据

## 注意点

- 不要把同一个裸指针分别交给两个独立的 `shared_ptr` 管理，否则会二次释放。
- `shared_ptr` 只保证控制块中的引用计数操作线程安全，不保证被管理对象本身线程安全。
- 优先使用 `std::make_shared` 创建，通常更高效，也更安全。

## 智能指针推荐创建方式

| 智能指针类型 | 推荐分配方法 | 示例 |
| --- | --- | --- |
| `std::unique_ptr` | `std::make_unique` | `auto p = std::make_unique<int>(10);` |
| `std::shared_ptr` | `std::make_shared` | `auto p = std::make_shared<int>(10);` |
| `std::weak_ptr` | 从 `shared_ptr` 获取 | `std::weak_ptr<int> w = p;` |

推荐原因：

- 更安全
- 更高性能
- 更简洁
- 更不容易遗漏异常路径下的资源释放

## 为什么 `make_shared` 常常性能更好

很多实现里：

- 普通写法可能需要两次内存分配
  - 一次给对象
  - 一次给控制块
- `make_shared` 常常可以一次性分配对象和控制块

所以它通常：

- 分配次数更少
- 局部性更好
- 性能更优

## 赋值/拷贝的本质

- 新对象共享同一控制块
- 强引用计数增加
- 原资源在最后一个拥有者释放时析构

## `shared_ptr` 怎么知道别的共享者释放了

- 所有共享者都指向同一个控制块
- 控制块里维护强引用计数
- 某个 `shared_ptr` 析构或重置时，计数减少
- 计数减到 0 时，释放被管理对象

## 什么时候会改变引用计数

- `shared_ptr` 从裸资源或工厂函数创建时
  - 强引用计数初始化为 1
- 拷贝构造、拷贝赋值时
  - 强引用计数增加
- 析构、`reset()`、重新赋值时
  - 强引用计数减少
- 强引用减到 0 时
  - 被管理对象析构
  - 但控制块不一定立刻销毁，还要看是否仍有 `weak_ptr`

# 3.7 `weak_ptr`

## 作用

- 弱引用，不拥有对象，不增加强引用计数
- 用于观察 `shared_ptr` 管理的对象是否仍然存活
- 常用于打破循环引用

## 常见写法

```cpp
std::shared_ptr<int> sp = std::make_shared<int>(42);
std::weak_ptr<int> wp = sp;

if (auto locked = wp.lock()) {
    // 对象仍然存活
}
```

## `weak_ptr` 能不能知道对象是否已释放

可以。

常见方式：

- `expired()`
  - 判断对象是否已失效
- `lock()`
  - 若对象仍存活，返回一个新的 `shared_ptr`
  - 若对象已释放，返回空 `shared_ptr`

## 为什么它能知道

- `weak_ptr` 也会关联同一个控制块
- 它虽然不增加强引用计数，但可以观察控制块中的对象状态
- 因此能通过 `expired()` / `lock()` 判断对象是否还活着

# 3.8 循环引用问题

```cpp
struct A { std::shared_ptr<B> b; };
struct B { std::shared_ptr<A> a; };
```

如果 `A` 和 `B` 互相持有 `shared_ptr`，即使外部指针都销毁，对象也可能无法释放。

解决方式：

- 让一端改成 `weak_ptr`

# 3.9 智能指针常见面试补充

## 智能指针的实现思路

如果让你手写一个简化版 `shared_ptr`，通常会提到：

- 成员：
  - 资源指针
  - 控制块指针
- 行为：
  - 构造函数接管资源
  - 析构函数减少计数，必要时释放资源
  - 拷贝构造增加计数
  - 移动构造转移所有权
  - 移动赋值转移控制权并处理旧资源
  - 赋值运算符处理旧资源释放、自赋值安全
  - 解引用 `*`
  - 成员访问 `->`

## 手写智能指针类常被追问的函数

- 构造函数
- 析构函数
- 拷贝构造
- 拷贝赋值
- 移动构造
- 移动赋值
- `operator*`
- `operator->`
- 可能还会补 `get()`、`reset()`、`swap()`

## 不建议说得太死的点

- “一定要用互斥锁”不准确。标准库实现通常使用原子操作维护引用计数，而不是简单地给每次访问都上互斥锁。
- “智能指针能解决悬空指针”也不应绝对化。若把裸指针从智能指针里取出来并在对象释放后继续使用，仍然会产生悬空指针。

## 智能指针和普通指针的区别

- 普通指针只保存地址，不负责释放
- 智能指针额外封装了所有权和生命周期管理
- 智能指针通常更安全，但也有额外语义和一定开销

## 智能指针有哪些

- `unique_ptr`
  - 独占所有权
- `shared_ptr`
  - 共享所有权
- `weak_ptr`
  - 弱引用，不拥有对象
- `auto_ptr`
  - 已废弃，不建议再提为现代用法

## `shared_ptr` 的线程安全性

- 不同 `shared_ptr` 实例共享同一控制块时，引用计数增减通常是线程安全的
- 但同一个 `shared_ptr` 对象本身若被多个线程同时读写，仍需同步
- 被管理对象本身也不因 `shared_ptr` 而自动线程安全
- 常见实现会用原子方式维护控制块计数，但这不等于“整个对象访问天然线程安全”

## `weak_ptr` 如何解决循环引用

- 循环中的一端改为 `weak_ptr`
- 这样不会增加强引用计数
- 最终强引用计数可以降到 0，对象得以释放

## 智能指针也会内存泄漏吗

会，典型情况包括：

- `shared_ptr` 循环引用
- 自定义删除器写错
- 同一裸指针被多个独立智能指针重复接管

所以更准确的说法是：

- 智能指针能显著降低泄漏风险
- 但不是“用了就绝对不会泄漏”
