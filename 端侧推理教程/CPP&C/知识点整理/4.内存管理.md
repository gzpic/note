# 4.1 C/C++ 程序中的内存区域

常见划分包括：

- 代码区 `.text`
  - 存放机器指令
- 只读数据区 `.rodata`
  - 存放字符串字面量、只读常量
- 已初始化数据区 `.data`
  - 存放已初始化且初值非 0 的全局变量、静态变量
- 未初始化数据区 `.bss`
  - 存放未初始化或初始化为 0 的全局变量、静态变量
- 堆区
  - 动态分配内存
- 栈区
  - 函数调用现场、局部变量、返回地址等

## 常见面试点

- “程序有哪些 section” 通常答 `.text`、`.rodata`、`.data`、`.bss`。
- “运行时还有哪些区域” 再补堆、栈、文件映射区、共享库映射区等虚拟内存区域。

# 4.2 初始化为 0 的全局变量在 `.bss` 还是 `.data`

通常回答：

- 未初始化的全局变量 / 静态变量在 `.bss`
- 初始化为 0 的全局变量 / 静态变量通常也放在 `.bss`
- 初始化为非 0 的全局变量 / 静态变量通常放在 `.data`

## 注意

- 这是常见实现，不是所有平台都必须完全一致，但面试里这样回答通常是对的。

# 4.3 程序启动过程

简化后的启动流程通常可以这样回答：

1. 操作系统创建进程，建立虚拟地址空间。
2. 加载器把可执行文件和依赖动态库映射到进程地址空间。
3. 初始化 `.data`、清零 `.bss`。
4. 运行全局/静态对象构造函数。
5. 进入运行时入口，再调用 `main()`。

如果是嵌入式裸机环境，一般还会强调：

1. 复位后先执行启动代码。
2. 设置栈指针。
3. 拷贝 `.data` 到 RAM，清零 `.bss`。
4. 调用系统初始化。
5. 跳转到 `main()`。

# 4.4 栈和堆的区别

## 栈

- 自动分配、自动释放
- 空间通常较小
- 分配和回收效率高
- 严格遵循后进先出

## 堆

- 手动申请、手动释放
- 空间通常更大
- 分配释放相对更慢
- 容易产生外部碎片

## 一句话对比

- 栈适合生命周期清晰、大小较小的数据。
- 堆适合运行期动态决定大小和生命周期的数据。

# 4.5 如何判断对象在栈上还是堆上

- 局部变量通常在栈上。
- 通过 `new` / `malloc` 申请的对象通常在堆上。
- 全局变量、静态变量不在堆也不在栈，而在静态存储区。

## 注意

- “局部变量一定在栈上”也不能说死。优化后，部分局部变量可能进寄存器。

# 4.6 自由存储区和堆的区别

- 堆（heap）更偏向操作系统或运行库层面的内存区域概念。
- 自由存储区（free store）是 C++ 里通过 `new/delete` 动态分配对象的抽象概念。

## 常见理解

- `malloc/free` 操作的是堆上的原始内存。
- `new/delete` 操作的是自由存储区中的对象生命周期。
- 很多实现里，自由存储区底层仍然会使用堆，但二者概念上不完全等价。

# 4.7 C 和 C++ 的内存分配方式

## 静态存储期

- 全局变量
- 静态变量

程序启动前或启动阶段完成初始化，生命周期持续到程序结束。

## 自动存储期

- 局部变量
- 函数参数

通常随函数调用进入和退出而创建、销毁。

## 动态存储期

- C：`malloc / calloc / realloc / free`
- C++：`new / delete`、`new[] / delete[]`

## `malloc/free` 与 `new/delete` 的区别

- `malloc/free` 只负责原始内存，不调用构造和析构。
- `new/delete` 既管理内存，也管理对象生命周期。
- `new` 失败默认抛出异常，`malloc` 失败返回 `nullptr`。
- `new` 返回对应类型指针，`malloc` 返回 `void*`。

# 4.8 `operator new/delete`、`new/delete`

## 区别

- `operator new` 本质是分配原始内存的函数。
- `new` 表达式会先调用 `operator new` 分配内存，再调用构造函数。
- `delete` 表达式会先调用析构函数，再调用 `operator delete` 释放内存。

## 面试回答模板

- `new = 分配内存 + 构造对象`
- `delete = 析构对象 + 释放内存`

# 4.8.1 为什么 C++ 里有了 `malloc/free` 还需要 `new/delete`

- `malloc/free` 只处理原始内存，不懂对象生命周期
- `new/delete` 面向对象，能自动调用构造和析构
- `new` 返回正确类型指针，`malloc` 返回 `void*`
- `new` 失败默认抛异常，`malloc` 失败返回 `nullptr`

## 一句话

- `malloc/free` 管“内存”
- `new/delete` 管“对象 + 内存”

# 4.8.2 `malloc` 申请的内存能不能用 `delete` 释放

不能。

原因：

- `malloc` 和 `free` 是一组
- `new` 和 `delete` 是一组
- 混用会导致未定义行为

同理：

- `new` 出来的对象不能用 `free`
- `malloc` 出来的内存不能用 `delete`

# 4.9 `delete[]` 为什么必须配对 `new[]`

对象数组的销毁需要知道数组里有多少个元素，尤其是元素带析构函数时，需要逐个析构。

因此：

- `new[]` 必须配 `delete[]`
- `new` 必须配 `delete`

混用会导致未定义行为。

## 注意

- “数组长度一定存在对象前 4 个字节”不是标准保证，只能说很多实现会保存额外元数据来支持 `delete[]`。

## `delete` 和 `delete[]` 的区别

- `delete`
  - 面向单个对象
  - 通常只调用一次析构函数
- `delete[]`
  - 面向对象数组
  - 会对数组中每个元素依次调用析构函数

# 4.9.1 `delete` 是如何知道释放大小的

对单个对象而言，`delete p;` 并不一定需要用户手动提供大小，运行库和分配器通常知道这块内存来自哪个分配单元。

对对象数组而言，`delete[] p;` 还需要知道数组元素个数，以便逐个调用析构函数。

## 面试稳妥说法

- 单对象释放大小由分配器/运行时管理
- 数组删除通常需要额外元数据记录元素数量
- 但“固定保存在对象前 4 字节”不是标准保证

# 4.10 深拷贝和浅拷贝

## 浅拷贝

- 只复制对象表面值
- 若成员里有裸指针，可能变成多个对象指向同一块资源

## 深拷贝

- 不仅复制对象本身，还复制其拥有的底层资源

## 常见风险

- 浅拷贝 + 手动释放资源，容易 double free
- 这也是为什么资源类通常要遵循 Rule of Three / Five

# 4.11 拷贝一段数据：`memcpy` vs `std::copy`

## `memcpy`

- 按字节复制原始内存
- 适合 POD、字节缓冲区、二进制块
- 对非平凡对象直接 `memcpy` 可能有问题

## `std::copy`

- 逐元素复制
- 会使用赋值运算或拷贝语义
- 更符合 C++ 类型系统

# 4.12 常见内存错误

- 未初始化就使用
- 越界访问
- 重复释放
- 释放后继续使用
- 内存泄漏
- 栈溢出
- 野指针 / 悬空指针

# 4.13 内存泄漏和内存溢出

## 内存泄漏

- 申请了资源但最终没有释放，导致这部分内存再也无法被程序有效使用。

常见原因：

- `new` 后忘记 `delete`
- `malloc` 后忘记 `free`
- 循环引用
- 基类析构函数非虚，导致派生类资源未正确释放
- 异常路径提前返回

## 内存溢出

- 程序申请或使用内存超出可用范围。

常见表现：

- 堆空间耗尽
- 栈溢出
- 缓冲区越界写

## 关系

- 内存泄漏长期积累后，可能最终导致内存溢出。

# 4.13.1 静态库和动态库

## 静态库

- Linux 常见后缀是 `.a`
- 链接时把库代码拷进可执行文件

### 优点

- 部署简单
- 运行时少依赖外部库文件

### 缺点

- 可执行文件更大
- 库更新后通常需要重新链接发布

## 动态库

- Linux 常见后缀是 `.so`
- 运行时再加载

### 优点

- 多个程序可共享同一份库代码
- 更新库时通常不必重新链接所有程序

### 缺点

- 依赖运行环境
- 部署和版本兼容更复杂

## 面试里的简表

| 对比项 | 静态链接 | 动态链接 |
| --- | --- | --- |
| 处理时机 | 链接期 | 运行期 |
| 常见产物 | `.a` | `.so` |
| 可执行文件体积 | 更大 | 更小 |
| 部署依赖 | 更少 | 依赖动态库环境 |
| 更新库是否方便 | 不方便，常需重新链接 | 更方便 |

# 4.13.2 为什么 C++ 没有默认垃圾回收

- C++ 设计目标之一是可预测的性能和底层控制能力
- GC 会带来额外时间开销、空间开销和停顿问题
- 系统编程、嵌入式、实时场景往往更看重确定性

## 面试稳妥说法

- C++ 不是不能做 GC，而是语言层面没有强制内建 Java 那种垃圾回收机制
- 它更倾向 RAII、智能指针和显式资源管理

# 4.13.3 为什么不能把所有函数都写成内联

- 内联会把函数体展开到调用点
- 若函数很多、函数体很大，会导致代码膨胀
- 含复杂循环、递归、虚调用等场景，也未必适合内联

## 一句话

- 内联是“用代码体积换调用开销”，不是越多越好

# 4.13.1 栈溢出和内存溢出

## 栈溢出

- 常见原因是递归过深、局部变量过大、无限调用
- 本质是栈空间不够

## 内存溢出

- 更广义，指程序申请或使用内存超出系统可提供范围
- 可能发生在堆、栈、映射区等

## 关系

- 栈溢出属于内存问题的一种
- 但“栈溢出”和“内存溢出”不是完全同义词

# 4.13.2 C++ 异常处理

核心机制：

- `try`
- `throw`
- `catch`

基本流程：

1. 在 `try` 块中执行代码
2. 发生异常时 `throw`
3. 控制流跳到匹配的 `catch`
4. 若当前层不处理，继续向上栈展开

## 注意

- 栈展开过程中，已经构造完成的局部对象会按顺序析构
- 这也是 RAII 在异常场景下仍然安全的重要原因

# 4.13.3 常见标准异常类

- `std::exception`
  - 标准异常基类
- `std::bad_alloc`
  - 内存分配失败
- `std::bad_cast`
  - `dynamic_cast` 引用失败等
- `std::bad_typeid`
  - `typeid` 相关异常
- `std::out_of_range`
  - 越界
- `std::runtime_error`
  - 运行期错误

## 面试建议

- 不必死背太多异常类
- 记住基类 `std::exception` 和几个常见派生类即可

# 4.14 如何避免和定位内存泄漏

## 避免

- 优先使用 RAII
- 优先使用智能指针和标准容器
- 保证 `new/delete`、`malloc/free` 成对出现
- 基类析构函数在多态场景下应设为虚函数
- 数组使用 `new[]/delete[]`

## 定位

- Linux 常见工具：Valgrind、ASan、LSan
- Windows 常见工具：CRT 调试堆、Visual Studio 诊断工具
- 代码层面可做简单分配计数或资源跟踪

# 4.14.1 `free` 回收的内存会立刻还给操作系统吗

通常不会立即全部归还给操作系统。

更常见的情况是：

- 分配器先把这块内存缓存到自己的空闲链表或空闲块结构中
- 后续相似大小的申请可直接复用
- 只有在特定条件下，分配器才可能把部分大块内存归还给操作系统

## 面试回答建议

- 不要答得太绝对
- 可回答为“通常先回到分配器管理，而不是立刻回到操作系统”

# 4.16 内存对齐

## 什么是内存对齐

- 数据对象的起始地址满足某种对齐要求，例如 4 字节对象放在 4 的倍数地址上。

## 为什么要对齐

- 提升访问效率
- 满足某些硬件平台的访问约束
- 便于编译器生成更高效的指令

## 自然对齐和 CPU 字长的关系

CPU 的总线宽度和访存机制会影响自然对齐单位。

| CPU架构 | 常见自然对齐 |
| --- | --- |
| 8-bit MCU | 1 byte |
| 32-bit CPU | 4 bytes |
| 64-bit CPU | 8 bytes |

常见理解：

- CPU 一次通常更喜欢按“字（word）”访问
- 对齐良好时访问通常更高效

## 结构体常见对齐规则

1. 第一个成员从偏移 0 开始。
2. 每个成员的起始偏移通常要满足该成员对齐要求。
3. 结构体总大小通常要向最大对齐要求对齐。

## 示例

```cpp
struct Bad {
    char a;
    int b;
    char c;
}; // 常见实现下大小可能是 12

struct Good {
    int b;
    char a;
    char c;
}; // 常见实现下大小可能是 8
```

## `#pragma pack(n)` 的影响

- 可修改对齐上限
- 会影响内存布局
- 可能节省空间，但也可能降低访问效率，甚至带来平台兼容性问题

补一句更准确的说法：

- `#pragma pack(n)` 不是“比 n 大的类型就完全不管”
- 更准确地说，是把成员对齐要求限制为“自身自然对齐”和 `n` 中较小的那个

## 使用场景

- 网络协议
- 文件格式
- 硬件寄存器映射
- 二进制序列化

# 4.17 对象池 / 内存池

## 思想

- 预先申请一批对象或内存块
- 需要时直接取，释放时放回池中复用

## 适用场景

- 高频创建销毁
- 固定大小对象
- 对性能和碎片敏感的系统

# 4.18 函数调用过程

一个普通函数调用，通常会经历：

1. 传递参数
2. 保存返回地址和必要寄存器
3. 建立新的栈帧
4. 执行函数体
5. 产生返回值
6. 恢复现场并返回调用点

## 注意

- 现代编译器会大量优化，不一定严格表现为“每一步都在栈上完成”。
- 小对象、返回值、参数都可能通过寄存器传递。

# 4.19 函数调用时的栈帧与“压栈过程”

传统教材里常把函数调用描述为：

- 参数入栈
- 返回地址入栈
- 保存调用现场
- 分配局部变量空间

这个描述适合帮助理解栈帧，但面试回答时最好补一句：

- 实际参数传递顺序、是否入栈、由谁清理栈空间，取决于平台 ABI、调用约定和编译器优化。

## 稳妥回答

- 函数调用会建立新的调用现场，通常涉及参数传递、返回地址保存、局部变量和寄存器保存。
- 栈是常见实现方式，但不是所有数据都一定通过栈传递。

# 4.20 临时对象作为返回值时的处理

不要简单背“返回值一定存在寄存器里”。

更稳妥的说法是：

- 函数返回局部对象本身是合法的，只要返回的是“对象值”而不是“局部对象地址或引用”。
- 编译器可能使用：
  - 返回值优化（RVO/NRVO）
  - 移动构造
  - 寄存器返回
  - 调用者提供返回值空间

例如：

```cpp
std::string make_name() {
    std::string s = "codex";
    return s;
}
```

这里不会因为局部变量离开作用域就出错，因为返回的是对象值，编译器会通过拷贝省略或移动语义完成返回。

## 反例

```cpp
int* bad() {
    int x = 1;
    return &x;  // 错误：返回局部变量地址
}
```

这才会导致悬空指针问题。

# 4.22 类的数据成员和成员函数在内存中的分布

## 数据成员

- 非静态数据成员存储在对象内存中
- 静态数据成员不在对象内部，而在静态存储区

## 成员函数

- 普通成员函数、静态成员函数都不存放在对象内部
- 它们通常位于代码区，由所有对象共享

## 对象大小

- 对象大小通常只与非静态数据成员、对齐填充、以及虚函数带来的 `vptr` 等有关
- 成员函数本身不会直接增加对象大小

# 4.23 存储器分层

从快到慢、从小到大，常见可概括为：

- 寄存器
- L1/L2/L3 Cache
- 主存（RAM）
- 外存（SSD/HDD）

## 作用

- 上层更快更贵更小
- 下层更慢更便宜更大
- 通过局部性原理提升整体访问效率

## 面试注意

- 每层“具体多大”强依赖 CPU 和机器型号，不要背死数值
- 回答层次、特点和作用通常更重要

# 4.24 程序从堆中动态分配内存时，虚拟内存上怎么操作

可以用更稳妥的方式回答：

- 程序调用 `malloc/new` 时，先向运行时分配器申请内存
- 分配器再通过 `brk/sbrk` 或 `mmap` 等机制向操作系统申请虚拟地址空间
- 操作系统建立或更新虚拟地址到物理页的映射关系
- 真正物理页的分配很多时候还是按需发生，可能在首次访问时触发缺页再完成映射

## 注意

- 不要简单说成“直接在硬盘里创建一块虚拟内存”
- 更准确地说，是在进程虚拟地址空间中保留或映射一段区域，再由页表管理

# 4.25 一般情况下栈空间多大

- Linux 下主线程栈常见默认值大约是 8MB
- Windows 下主线程栈常见默认值常见是 1MB 左右

## 注意

- 具体大小受系统、编译器、链接选项、线程属性等影响
- 面试里要强调“常见默认值”，不要说成绝对固定

# 4.26 从堆上建立对象和从栈上建立对象哪个快

通常回答：

- 栈上更快

原因：

- 栈分配通常只是移动栈指针
- 堆分配通常需要经过分配器管理空闲块、维护元数据

## 但也别答太绝对

- 栈对象生命周期短且自动回收，适合简单场景
- 堆对象更灵活，适合运行时大小和生命周期不固定的场景

# 4.27 Linux 进程地址空间与内核区/用户区

Linux 常从权限角度分为：

- 用户空间
- 内核空间

## 用户空间里常见区域

- 代码段
- 数据段
- 堆
- 栈
- 共享库/映射区

## 核心区别

- 用户态代码不能直接随意访问内核空间
- 进入系统调用、中断、异常处理时会切换到内核态

## 面试回答建议

- 若题目问“Linux 内存区划分”，先答用户空间/内核空间
- 若继续追问，再展开用户空间中的代码段、数据段、堆、栈、映射区
