### **5.1 关于 const**

- 变量/对象 const

`const int MAX_USERS = 100;
MAX_USERS = 200;  // 编译错误：不能修改 const 变量`

- 函数参数 const

`void printString(const std::string& str) {
    str = "modified";  // 编译错误：不能修改 const 引用参数
    std::cout << str;  // 只读访问
}`

- 成员函数 const

`class Widget {
private:
    int value;
public:
    int getData() const {  // const 成员函数
        // value = 10;  // 编译错误：不能修改成员变量
        return value;  // 只读访问
    }
    
    void setData(int v) {  // 非 const 成员函数
        value = v;  // 可以修改
    }
};

const Widget w;
w.getData();  // 正确：const 对象可以调用 const 成员函数
w.setData(5); // 编译错误：const 对象不能调用非 const 成员函数`

注意：`const` 对象只可以调用 `const` 成员函数，因为没有 `const` 修饰的成员函数，编译器也不知道它内部是否会修改对象状态，如果允许调用，相当于自己打破了对变量是 `const` 的承诺

## `const` 和 `#define` 的区别

- `const` 有类型，参与编译器类型检查
- `#define` 本质是预处理替换，没有类型信息
- `const` 更便于调试
- `#define` 常用于宏常量、条件编译、头文件保护

## 注意

- 不要机械地说 “`const` 一定在数据段，`#define` 一定在代码段”
- 是否真的分配存储、放在哪个段，取决于编译器实现和使用方式

### **5.2 关于 static**

- 函数作用域：静态局部变量

`void func() {
    static int count = 0;  // 只初始化一次
    count++;
    cout << count;  // 每次调用输出递增：1, 2, 3...
}`

函数内部的 `static`变量，生命周期是整个程序，但作用域仅限于函数内部。第一次调用时初始化，后续不再重置。

- 类作用域：静态类成员/函数

`class Widget {
public:
    static int shared;      // 声明：所有对象共享一份
    static void print() {
        cout << shared;
    }  // 只能访问静态变量
};
int Widget::shared = 0;     // C++17 之前必须在类外定义

// 在 C++17 之后引入内联变量，可以在类内直接定义并初始化静态成员，编译器会自动保证所有编译单元共享同一个实体
class Widget {
public:
    inline static int shared = 0; 
    //......
};`

类内的 `static` 变量属于**类级别**而非对象级别，因此所有对象的 `static` 变量都共享一份内存，且静态函数只能访问静态成员，因为非静态成员属于对象级别，级别不同没办法。

另外注意，`static` 修饰的类函数的签名是没有 `this` 指针的，其他类函数有隐藏的 `this` 指针。非静态成员函数的实际签名是：

`bool(ClassType* this, const T&, const T&);  // 隐式this指针作为第一个参数`

静态成员函数的实际签名就没有 `this` 指针了。具体应用比如 `std::sort` 的第三个比较函数，其签名必须是：

`bool(const T&, const T&);  // 两个参数`

因此不能写成直接的类函数，要么写在类内部作为静态成员函数，要么写成类外的自由函数。

- 文件作用域：静态全局变量/函数

`// file1.cpp
static int secret = 42;          // 只在 file1.cpp 内可见
static void helper() {}          // 只在 file1.cpp 内可用

// file2.cpp
// extern int secret;  // 错误：无法访问 file1 的 static 变量`

`static` 在文件作用域的作用就是隐藏符号，在链接器链接时忽略它，避免跨文件污染和冲突。

初始化时机：情况1是首次执行到声明语句时再初始化，情况2和3都是在 main() 之前初始化

## 静态变量什么时候初始化

- 静态存储期对象的存储空间在程序启动阶段就已经准备好
- 全局/命名空间作用域静态对象通常在 `main` 前初始化
- 函数内静态局部变量在第一次执行到定义语句时初始化

## C++11 之后

- 局部静态变量初始化要求线程安全

### **5.3 关于 extern 和 extern “C”**

`extern` 告诉编译器从其他地方找定义

`// a.cpp
int house = 100;  // 实际存在的变量

// b.cpp
extern int house;  // 声明：从其他文件里拿
cout << house;

// 假如 c.cpp 里面也有定义
int house = 50; 
// 链接时会报错
// multiple definition of `house'
// a.obj: first defined here
// c.obj: defined again`

所以标准做法是用一个文件专门进行定义，其他文件直接声明（`extern`）即可

`extern "C"` 关键字修饰函数，是提醒编译器在编译该函数时按C语言规则编译，避免命名冲突。因为C++为了兼容函数重载，函数名并非唯一标识符，会有关于参数类型、命名空间等信息作为修饰。

`// C++ 编译器会对函数名做修饰（Name Mangling）
void func(int);        // 实际符号可能是：_Z4funci

// C 编译器不会修饰
void func(int);        // 实际符号就是：func`

这是为了兼容C，因此在编写C的库时，需要在头文件里面用 `extern "C"` 修饰：

`// mylib.h
#ifdef __cplusplus
extern "C" {
#endif

void my_c_func(int x);  // C 函数声明

#ifdef __cplusplus
}
#endif`

如果 C 头文件里面没加，而且又改不了，就在cpp代码里面加：

`extern "C" {
    #include "bad_c_lib.h"  // 这个头文件没做 extern "C" 处理
}`

### **5.3 关于 explicit**

避免编译器自动进行隐式类型转换，要求必须显式调用构造函数或转换操作。具体来说是下面这种情况：

`class String {
public:
    String(int size) { /* String 可以用 int 构造 */ }  // 编译器将其作为隐式转换构造函数
    void print() const;
};

void printString(const String& s) {
    s.print();
}

// 意外用法
printString(10);  // 本意是报错，但编译器会认为可以将 int 通过隐式构造转换成 String，再调用函数
                  // 编译器会默默创建一个 String(10) 临时对象`

为了避免这种情况，用 `explicit` 修饰即可：

`explicit String(int size) { /* ... */ }  // 禁止隐式转换`

算是解决引入编译器新特性后出现的问题，史山这块…

### **5.3.1 变量声明和定义的区别**

- 声明：告诉编译器名字和类型，不一定分配存储
- 定义：真正创建实体，通常会分配存储

例如：

```cpp
extern int g;  // 声明
int g = 10;    // 定义
```

## 面试常见结论

- 一个变量可以多次声明
- 但通常只能定义一次

### **5.4 关于 constexpr**

让变量或函数在编译器期间就计算出结果，省去运行时再计算的时间。两个核心需求在于：

- 数组大小必须是编译期常量

`int n = 5;
int arr1[n];  // 错误！n是运行时才确定的值

constexpr int m = 5;
int arr2[m];  // 正确！m在编译时就是确定的5`

不过实际上有些编译器会把这个优化掉

- 模板参数必须是编译期常量

`template<int N>
struct Foo {};

int x = 5;
Foo<x> f1;  // 错误！x不是编译期常量

constexpr int y = 5;
Foo<y> f2;  // 正确！`

## `constexpr` 和 `const`

- `const` 表示只读，不一定是编译期常量
- `constexpr` 要求可在编译期求值
- `constexpr` 变量隐含 `const`
- 并不是所有 `const` 都能当模板参数或数组大小

### **5.5 关于 volatile**

告诉编译器每次访问其修饰的变量时都需要从内存中读取最新值，而非用寄存器中的缓存值。主要用于易变的变量。

经典的应用场景如下：写一个嵌入式驱动，读取某个温度传感器的状态：

`// 假设 0x40011000 是温度传感器的状态寄存器地址
uint32_t* TEMP_STATUS = (uint32_t*)0x40011000;

// 等待传感器就绪
while ((*TEMP_STATUS & 0x01) == 0) {
    // ...
}`

如果不用 `volatile` ，编译器优化后很有可能优化成只读一次内存，后续全部用寄存器的缓存。导致传感器状态变了也检测不到，因此正确写法是用 `volatile` 修饰：

`volatile uint32_t* TEMP_STATUS = (volatile uint32_t*)0x40011000;`

### **5.5.1 宏和函数的区别**

- 宏在预处理阶段展开，本质是文本替换
- 函数在编译后生成可调用实体
- 宏没有类型检查，函数有类型检查
- 宏可能带来重复求值、副作用等风险
- 函数调用通常更安全、可调试性更好

## 一个经典问题

```cpp
#define MAX(a, b) ((a) > (b) ? (a) : (b))
```

如果传入带副作用表达式，例如 `MAX(i++, j++)`，就可能出现重复求值问题。

### **5.6 关于 mutable**

将修饰的变量标记为 const 成员函数也能修改的变量

`class Cache {
private:
    mutable int count = 0;  
public:
    void f1(int key) const { count++ ;}
};`

另一个应用场景是 Lambda 表达式，Lambda 的 `operator()` 默认就是 const，因此不能修改捕获的变量。用 mutable 修饰就行：

`// 按值捕获 count，但需要在 lambda 内部修改它
int count = 0;
auto counter = [count]() mutable {
    return ++count;  // 每次调用都+1
};

std::cout << counter() << '\n';  // 1
std::cout << counter() << '\n';  // 2
std::cout << counter() << '\n';  // 3`

### **5.7 关于 auto 和 decltype**

`auto` 推导会偷工减料（比如丢掉引用、const），`decltype` 就是精确推导。比如：

`int x = 42;
int& ref = x;

// auto 推导：引用被丢了，变成纯 int
auto a = ref;      // a 的类型是 int（不是 int&）
a = 100;           // 改的是 a 自己，x 还是 42

// decltype 推导：引用完整保留
decltype(ref) b = ref;  // b 的类型是 int&
b = 200;           // 改的是 x 本身，x 变成 200`

`auto` 经常用于只需要值的场景，为了声明不会修改值同时不拷贝提升效率，经常用`const auto&`，尤其在遍历容器的时候：

`std::vector<int> vec = {1,2,3,4};

for (const auto& x : vec) {
	// ...
}`

另外注意，如果 `auto` 指向的原对象就是 `const`，那么即使用 `auto&` 也无法修改。

### **5.8 关于 atomic 和内存顺序**

`std::atomic` 用来进行原子操作，解决的是“单次读写不可被打断”的问题；而内存顺序（memory order）解决的是“多线程之间看到这些操作的先后关系”。

常见六种内存序：

- `memory_order_relaxed`
  - 只保证该原子操作本身原子，不额外提供同步顺序
- `memory_order_consume`
  - 很少单独使用，实践中通常近似按 `acquire` 理解
- `memory_order_acquire`
  - 该操作之后的读写，不能被重排到它前面
- `memory_order_release`
  - 该操作之前的读写，不能被重排到它后面
- `memory_order_acq_rel`
  - 同时具有 acquire 和 release 语义，常用于读改写操作
- `memory_order_seq_cst`
  - 最强、最直观的顺序一致性语义，也是默认值

## 面试回答建议

- 不确定时优先回答 `seq_cst` 是最强顺序，也是默认值。
- 真正常见的工程搭配是 `release` + `acquire`。
- `relaxed` 常用于计数器、统计值等只关心原子性、不关心先后同步的场景。

### **5.9 头文件包含顺序与 `""` / `<>`**

## `#include "file.h"` 和 `#include <file.h>` 的区别

- `""`：
  - 先在当前文件所在目录或项目头文件路径中查找
  - 再去系统头文件路径查找
- `<>`：
  - 通常直接在编译器配置的系统/库头文件路径中查找

## 常见使用习惯

- 项目内头文件常用 `""`
- 标准库或第三方库头文件常用 `<>`

## 头文件包含顺序建议

- 当前源文件对应头文件
- 当前模块内头文件
- 第三方库头文件
- 标准库头文件

核心目的是：

- 依赖清晰
- 尽早暴露缺失包含问题

### **5.10 条件编译与头文件保护**

常见条件编译指令：

- `#define`
- `#ifdef`
- `#ifndef`
- `#endif`

## 经典用途

- 防止头文件重复包含
- 按平台切换代码
- 按编译配置打开或关闭调试逻辑

## 头文件保护示例

```cpp
#ifndef MY_HEADER_H

// declarations

#endif
```

### **5.11 `gcc/g++` 的 `-O1`、`-O2`**

常见理解：

- `-O0`
  - 基本不做优化，便于调试
- `-O1`
  - 开启较基础优化，在编译时间和性能之间折中
- `-O2`
  - 在 `-O1` 基础上启用更多优化，通常生成更高性能代码

## 注意

- 不是“级别越高一定越好”
- 优化会影响调试体验，也可能改变代码时序和体积
- 实际项目常见 `Debug + -O0`、`Release + -O2` 或更高

### **5.12 `nullptr` 和 `NULL`**

- `nullptr` 是专门的空指针字面量，类型是 `std::nullptr_t`
- `NULL` 往往只是宏，很多实现里本质上是 `0` 或 `0L`
- 现代 C++ 应优先使用 `nullptr`

## 为什么更推荐 `nullptr`

- 不会和整数重载混淆
- 语义更清晰，就是空指针

### **5.13 `static_assert`**

`static_assert` 用于编译期断言。

```cpp
static_assert(sizeof(int) == 4, "int size must be 4");
```

## 和 `assert` 的区别

- `static_assert`
  - 编译期检查
- `assert`
  - 运行期检查

### **5.14 `noexcept`**

- `noexcept` 用来声明函数不应向外抛异常
- 若标记为 `noexcept` 的函数真的向外抛异常，程序通常会直接 `std::terminate`

## 面试常答点

- 析构函数通常应避免抛异常
- 移动构造/移动赋值常写成 `noexcept`
- 标准容器在某些场景下会更愿意选择 `noexcept` 的移动操作

### **5.15 `final` 和 `override`**

- `override`
  - 显式说明“这是在重写基类虚函数”
  - 若签名不匹配，编译器会报错
- `final`
  - 可修饰虚函数，表示禁止继续重写
  - 也可修饰类，表示禁止继续继承

### **5.16 `=default` 和 `=delete`**

- `=default`
  - 显式要求编译器生成默认实现
- `=delete`
  - 显式禁用某个函数

例如：

```cpp
class A {
public:
    A() = default;
    A(const A&) = delete;
};
```

## 面试可补一句

- 现代 C++ 更推荐用 `=default` / `=delete` 明确表达设计意图
