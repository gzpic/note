## 四、L1 Cache / Shared Memory ——**SM 内的核心缓存**

> 这是 GPU 和 CPU cache 差异最大的地方

### 在哪？

- **每个 SM 内部**
- L1 Cache 和 Shared Memory：
    - Ampere 开始是 **统一物理 SRAM**
    - 逻辑上可配置比例

### 容量（典型）

- 64KB ~ 192KB / SM（架构可变）

---

### 4.1 L1 Cache（自动缓存）

### 特点

- 延迟：~20 cycle
- 自动管理（类似 CPU cache）
- Warp 粒度访问友好

### 主要缓存什么？

- Global Memory 的：
    - 连续访问
    - 局部性访问
- Tensor Core 访存路径也会命中 L1

### 在推理里的作用

- 权重块（W）
- KV cache 的局部访问
- FlashAttention 的 block 内数据复用

## 五、L2 Cache ——**全 GPU 共享的中枢缓存**

### 在哪？

- GPU 芯片内部
- 所有 SM 共享

### 特点

- 延迟：~200 cycle
- 容量：
    - Ampere：~40MB
    - Hopper：~50MB+
- 带宽：极高（TB/s 级）

---

### L2 主要干什么？

### 1️⃣ 屏蔽显存延迟

- HBM 延迟：**500~800 cycle**
- L2 命中：~200 cycle

### 2️⃣ SM 之间共享数据

- KV Cache
- 模型权重
- 常量参数

### 3️⃣ 减少显存带宽压力

- 多个 SM 反复读同一块权重：
    - 命中 L2
    - 不回 HBM

---

### 推理场景中特别重要的点

- **Decode 阶段**
    - KV Cache 极大
    - L2 命中率决定吞吐
- GQA / MQA
    - K/V 被多个 head 共享
    - L2 复用价值巨大

1. 所以寄存器占用过多可能会影响l1缓存剩余容量？

这是一个非常敏锐且极具深度的问题。答案是：**物理上不占用，但逻辑上（发生溢出时）会严重挤占 L1 缓存的“有效容量”。**

为了把这个问题讲透，我们需要把 **寄存器文件 (Register File)**、**L1 缓存** 和 **Local Memory (局部内存)** 这三个概念的关系理清楚。

---

### 1. 物理架构：井水不犯河水

首先，在硬件物理层面，**寄存器文件（RF）和 L1 缓存是完全独立的两块硬件**。

- **寄存器文件 (RF):** 它是 SM 上最大、最快的一块存储区（通常 256KB/SM，比 L1 还大）。它是给线程存放私有变量的“VIP包厢”。
- **L1 缓存:** 它是存放从显存（Global Memory）搬来的数据的“临时仓库”。

所以，仅仅是“使用”寄存器（只要没用超），是**绝对不会**物理上减少 L1 缓存的大小（Size）的。你用 10 个寄存器，L1 是 128KB；你用 255 个寄存器，L1 还是 128KB。

### 2. 关键转折点：寄存器溢出 (Register Spill)

问题的核心在于**“过多”**这两个字。
当你的核函数太复杂，或者定义的局部变量太多，导致 SM 上的寄存器（VIP包厢）不够用了，编译器就必须把多出来的变量**“溢出” (Spill)** 出去。

溢出到哪里去了呢？**Local Memory (局部内存)**。

- **注意：** Local Memory 虽然名字叫“Local”，但它在物理上**并不在片上**，它其实是在**显存（DRAM）**里划出来的一块私有区域，速度非常慢。

### 3. 为什么会影响 L1？（罪魁祸首）

为了防止这些溢出的变量读写太慢拖死性能，**L1 缓存必须充当 Local Memory 的缓冲区**。

这就导致了“鸠占鹊巢”的现象：

2. **发生溢出：** 你的变量被踢到了 Local Memory。
3. **读写变量：** 每次你需要用这个变量，GPU 必须把它从 DRAM 加载回来。
4. **进入 L1：** 这个变量会先进入 L1 缓存（因为 L1 负责缓存 Local Memory）。
5. **挤占空间：** **L1 缓存里的 Cache Line 被这些溢出的“私有变量”占满了**。原本你应该用来缓存 Global Memory（比如矩阵数据、图像数据）的空间，被这些本来应该在寄存器里的变量挤出去了。

**结论：** 寄存器占用过多导致**溢出**时，这些溢出数据会疯狂进出 L1，导致 L1 中用于原本核心业务数据的**有效容量**大幅减少，Cache Miss 率飙升。

---

### 4. 架构演进视角（不同 GPU 的行为）

- **早期架构 (如 Kepler/Maxwell):** * L1 缓存甚至**专门**就是为了给 Register Spills 用的（Global Memory 的加载甚至不走 L1，只走 L2/Texture）。在这种架构下，溢出简直是灾难。
- **现代架构 (Volta/Ampere/Hopper):**
    - L1 缓存是统一的（Unified），既缓存 Global Memory 也缓存 Local Memory（溢出数据）。
    - 这意味着你的溢出数据直接和你的全局数据在 L1 里**抢地盘**。

6.为啥要两级缓存

为什么非要设计成“L1 + L2”这种两级结构？不能直接搞一个超级大、超级快的通用缓存吗？

答案是：**不能。因为物理定律（光速和电路延迟）和成本锁死了“大”和“快”不能兼得。**

NVIDIA（以及所有现代处理器）设计两级缓存，本质上是在做**“时空权衡”**。为了让你理解这个必要性，我们可以从三个角度来看：

### 1. 物理距离与速度的悖论（最根本的原因）

想象一下你的 GPU 是一个巨大的工厂（比如 H100，光刻面积非常大）：

- **SM (计算核心)** 就像是流水线上的**工人**。
- **数据** 就像是**零件**。

如果只有一个**单一的大缓存（比如 50MB）**放在芯片正中间：

- **距离问题：** 住在芯片边缘的 SM（工人）要拿数据，信号在电路上传输是需要时间的。这一来一回的物理距离，会导致延迟太高，工人大部分时间都在等零件，而不是干活。
- **拥堵问题：** 100 多个 SM 同时伸手向同一个大仓库要数据，仓库门口（带宽接口）会瞬间堵死。

**解决办法：**

- **造 L1 (小仓库):** 直接放在工人的手边。因为很小（几百KB），离计算单元极近，所以**极快**（几乎零延迟）。
- **造 L2 (大仓库):** 放在中间。虽然慢一点，但它大，能服务所有人。

### 2. 职能分工的必要性（私有 vs 共享）

如果只有一个缓存，如何处理“私有数据”和“共享数据”的矛盾？

- **如果只有 L1 (全私有):**
    - *问题：* 假如 SM A 算出了一个结果，SM B 想要用。SM B 必须去 SM A 的 L1 里拿？这需要复杂的线路把 100 多个 SM 互相连起来（全连接拓扑），芯片布线会复杂到爆炸，功耗也会失控。
    - *结果：* 无法高效共享数据。
- **如果只有 L2 (全共享):**
    - *问题：* 除了上面说的慢，还面临**粒度**问题。SM 计算时需要非常细粒度的操作（比如改一个字节），而 L2 为了对接显存，通常处理较大块的数据（32字节）。让 L2 处理所有细碎的读写，效率极低。

**两级缓存的完美配合：**

- **L1 (私有):** 既然大部分临时变量只有我自己用，我就锁在自己的 L1 里，不用告诉别人，不用走远路，怎么快怎么来。
- **L2 (共享):** 只有当大家都要用的数据（比如模型权重），或者我要写回显存的数据，才放到 L2。L2 充当了**“集线器”**，解决了互联难题。

### 3. 带宽的“漏斗效应”（为了省钱省电）

显存（HBM/GDDR）虽然带宽很大，但比 L2 还是慢得多且耗电。

- **L1 的作用：** 过滤掉 90% 的**重复访问**和**不规则访问**。
    - 比如一个 warp 里的 32 个线程乱序访问内存，L1 会把它们整理好（Coalescing），变成整齐的请求发给 L2。
- **L2 的作用：** 过滤掉剩下的 9% 的请求。
    - L1 没命中的，L2 看看自己有没有。如果有，就别去烦显存了。

**如果没有 L1，直接让 L2 面对 32 个线程乱七八糟的请求，L2 的带宽会瞬间被吃光。如果没有 L2，成千上万个 L1 Miss 直接打到显存上，显存带宽瞬间崩溃。**
