# 内存对齐

## 1. 内存对齐由什么决定

内存对齐不是单一规则决定的，而是由硬件、ABI、编译器和类型本身一起决定。

### 1.1 CPU / 硬件架构

CPU 的访存粒度决定了什么样的地址更适合访问。

- 8 位 CPU 常见自然对齐是 1 字节
- 32 位 CPU 常见自然对齐是 4 字节
- 64 位 CPU 常见自然对齐是 8 字节

如果数据地址满足自然对齐，CPU 往往可以更高效地完成访问；如果未对齐，可能需要多次访存、额外拼接，性能更差，某些架构下甚至可能触发异常。

### 1.2 数据类型的自然对齐

大多数编译器默认采用自然对齐规则：类型通常按自身大小对齐。

| 类型 | 常见大小 | 常见对齐 |
| --- | --- | --- |
| `char` | 1 | 1 |
| `short` | 2 | 2 |
| `int` | 4 | 4 |
| `float` | 4 | 4 |
| `double` | 8 | 8 |
| `long long` | 8 | 8 |

### 1.3 ABI 规则

真正落地时要看平台 ABI。ABI 会规定：

- 基本类型对齐
- 结构体对齐
- 栈对齐
- 函数调用约定

所以同一段代码在不同平台、不同编译器上，布局可能不同。

### 1.4 编译器设置

编译器可以通过下面这些方式人为改变默认行为：

- `alignas(16)`
- `__attribute__((aligned(16)))`
- `#pragma pack(n)`

其中 `alignas` / `aligned` 用于提高或指定对齐，`#pragma pack` 常用于压缩结构体布局。

## 2. 为什么要内存对齐

核心目的有两个：

- 提高 CPU 访问效率
- 满足硬件或 ABI 的访问要求

可以把它理解成“用少量空间换取更好的访存效率”。

## 3. 结构体对齐规则

结构体对齐记住 3 条就够了：

1. 每个成员都按自己的对齐要求放置
2. 结构体整体对齐值等于成员中最大的对齐要求
3. 结构体总大小必须是结构体对齐值的整数倍

例如：

```cpp
struct A {
    char c;
    int i;
    short s;
};
```

布局可以理解为：

```text
c      1
pad    3
i      4
s      2
pad    2
总计   12
```

这里：

- `char` 对齐要求是 1
- `int` 对齐要求是 4
- `short` 对齐要求是 2
- 最大对齐要求是 4

所以整个结构体按 4 对齐，最终大小也必须是 4 的倍数。

## 4. 为什么结构体尾部也要补 padding

尾部补齐的核心原因是保证结构体数组中每个元素也满足相同的对齐要求。

例如：

```cpp
struct A {
    char c;
    int i;
};
```

如果 `sizeof(A)` 不是 4 的倍数，那么：

- 第一个元素起始地址也许是对齐的
- 第二个元素起始地址就可能不对齐

这样数组里的后续元素访问效率会变差。

所以面试里可以直接回答：

> 结构体尾部 padding 的目的，是保证结构体数组中每个元素的起始地址仍满足对齐要求。

## 5. 结构体对齐是不是按最大类型决定

这个说法基本对，但更准确的说法是：

> 结构体的对齐值，通常等于成员中“最大对齐要求”的那个值，而不一定只是“最大类型大小”。

大多数常见基本类型里，“大小”和“对齐”经常相同，所以容易混在一起记。

例如：

```cpp
struct A {
    char c;
    double d;
};
```

通常：

- `char` 对齐 1
- `double` 对齐 8

所以结构体对齐值通常是 8。

## 6. `#pragma pack(n)` 到底改变了什么

`#pragma pack(n)` 不会改变类型本身的大小，只会限制成员的对齐要求。

可以记成：

```cpp
member_align = min(默认对齐, n)
```

因此：

- `sizeof(int)` 还是 4
- `sizeof(double)` 还是 8

变的是“它从哪个边界开始摆放”。

例如：

```cpp
#pragma pack(2)
struct A {
    char c;
    int i;
};
```

布局：

```text
c    1
pad  1
i    4
总计 6
```

这里 `int` 仍然占 4 字节，只是对齐要求由 4 降成了 2。

## 7. pack 小于最大类型时，大类型占几字节

结论很直接：

> `pack` 变小，不会改变类型本身占用的字节数，只会改变对齐要求。

例如：

```cpp
#pragma pack(4)
struct A {
    char c;
    double d;
};
```

此时：

- `double` 仍然占 8 字节
- 只是对齐要求从 8 变成了 4

布局通常是：

```text
c    1
pad  3
d    8
总计 12
```

而不是把 `double` 压成 4 字节。

## 8. 典型例题

### 8.1 例题一

```cpp
struct A {
    char a;
    double b;
    int c;
};
```

分析：

- `a` 占 1
- 为了让 `b` 按 8 对齐，补 7
- `b` 占 8
- `c` 占 4
- 当前总大小是 20
- 结构体按 8 对齐，所以尾部还要补 4

所以：

```cpp
sizeof(A) == 24
```

### 8.2 例题二

```cpp
#pragma pack(2)
struct A {
    char c;
    double d;
    int i;
};
```

分析：

- `c` 占 1
- `d` 按 2 对齐，所以补 1
- `d` 占 8
- `i` 占 4，按 2 对齐即可
- 总大小为 14
- 结构体整体按 2 对齐，14 已经满足条件

所以：

```cpp
sizeof(A) == 14
```

## 9. 面试速记版

- 内存对齐本质上是为了让 CPU 更高效访问数据
- 成员按自己的对齐要求放置
- 结构体对齐值通常等于最大成员对齐值
- 结构体大小必须是结构体对齐值的整数倍
- 尾部 padding 是为了保证结构体数组元素也能正确对齐
- `#pragma pack` 只改变对齐，不改变类型大小
- 最终结果仍然依赖具体编译器和 ABI

## 10. 一句话总结

默认情况下，数据通常按自然对齐排布；结构体成员各自对齐，整体按最大对齐值对齐，`#pragma pack` 只能压缩对齐要求，不能改变类型本身大小。
