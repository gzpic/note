### **13.1 进程和线程**

**进程**：正在运行的程序，是操作系统进行资源分配（比如内存、CPU时间、I/O设备）的基本单位。每个进程都有属于自己的代码、数据、内存空间等，彼此之间互相隔离。

**线程**：进程内部的一个执行流，是CPU进行调度和执行的基本单位。所有线程共享进程的内存空间，但独立地完成一部分任务。16线程的CPU在同一个时间点内至多有16个线程，操作系统中成百上千个线程是极速的时间片轮转所致。

### **13.2 死锁和避免死锁**

锁就是访问共享资源的“许可证”，持有锁就是某个进程（或线程）取得了访问某个共享资源的权限。

**死锁**：多个进程（或线程）持有锁，同时需要对方的锁，彼此互不相让，直接卡死。比如线程1拿着 Lock-A，需要 Lock-B；线程2拿着 Lock-B，需要 Lock-A。两个就这样互相等着卡死了，这就是死锁。

**死锁条件**：

1. 互斥条件，一个资源只能被一个进程使用，也就是该资源只有一把锁。
2. 请求与保持条件，吃着碗里的看着锅里的，请求的同时我拿着资源不放。
3. 不剥夺条件，已获得资源无法被其他进程强行拿走。
4. 循环等待条件，是死锁的最终表现形式。多个进程，每个进程都在等待下一个进程所持有的资源，构成**环状的依赖关系**。

**避免死锁**：

5. 死锁预防，打破四个条件
6. 死锁避免，使用算法来进行资源分配，防止系统进入不安全状态，如银行家算法；
7. 死锁检测和解除，抢占资源或者终止进程；

### **13.3 分页机制、虚拟内存和内存碎片**

操作系统通过分页管理机制进行内存管理

**分页机制与虚拟内存**：

- 把逻辑地址空间（程序视角）切成固定大小的页（比如 4KB）。
- 把物理内存也切成同样大小的帧。
- 通过页表建立“页 → 帧”的映射关系，不要求连续，从而实现离散分配，避免外部碎片。

本质上就是在物理内存和程序员中间加了一层抽象，可以称为虚拟内存。

**按需调页**：

- 允许程序把一部分页映射到内存上，另一部分页放到外存上。
- 当访问到的页不在内存时，CPU 会触发**缺页中断**，操作系统从外存调入该页
- 页表项中用“存在位”标记该页是否在内存中。这种机制实现了内存逻辑的扩展，即程序可用的地址空间 > 实际内存

**内存碎片**

内存碎片是在存储分配过程中产生的、不能供用户作业使用的小块区域。分为**内部碎片**和**外部碎片**。内部碎片和外部碎片分别基于物理内存和虚拟内存视角。

从**物理内存**视角看，内部碎片指的是分配给每一页的帧可能用不满，导致的浪费。

物理内存视角的外部碎片指的是内存中有足够的空闲空间总量，但并不连续，导致无法满足分配需求。比如中间一块空间释放了，但前后仍然未释放，较大的内存需求就无法在中间这块使用了。

分页机制基本解决了物理内存角度下的外部碎片，因为各个离散的页都是等效的，不存在中间释放了，前后限制导致用不了的情况（类似数组 vs 链表）。但内部碎片就增多了，一个比较极端的例子：比如物理内存上每一帧设置为 4KB，对应的页也就是 4KB。如果程序对应的页只用了 8 字节，则剩余的 4088 字节就全是内部碎片。

而**虚拟内存**视角上看，外部碎片表现形式基本一致，比如：

`new char[100];  // 分配128字节块
new char[50];   // 分配64字节块
delete[] ptr1;  // 释放128字节，留下空洞
// 后续申请130字节时，可能无法复用该空洞`

为了解决这种在虚拟地址空间上的外部碎片，可以采用内存池技术（具体见 7.1），本质上解决思路和分页机制是类似的，都是将空间规范化以消除外部碎片。

虚拟内存视角的内部碎片是更微观的，比如 2.7 提到的C++内存对齐原理：

`struct A {
    char a;      // 1字节
    int b;       // 4字节  ← 规则二：成员偏移量对齐
    char c;      // 1字节  ← 规则三：结构体总大小对齐
}; // 实际大小 = 1+3+4+1+3 = 12字节`

填充的这些空白部分就是逻辑内存视角的内部碎片。类似的用 `malloc` 和 `new` 申请但未使用的空间也算内部碎片。
