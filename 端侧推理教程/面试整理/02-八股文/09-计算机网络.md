# 计算机网络

## 1. 网络分层模型

### OSI 七层

- 应用层
  - 面向应用协议，如 HTTP、FTP、SMTP
- 表示层
  - 数据表示、编码、压缩、加密
- 会话层
  - 建立、维护、终止会话
- 传输层
  - 端到端传输，典型协议 TCP、UDP
- 网络层
  - 路由与寻址，典型协议 IP
- 数据链路层
  - 成帧、差错检测、MAC 寻址
- 物理层
  - 比特流传输

### 常见面试说法

- 实际工程里更常按 TCP/IP 分层来讲
- 但问“七层模型”时就按 OSI 七层答

## 2. Socket 基础

常见创建接口：

```c
int socket(int domain, int type, int protocol);
```

三部分含义：

- `domain`
  - 协议族 / 地址族，如 `AF_INET`、`AF_INET6`、`AF_UNIX`
- `type`
  - socket 类型，如 `SOCK_STREAM`、`SOCK_DGRAM`
- `protocol`
  - 具体协议，常填 `0`

### 服务端常见函数

- `socket`
- `bind`
- `listen`
- `accept`
- `read/write` 或 `recv/send`
- `close`

### 客户端常见函数

- `socket`
- `connect`
- `read/write` 或 `recv/send`
- `close`

## 3. I/O 模型

常见五类：

- 阻塞 I/O
- 非阻塞 I/O
- I/O 多路复用
- 信号驱动 I/O
- 异步 I/O

### 一句话理解

- 阻塞 I/O
  - 调用后线程直接等结果
- 非阻塞 I/O
  - 调用立即返回，自己反复检查
- I/O 多路复用
  - 用一个线程同时等多个 fd 的就绪事件
- 信号驱动 I/O
  - 就绪时用信号通知
- 异步 I/O
  - 内核把数据准备好后再通知应用，应用不自己做那次同步读取

## 4. 同步 / 异步，阻塞 / 非阻塞

- 同步 / 异步
  - 更偏向“结果完成后怎么通知调用方”
- 阻塞 / 非阻塞
  - 更偏向“当前调用会不会把线程挂起”

### 面试稳妥说法

- 同步
  - 调用方通常要主动等待结果
- 异步
  - 结果完成后通过通知、回调等方式告诉调用方
- 阻塞
  - 当前线程会被挂起等待
- 非阻塞
  - 调用立即返回，结果稍后再处理

## 5. BIO、NIO

- BIO
  - Blocking I/O，阻塞 I/O
- NIO
  - 常指非阻塞 + 事件驱动 + 多路复用风格

### 区别

- BIO 模型简单，但高并发下线程数和阻塞开销大
- NIO 更适合大量连接、少量活跃的网络场景

## 6. select、poll、epoll

### 共同点

- 都属于 I/O 多路复用机制
- 都能同时监听多个文件描述符的事件

### select

- 位图管理 fd
- 常见实现有 fd 数量上限
- 每次调用都要重新传入集合并遍历

### poll

- 用数组描述 fd
- 解决了 `select` 的位图上限问题
- 但仍需线性扫描

### epoll

- 把“注册关注事件”和“等待就绪事件”分开
- 更适合海量连接、少量活跃连接

### 为什么 epoll 常被认为更高效

- 不必每次重新拷贝整个关注集合
- 不必像 `select/poll` 那样每轮线性扫描所有 fd
- 活跃连接少时收益更明显

### LT 和 ET

- LT（水平触发）
  - 只要 fd 仍可读/可写，就可能持续通知
- ET（边缘触发）
  - 状态从“不可读”变“可读”时通知一次
  - 常要求配合非阻塞 I/O 和循环读写直到 `EAGAIN`

一句话：

- LT 更稳妥
- ET 更高效，但更容易写错

## 7. Reactor 和 Proactor

### Reactor

- 更常用于同步非阻塞 I/O
- 应用自己负责真正的读写
- 框架负责监听“就绪事件”并分发处理

### Proactor

- 更偏异步 I/O
- 应用关注的是“完成事件”
- 数据准备和搬运更多由内核/异步框架完成

### 工程上常见理解

- Reactor 感知“就绪”
- Proactor 感知“完成”

### 常见模式

- 单 Reactor 单线程
- 单 Reactor 多线程
- 多 Reactor 多线程 / 多进程

常见面试回答：

- Redis 常被说成单 Reactor 单线程
- Nginx 常被说成多 Reactor 多进程

## 8. 零拷贝

- 目标是减少用户态和内核态之间不必要的数据拷贝，以及减少上下文切换
- 常见手段有：
  - `sendfile`
  - `mmap + write`

### 好处

- 降低 CPU 开销
- 降低内存带宽消耗
- 提升文件传输和网络发送效率

### 注意

- “零拷贝”通常不是绝对 0 次复制
- 更准确是“尽量避免无意义的用户态参与拷贝”

## 9. Linux 网络协议栈

发送方向可粗略理解为：

- 应用层
- socket 层
- 传输层（TCP/UDP）
- 网络层（IP）
- 数据链路层
- 网卡驱动
- 网卡发送

接收方向则反过来：

- 网卡接收
- 驱动
- 数据链路层
- 网络层
- 传输层
- socket 缓冲区
- 应用层读取

## 10. TCP 最大连接数

影响因素通常包括：

- 文件描述符上限
- 端口资源
- 内存
- backlog / 内核参数
- 应用本身处理能力

### 面试稳妥说法

- 不要只答“65535”
- 服务端可承载连接数是系统资源、内核参数和应用架构共同决定的

## 11. 程序动态加载与链接

### 链接过程

- 把多个目标文件和库组合成可执行文件或共享库
- 分静态链接和动态链接

### 程序动态加载

- 程序启动时，加载器会把可执行文件和依赖共享库映射到进程地址空间
- 动态库通常在运行时由加载器解析符号并完成重定位

## 12. Linux 内核与架构

### Linux 内核主要模块

- 进程管理
- 内存管理
- 文件系统
- 设备驱动
- 网络协议栈

### Linux 架构可粗略看成

- 内核空间
  - 调度、内存、文件、驱动、网络等

## 13. 伙伴算法

- 伙伴算法是一种常见的内核页级内存分配算法
- 思路是把大块内存不断二分成 2 的幂次大小的块
- 分配时找最小满足需求的块
- 释放时若“伙伴块”也空闲，则合并回更大的块

优点：

- 合并效率高
- 适合页级分配

缺点：

- 可能产生内部碎片

## 14. 乐观锁和悲观锁

- 乐观锁
  - 先操作，提交时检查是否冲突
  - 常见实现：CAS、版本号
- 悲观锁
  - 先加锁再操作
  - 认为冲突很可能发生

适用场景：

- 读多写少：乐观锁更常见
- 写多冲突大：悲观锁更常见
