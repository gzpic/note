# 问题定位方法

## 1. 我通常怎么分层定位

建议固定排查顺序，不要一上来就盯某个 kernel：

1. 是整体慢，还是局部慢
2. 是 Prefill 慢，还是 Decode 慢
3. 是 CPU 提交慢，还是 GPU 执行慢
4. 是算力瓶颈，还是带宽瓶颈
5. 是显存容量问题，还是显存访问模式问题

面试里可以直接说：

- 我习惯先从宏观时间线确认瓶颈层级，再下钻到单个算子和单个 kernel。

## 2. 工具分工

### 2.1 `perf`

- 看 CPU 热点
- 看函数级采样
- 看上下文切换
- 看 cache miss、branch miss 等系统层行为

### 2.2 `nsys`

- 看整体 timeline
- 看 CPU/GPU 协同
- 看 kernel launch gap
- 看 memcpy、kernel、同步点的相对位置
- 看 stream overlap 是否做起来了

### 2.3 `ncu`

- 看单个 kernel 为什么慢
- 看 occupancy
- 看 stall 原因
- 看寄存器压力
- 看 shared memory 使用
- 看 DRAM / L2 吞吐
- 看 Tensor Core 利用率

## 3. 这些工具分别擅长什么

- `perf` 适合查 CPU 侧热点和调度问题。
- `nsys` 适合先回答“整体慢在哪”。
- `ncu` 适合回答“这个 kernel 为什么慢”。

不要混用结论：

- `nsys` 不擅长解释单个 kernel 内部细节。
- `ncu` 不适合看整机宏观时序。
- `perf` 不适合直接看 GPU kernel 内部执行。

## 4. 常见定位思路

### 4.1 TTFT 高

优先排查：

- 模型加载时间
- 首次 build / compile 开销
- Prefill 计算量
- CPU 侧输入准备和 launch 开销
- 是否有不必要的同步

### 4.2 Decode 延迟高或抖动

优先排查：

- KV Cache 读取模式
- L2 命中情况
- 小 kernel 是否太多
- 调度是否频繁抢占
- 显存回收和 block manager 是否有抖动

### 4.3 GPU 利用率低

优先排查：

- batch 太小
- kernel 粒度太碎
- launch 开销过大
- 依赖链太长
- occupancy 太低

### 4.4 OOM 或显存异常

优先排查：

- 历史 KV 是否释放
- 内存碎片
- workspace 预算是否超了
- 动态 shape 是否导致额外 buffer
- 调度器是否把并发打太高

### 4.5 OOM 和 page fault

#### OOM 是什么

- `OOM` = `Out Of Memory`
- 本质上就是内存不够，申请失败，或者系统因为内存压力过大把进程杀掉

在端侧大模型推理里，常见来源有：

- 模型权重太大
- `KV Cache` 随上下文增长太快
- batch 太大
- prompt 太长
- workspace / 临时 buffer 太大
- 并发请求太高
- 有内存泄漏或缓存没释放

#### page fault 是什么

- page fault 是访问页面时触发的缺页处理
- 访问的页当前不在可直接使用的位置，就需要由操作系统处理

粗理解可以分成：

- `minor page fault`：页基本已经在内存里，只是页表没准备好
- `major page fault`：需要从磁盘或外部存储调页，代价很大

在端侧大模型推理里，常见来源有：

- `mmap` 后首次访问触发按需调页
- 大块内存首次触碰，触发 lazy allocation
- `UMA / Unified Memory` 下发生页迁移
- 系统内存紧张导致页面被回收后重新访问
- 长上下文下 KV Cache / buffer 太大，页行为变差

#### 在端侧大模型里为什么特别常见

- 端侧内存本来就小
- `UMA` 下 CPU、GPU、DMA、外设共享同一套内存
- 大模型除了权重，还会持续增长 `KV Cache`
- decode 阶段访问模式更容易把 cache 和分页行为放大

一句话：

- OOM 更像“放不下”
- page fault 更像“放得下但访问代价很高”

#### 定位工具

- `tegrastats`：Jetson 上看内存、swap、CPU/GPU、频率、温度
- `free -h`：看剩余内存和 swap
- `vmstat`：看内存压力、缺页、swap 行为
- `top` / `htop`：看进程级内存占用
- `perf stat`：看 page faults、cache miss 等
- `perf record/report`：看 CPU 是否卡在缺页、mmap、内存管理热点
- `nsys`：看 page fault、memcpy、同步是否把关键路径拖长
- `dmesg`：看 OOM killer 记录
- `/proc/<pid>/smaps`、`pmap`：看进程映射和内存构成
- 框架日志 / 自己埋点：看加载、prefill、decode、KV Cache 分配阶段的耗时变化

#### 定位方法

1. 先判断是 `OOM` 还是 `page fault`
2. 看问题出现在加载期、prefill 期还是 decode 期
3. 拆内存构成：权重、KV Cache、workspace、临时 buffer
4. 看是否有 swap、major fault、OOM killer
5. 看是否是 `mmap` / lazy allocation / UMA 迁移导致的首次访问问题

更具体地说：

- 如果直接申请失败、进程被 kill，优先按 `OOM` 查
- 如果不一定崩，但首轮慢、偶发卡顿、长时间抖动，优先怀疑 page fault 或页迁移

#### 处理方法

如果是 `OOM`：

- 降模型大小
- 做权重量化
- 压缩或量化 `KV Cache`
- 限制上下文长度和并发
- 减少 workspace 和中间 buffer
- 做更激进的 memory plan
- 预分配静态 buffer，减少动态分配

如果是 page fault：

- 预热、预分配、提前触页
- 减少 `mmap` 后的随机访问
- 降低 `UMA / Unified Memory` 页面迁移
- 减少内存压力，避免 swap
- 固定 shape、复用 buffer、减少频繁申请释放

#### 面试里怎么答

- OOM 就是内存放不下，page fault 是页访问时触发的缺页处理。
- 在端侧大模型里，OOM 通常来自权重、KV Cache、workspace 或并发把内存顶满；page fault 常来自 `mmap`、按需分配、`UMA` 迁移和内存压力过大。
- 我会先用 `tegrastats`、`free`、`vmstat`、`dmesg` 判断系统内存压力，再结合埋点、`nsys`、`perf` 区分是加载、prefill 还是 decode 阶段出问题，最后再拆到权重、KV Cache 和 workspace 的占用。

## 5. 如何判断是算力瓶颈还是带宽瓶颈

### 5.1 更像带宽瓶颈

- DRAM 吞吐高
- L2 / global load 很重
- Tensor Core 利用率不高
- stall 以 memory dependency、long scoreboard 为主

### 5.2 更像算力瓶颈

- Tensor Core 或 FMA 利用率高
- DRAM 吞吐没有顶满
- 指令主要集中在 mma / fma
- 算子本身 FLOPs 很大

### 5.3 两边都不高

- 往往不是纯算力或纯带宽问题
- 更可能是并行度不足、launch gap、线程分歧或依赖链太长

### <span style="color:red">5.4 怎么判断一个算子是 memory-bound 还是 compute-bound</span>

先给结论：

> 不能只靠理论拍脑袋判断，理论分析只能帮助你先形成怀疑，最后还是要靠工具确认。

#### 理论上可以先怎么猜

可以先看这个算子的访问模式和计算密度：

- 如果数据搬运很多、重复访存重、计算相对少，先怀疑 `memory-bound`
- 如果 FLOPs 很大、矩阵乘占主导、Tensor Core 理论上应该能吃满，先怀疑 `compute-bound`

常见直觉：

- 大 GEMM 常常更像 `compute-bound`
- decode 阶段的 attention / KV cache gather 常常更像 `memory-bound`
- 小碎 kernel 往往两边都不高，更像 launch / 并行度问题

但这一步只能叫：

- 经验预判

不能直接当最终结论。

#### 真正要用什么工具确认

- 先用 `nsys` 找到到底是哪个 kernel 慢
- 再用 `ncu` 看这个 kernel 的内部指标

因为：

- `nsys` 负责告诉你“谁慢”
- `ncu` 负责告诉你“为什么慢”

#### 用 `ncu` 重点看什么

如果更像 `memory-bound`，常见现象是：

- DRAM 吞吐高
- L2 / global load 压力大
- Tensor Core 利用率不高
- stall 以 `memory dependency`、`long scoreboard` 为主

如果更像 `compute-bound`，常见现象是：

- Tensor Core 或 FMA 利用率高
- DRAM 吞吐没有顶满
- 指令主要集中在 `mma` / `fma`
- 算子 FLOPs 很大，SM 忙于计算

如果两边都不高，通常不是单纯的算力或带宽问题，而是：

- occupancy 不够
- kernel 太碎
- launch gap 大
- 依赖链太长
- 分支或同步拖慢了执行

#### 一套很实用的判断顺序

1. 先用经验看算子类型，形成初步怀疑
2. 再用 `nsys` 找热点 kernel
3. 再用 `ncu` 看 DRAM、L2、Tensor Core、stall 原因
4. 最后再下结论，不要反过来

#### 面试里怎么说最稳

- 我不会只靠理论说一个算子是 memory-bound 还是 compute-bound。
- 理论上我会先根据算子的计算密度和访存模式做预判，但真正下结论一定会用 `nsys` 找热点，再用 `ncu` 看 DRAM 吞吐、Tensor Core 利用率和 stall 原因。
- 如果 DRAM 很高而计算单元不满，我更倾向于 memory-bound；如果计算单元很满而 DRAM 没顶满，我更倾向于 compute-bound；如果两边都不高，我会继续查 occupancy、launch gap 和依赖链问题。

### <span style="color:red">5.5 怎么判断优化是不是已经接近极限了</span>

先给结论：

> 不存在绝对意义上的“完全极限”，更常见的说法是：已经比较接近当前硬件、当前算法路径和当前实现方式的上限。

#### 先看是不是接近硬件上限

如果更像 `compute-bound`，重点看：

- Tensor Core / FMA 利用率是否已经很高
- SM 活跃度是否已经很高
- DRAM 没顶满，但计算单元已经接近吃满

如果更像 `memory-bound`，重点看：

- DRAM 吞吐是否已经接近平台上限
- L2 命中率是否已经比较理想
- 再继续优化 kernel，收益是不是已经很小

#### 再看是不是接近算法上限

有些瓶颈不是实现差，而是问题本身决定的：

- decode 天然更偏 `memory-bound`
- 小 batch / 小矩阵很难把硬件完全打满
- 有些同步、依赖链和串行段本来就很难彻底消掉

如果瓶颈已经落在这些位置上，就要警惕：

- 可能不是代码写得不好
- 而是当前算法路径已经接近上限

#### 看优化收益是不是明显递减

这是工程上非常实用的标准：

- 前几轮优化收益很大，比如 `20%`、`30%`
- 后面再改很多代码，只剩 `1%`、`2%`、`3%`
- 甚至收益开始不稳定，某些 case 变快，某些 case 变慢

这通常说明：

- 你已经进入收益递减区

#### 看瓶颈是不是长期稳定

如果你多轮 profile 后发现：

- 热点总是稳定落在同一个 kernel
- 而且总是稳定落在同一个硬件瓶颈上
- 比如长期都是 DRAM 顶满，或者 Tensor Core 已经很高

那通常说明：

- 这一版实现已经比较接近当前路径的天花板

#### 看和成熟实现的差距还大不大

常见对比对象：

- `cuBLAS`
- `CUTLASS`
- `TensorRT`
- `FlashAttention`

如果你和这些成熟实现相比：

- 差距已经不大
- 而且 profile 也显示你已经接近硬件瓶颈

那继续深挖自定义优化的价值通常就不高了。

#### 一套很实用的判断顺序

1. 先确认当前热点是不是已经稳定
2. 再确认它更像算力上限还是带宽上限
3. 再看最近几轮优化收益是不是明显递减
4. 最后看和成熟库 / 理论峰值还有多大差距

#### 什么时候说明“这一版已经比较到顶了”

通常同时满足下面几条时，可以说比较接近极限：

- 主要热点已经逼近硬件瓶颈
- profile 结论长期稳定
- 后续优化收益持续很小
- 和成熟实现差距已经不大
- 再想提升，需要改算法、改数据布局或改系统设计，而不是只改几行 kernel 代码

#### 面试里怎么说最稳

- 我一般不会说“已经绝对到极限了”，而会说“已经比较接近当前平台和当前实现路径的上限”。
- 判断方法是先看 profile 是否稳定落在同一个硬件瓶颈上，再看和理论带宽或算力上限还有多少距离，最后看最近几轮优化收益是否明显递减。
- 如果热点长期稳定、DRAM 或 Tensor Core 已经接近上限、继续改代码只能换来很小收益，我会认为这一版实现已经比较到顶，下一步应该考虑改算法、改布局或者改系统调度。

### <span style="color:red">5.6 怎么判断该继续优化 kernel，还是该转去优化系统</span>

这个判断非常重要，因为很多时候问题已经不在单个 kernel 上了。

#### 适合继续抠 kernel 的情况

通常满足下面这些特征时，还值得继续做 kernel 级优化：

- 热点非常集中，单个或少数几个 kernel 占了大头
- `nsys` 看起来整体时间线比较干净，没有太多 launch gap 和同步空洞
- `ncu` 显示 kernel 内部还有明显优化空间
- 和成熟库相比差距还比较大

这时继续优化的方向通常是：

- 数据布局
- 向量化
- tile / block 参数
- shared memory / register 使用
- kernel 融合

#### 适合转去优化系统层的情况

如果出现下面这些现象，继续死抠单个 kernel 往往收益不高：

- 热点不集中，而是很多小 kernel 分散耗时
- `nsys` 上 launch gap、同步点、memcpy 占比明显
- CPU 提交、调度、前后处理占了不少时间
- 并发一上来系统就抖，不是某一个 kernel 单独变慢
- decode 阶段本质是 KV Cache、带宽、调度问题

这时更应该看的方向是：

- batch 组织
- 调度策略
- copy / compute overlap
- prefix cache / paged KV / memory plan
- 前后处理和 CPU 提交路径

#### 一个很实用的判断标准

可以直接问自己这三个问题：

1. 现在最大的时间到底还在不在某个 kernel 里？
2. 就算这个 kernel 再快 `10%`，整体收益会不会还是很小？
3. 当前主要损失是在计算本身，还是在 launch、同步、搬运、调度？

如果答案是：

- 热点不集中
- 单 kernel 提升带不动整体
- 时间主要耗在系统开销

那就该转去优化系统层。

#### 放到 LLM 场景里怎么理解

- `Prefill` 更常还有 kernel 级优化空间，因为 GEMM 和 attention 占比高
- `Decode` 更容易在系统层见顶，因为它常常受 KV Cache、带宽、调度和小 kernel 影响

所以很多时候：

- prefill 慢，先看 kernel
- decode 慢，先看系统

#### 面试里怎么说最稳

- 我会先看热点是不是集中在少数 kernel 上，如果是，并且 `ncu` 还显示有明显空间，那我会继续做 kernel 优化。
- 但如果时间已经被很多小 kernel、launch gap、同步、memcpy 和调度开销切碎了，我就不会继续死抠单个 kernel，而会转去优化 batch、调度、内存规划和 copy/compute overlap。
- 在大模型场景里，prefill 更适合继续抠算子，decode 更常需要从系统层解决问题。

### <span style="color:red">5.7 kernel launch 的步骤是什么，launch overhead 怎么减小</span>

#### kernel launch 大致经历哪些步骤

一次 kernel launch 通常不只是“调用一下函数”，中间大致会经历：

1. CPU 准备 launch 参数  
   包括 grid / block、kernel 参数、stream、函数入口等。
2. runtime / driver 处理调用  
   做状态检查、上下文确认、参数打包等。
3. 写入命令队列  
   把 launch 命令写进 GPU 可执行的 command buffer / queue。
4. stream / 依赖调度  
   判断这个 kernel 什么时候可以执行，前面有没有同步和依赖。
5. GPU 前端取命令并启动  
   再把 block / CTA 分发到 SM 上，kernel 才真正开始算。

所以：

> launch overhead 指的是 kernel 真正开始计算之前，这些 CPU 提交、driver/runtime 处理、命令下发和调度的固定成本。

#### 为什么 launch overhead 会变成问题

当单个 kernel 很大时，这部分开销通常不显著。

但如果是下面这些场景，就会很突出：

- 很多小 kernel
- 一串碎的 elementwise / norm / rope
- 小矩阵 GEMM
- decode 阶段大量短 kernel

常见表现：

- GPU timeline 上 kernel 很碎
- kernel 和 kernel 之间有明显 gap
- GPU 利用率不高，但延迟仍高
- CPU 在 runtime / driver 调用上花了不少时间

#### 怎么定位是不是 launch overhead

最常用的工具是：

- `nsys`
- `perf`

看法很直接：

- 用 `nsys` 看 timeline，判断 kernel 之间是否有明显 launch gap
- 看同步点是不是太多
- 看 memcpy、kernel、CPU 提交是否没有重叠起来
- 用 `perf` 看 CPU 是否花很多时间在 runtime、driver、提交和同步上

一句话：

- `nsys` 负责看时间线上的空洞
- `perf` 负责看 CPU 是否在忙着 launch

#### launch overhead 怎么减小

最核心的方法只有一条：

> 少 launch，少同步，把单次 launch 的工作量做大。

常见手段：

- 算子融合  
  把多个小 kernel 合成一个大 kernel。
- 减少 tiny kernel  
  尽量不要让 norm、bias、activation、rope 全都单独 launch。
- 用 `CUDA Graph`  
  对固定执行流程 capture，降低 CPU 侧重复 launch 开销。
- 减少不必要同步  
  少用 `cudaDeviceSynchronize()`，少做无意义 stream 同步。
- 合理用 stream  
  让 copy 和 compute、不同任务尽量 overlap。
- 增大单次工作量  
  通过 batch、融合或合并任务，把固定开销摊薄。
- 复用上下文和资源  
  避免反复初始化、反复 graph capture、反复 setup。

#### 放到 LLM 场景里怎么理解

- `Prefill` 更容易靠融合和大 kernel 把 launch overhead 摊薄
- `Decode` 更容易被很多小 kernel 和调度碎片化拖慢

所以在 LLM 里常见结论是：

- prefill 慢，先看大算子
- decode 慢，除了看 KV Cache 和带宽，也要重点看 launch gap

#### 面试里怎么答

- kernel launch 一般要经历 CPU 准备参数、runtime/driver 处理、命令写入队列、stream 依赖调度，再到 GPU 前端真正把任务分发到 SM。
- launch overhead 就是 kernel 真正开始算之前这部分固定成本。
- 它在大 kernel 上不明显，但在大量小 kernel 场景下会非常显眼。
- 我一般会先用 `nsys` 看 timeline 上有没有明显 launch gap，再用 `perf` 看 CPU 是否花很多时间在 runtime/driver 调用上；优化上主要靠算子融合、减少 tiny kernel、用 CUDA Graph 和减少不必要同步。

## 6. `ncu` 看指标时的判断逻辑

- `gld_efficiency` 低
  - 优先怀疑不连续访问和对齐问题
- local memory 高
  - 优先怀疑寄存器 spill
- eligible warps 低
  - 可运行 warp 不够，隐藏不了延迟
- control 指令占比高
  - 分支和循环结构可能有问题
- shared memory 使用高但收益不明显
  - 可能存在 bank conflict 或同步成本过高

## 7. 大模型场景下的固定排查模板

### 7.1 Prefill 慢

- 先看 GEMM 是否走到高性能路径
- 再看 attention 是否用了合适实现
- 再看 norm、rope、elementwise 是否小 kernel 太多
- 最后看有没有融合空间

### 7.2 Decode 慢

- 先看 KV Cache 读写
- 再看小矩阵 GEMM 和 attention kernel
- 再看 launch 开销和 graph / stream 优化空间
- 最后看调度策略和 batch 组织方式

## 8. 面试里可直接背的定位答案

- 我定位问题通常分三层：先用 `nsys` 看整条链路慢在哪，再用 `ncu` 看关键 kernel 慢在哪里，最后如果怀疑 CPU 侧准备或调度问题，再用 `perf` 看热点。
- 如果是 Prefill 慢，我会重点看 GEMM、Attention 和融合。
- 如果是 Decode 慢，我会重点看 KV Cache、L2 命中率、小 kernel 数量和 launch gap。
- 如果 GPU 利用率不高，我不会直接下结论说算力不足，而是先判断是不是 batch 太小、occupancy 太低或者依赖链太长。
