# 计算机网络

## 高频题

### 1. TCP 和 UDP 区别

- `TCP`
  - 面向连接
  - 可靠传输
  - 有流量控制和拥塞控制
- `UDP`
  - 无连接
  - 不保证可靠
  - 更轻量，时延更低

### 2. 三次握手和四次挥手

- 三次握手核心是双方确认收发能力和初始序列号
- 四次挥手核心是 TCP 连接是全双工，关闭两个方向通常要分开做
- 高频追问：
  - 为什么不是两次握手
  - 为什么会有 `TIME_WAIT`

### 3. `socket` 基础流程

- 服务端：
  - `socket`
  - `bind`
  - `listen`
  - `accept`
- 客户端：
  - `socket`
  - `connect`

### 4. 同步/异步 和 阻塞/非阻塞

- 同步/异步
  - 更偏结果通知方式
- 阻塞/非阻塞
  - 更偏当前线程是否挂起

面试最常见误区就是把这两组概念混在一起。

### 5. `select`、`poll`、`epoll`

- 都属于 I/O 多路复用
- `select`
  - fd 数有限
  - 每次都要重传集合并扫描
- `poll`
  - 没有位图上限
  - 但仍然线性扫描
- `epoll`
  - 注册和事件返回分离
  - 更适合大量连接、少量活跃

### 6. LT 和 ET

- `LT`
  - 只要还可读可写，就可能持续通知
- `ET`
  - 从不可读变可读时通知一次
  - 一般要配合非阻塞和循环读写直到 `EAGAIN`

### 7. Reactor 和 Proactor

- `Reactor`
  - 感知“就绪”
  - 应用自己去读写
- `Proactor`
  - 感知“完成”
  - 更偏异步 I/O 完成通知

### 8. 零拷贝

- 核心目标是减少用户态和内核态间不必要的数据复制与切换
- 常见做法：
  - `sendfile`
  - `mmap + write`
- 更稳的说法：
  - 零拷贝通常不是字面上的完全 0 次复制

## 回答骨架

### 网络题怎么答得稳

1. 先说协议或机制解决什么问题
2. 再说核心流程
3. 再说优缺点
4. 最后补适用场景

例如“为什么 `epoll` 常被认为更高效”：

- 因为它把关注集合维护和就绪事件返回分开了。
- 不必像 `select/poll` 那样每次都重新拷贝和线性扫描全部 fd。
- 所以在连接很多但活跃连接不多的场景里更占优。

## 高频口头结论

### 为什么需要三次握手

- 两次不够，因为服务端无法确认客户端是否具备接收能力
- 三次之后双方都能确认收和发都正常

### 为什么需要 `TIME_WAIT`

- 保证最后一个 ACK 有机会到达
- 避免旧连接残留报文影响新连接

### BIO 和 NIO 怎么讲

- `BIO` 更简单，但每连接线程模型扩展性差
- `NIO` 更适合高并发网络 I/O

## 易错点

- 不要把同步/异步和阻塞/非阻塞答成同义词
- 不要把 `epoll` 的优势说成“完全没有扫描成本”
- 不要把零拷贝答成“绝对不发生任何复制”
