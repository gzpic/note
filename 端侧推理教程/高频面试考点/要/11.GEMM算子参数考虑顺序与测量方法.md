# GEMM写算子时参数考虑顺序与测量方法

这篇笔记的核心不是列参数，而是回答两个问题：

- GEMM 算子调参时，各个参数应该按什么顺序考虑
- 每一步应该看什么指标，怎么判断方向对不对

## 一、总原则

写 GEMM 算子时，参数不是彼此独立乱调的，而是有一个比较固定的考虑顺序：

1. 先定问题
2. 再定 block 级映射
3. 再定 warp / thread 级映射
4. 再定 shared memory / global memory 搬运
5. 最后定 pipeline、unroll、epilogue 这类深优化

一句话记忆：

先定 `dtype` 和硬件路线，再定 `block tile`，再定 `warp/thread tile`，再看 `reg` 和 `smem` 能不能扛住，接着修 `global/smem` 搬运，最后用 pipeline 把 latency 藏起来。

## 二、第一步：先定问题本身

### 1. 数据类型

先定：

- FP32
- FP16
- BF16
- INT8

以及：

- 走 Tensor Core 还是普通 CUDA Core
- accumulate 类型是什么

为什么这是第一优先级：

- 决定能不能走 `mma.sync`
- 决定向量化宽度
- 决定寄存器压力
- 决定后面 tile 设计和 load/store 方式

### 2. 矩阵规模和形状

先看：

- `M`
- `N`
- `K`
- 是否是方阵
- 是否是 skinny GEMM
- 是否是 batched GEMM

为什么要先看：

- 大方阵更容易把吞吐打满
- 小矩阵更容易受 launch overhead、occupancy、边界浪费影响
- `K` 很大时更适合做深流水和更高复用
- `M/N` 很小时 block tile 不能随便放大

### 3. 数据布局

要先明确：

- A/B/C 是 row major 还是 column major
- leading dimension 是否连续
- 地址是否对齐到 16B / 32B / 128B

它决定：

- global load 是否 coalesced
- shared memory 怎么排
- `ldmatrix`、向量化 load 是否顺手
- 是否要预转置

## 三、第二步：先定 block 级 tile

最先要定的核心参数通常是：

- `BlockM`
- `BlockN`
- `BlockK`

也就是一个 thread block 负责算多大一块 `C`。

### 1. block tile 影响什么

#### 数据复用

block tile 大一些：

- A/B 在 shared memory 中的复用更高
- arithmetic intensity 更高

但也会带来：

- shared memory 占用更高
- registers 更高
- occupancy 下降

#### 并行粒度

block tile 太大：

- block 数量变少
- 小矩阵时 SM 吃不满

block tile 太小：

- 数据复用差
- global traffic 大
- 控制开销占比变高

#### 是否贴合硬件指令粒度

如果走 Tensor Core，block tile 最好能由硬件 MMA shape 自然拼起来。

### 2. block tile 怎么测

重点测：

- kernel 总耗时
- TFLOPS
- occupancy
- DRAM throughput
- L2 throughput
- shared memory usage
- registers per thread

判断逻辑：

- block tile 变大后，TFLOPS 提升且 DRAM 压力下降，说明 reuse 在变好
- 再继续变大性能反而掉，通常是：
  - occupancy 太低
  - smem 太大
  - reg 太多
  - 边界浪费严重

## 四、第三步：再定 warp 级 tile

block tile 之后，再看一个 warp 负责 `C tile` 中多大区域。

常见参数：

- `WarpM`
- `WarpN`
- `WarpK`

### 1. warp tile 影响什么

#### warp 内部的数据复用和通信

它决定：

- 一个 warp 要持有多少 accumulator
- A/B fragment 多大
- `mma.sync` 调用次数
- warp 内 mapping 是否规则

#### 寄存器压力

warp tile 越大：

- 每个线程持有的累加寄存器越多
- reg pressure 会迅速上升

#### 指令发射效率

warp tile 太小：

- MMA 次数太碎
- loop/control 开销更明显

warp tile 太大：

- 寄存器爆
- 活跃 warp 数下降
- latency hiding 变差

### 2. warp tile 怎么测

重点看：

- `registers per thread`
- `eligible warps per cycle`
- `sm__throughput`
- tensor 指令占比
- `stall_exec_dependency`
- `stall_not_selected`

判断逻辑：

- `eligible warps per cycle` 很低，通常说明依赖重或 reg 压力太大
- tensor 指令比例很高但吞吐还是上不去，通常是访存没喂饱
- warp tile 一放大性能反降，常见就是 reg 压力过大

## 五、第四步：再定 thread tile / per-thread work

也就是每个线程负责几个输出元素，或者持有多少 accumulator。

例如：

- `1x1`
- `2x2`
- `4x4`

### 1. thread tile 影响什么

#### 寄存器级复用

load 一次 A/B 后，多做几次 FMA，寄存器复用更好。

#### 减少总指令数

如果 thread tile 合理，load/store 和控制开销能被摊薄。

#### 寄存器占用快速上升

thread tile 是寄存器压力最直接的来源之一。

### 2. thread tile 怎么测

重点看：

- `registers per thread`
- local memory spill
- load 指令占比
- kernel 时间

判断逻辑：

- 一旦出现 spill，说明 thread tile 太激进
- 如果 load 占比太高，通常是 thread tile 太小、reuse 不够

## 六、第五步：再定 BlockK

`BlockK` 决定每轮从 global memory 拉多少 K 方向数据到 shared memory。

### 1. BlockK 影响什么

#### shared memory 占用

A/B tile 的 smem 基本都会随着 `BlockK` 增大而变大。

#### global memory 往返次数

`BlockK` 大一点：

- 每轮做更多计算
- loop 轮数更少

#### pipeline 深度和 async copy 效果

`BlockK` 太小：

- 每轮工作量太少
- 不利于摊平搬运开销

`BlockK` 太大：

- smem 压力大
- occupancy 可能明显下降

### 2. BlockK 怎么测

重点看：

- smem usage
- DRAM throughput
- `stall_memory_dependency`
- `stall_long_scoreboard`
- kernel 总耗时

判断逻辑：

- memory dependency 高，可能是 `BlockK` 太小，搬运太频繁
- smem 很高且 occupancy 掉很多，可能是 `BlockK` 太大

## 七、第六步：shared memory layout

这一层不是装饰项，是核心优化点。

要考虑：

- A tile 怎么放
- B tile 怎么放
- 是否转置存
- 是否 padding
- 是否 swizzle
- 是否适配 `ldmatrix`

### 1. 为什么这步很重要

性能损失经常发生在：

- global -> smem 不连续
- smem -> reg 有 bank conflict
- Tensor Core fragment 装载不匹配

### 2. 怎么测

重点看：

- shared memory bank conflict
- smem load/store efficiency
- kernel 时间

判断逻辑：

- bank conflict 高时，优先改 smem layout
- 很多时候加 padding 或 swizzle，比继续调 tile 更有效

## 八、第七步：global memory 搬运方式

这里主要考虑：

- 标量 load 还是向量化 load
- `float4` / `half2` 等向量化宽度
- 是否用 `cp.async`
- 是否双缓冲

### 1. 目标是什么

#### global load coalesced

warp 内线程访问尽量连续。

#### 一次搬更多

向量化 load 往往能减少指令数，提高带宽利用率。

#### 搬运和计算重叠

这就是 `cp.async + pipeline` 的意义。

### 2. 怎么测

重点看：

- `gld efficiency`
- DRAM throughput
- 指令数
- `stall_memory_dependency`

判断逻辑：

- `gld efficiency` 低，优先怀疑访问模式或对齐
- memory stall 高但 DRAM 吞吐没满，往往是 latency hiding 不够，不是纯带宽打满

## 九、第八步：最后调 pipeline stage

常见选择：

- 单缓冲
- 双缓冲
- 三阶段 pipeline

### 1. stage 影响什么

#### latency hiding

stage 多一点，能更好把 global memory latency 藏到计算后面。

#### smem 占用

stage 越多，buffer 越多，smem 会上涨。

#### 实现复杂度

stage 数高了之后，barrier / wait / fence 逻辑更复杂。

### 2. 怎么测

重点看：

- `stall_memory_dependency`
- `stall_long_scoreboard`
- `sm__throughput`
- occupancy
- 总耗时

判断逻辑：

- stage 从 1 到 2 往往收益明显
- 从 2 到 3 不一定总有收益，常常受限于 smem 和实现复杂度

## 十、推荐的参数考虑顺序

### 第一层：先定大方向

- 数据类型
- Tensor Core 还是 CUDA Core
- 矩阵形状和 layout

### 第二层：先定粗粒度映射

- `BlockM`
- `BlockN`
- `BlockK`

### 第三层：再定中粒度映射

- `WarpM`
- `WarpN`
- thread tile

### 第四层：再定搬运和缓存

- global memory vectorize 方式
- shared memory layout
- 是否 `cp.async`

### 第五层：最后定深优化

- stage 数
- unroll 因子
- epilogue 写回方式

## 十一、每一步到底测什么

| 调参项 | 主要看什么 | 常见问题 |
|---|---|---|
| 数据类型 | 是否能走 Tensor Core、理论峰值 | 路线选错，天花板直接低 |
| BlockM/N | TFLOPS、occupancy、DRAM throughput | 太小 reuse 差，太大 occupancy 掉 |
| BlockK | smem、memory stall | 太小搬运频繁，太大 smem 爆 |
| Warp tile | reg、eligible warps、tensor 指令占比 | 太大 reg 压力高 |
| Thread tile | reg、spill、load 占比 | 太大 spill，太小 reuse 差 |
| Smem layout | bank conflict、smem 效率 | bank conflict |
| Vectorized load | gld efficiency、指令数 | 不连续、没对齐 |
| Pipeline stage | memory stall、总耗时 | stage 少隐藏不够，stage 多资源不够 |

## 十二、实际调参流程

### 第一轮：先做 baseline

先写一个能跑的版本：

- 正确性优先
- 不追求最优
- 能稳定测时间

比如：

- 一个基础 shared memory tiled GEMM
- 或一个基础 Tensor Core GEMM

先测：

- 正确性
- kernel 时间
- TFLOPS

### 第二轮：只扫 block tile

固定其它参数，扫：

- `BlockM ∈ {64, 128}`
- `BlockN ∈ {64, 128, 256}`
- `BlockK ∈ {16, 32, 64}`

目标：

- 找出甜点区间

### 第三轮：扫 warp / thread tile

在 block tile 甜点区继续扫：

- `WarpM/WarpN`
- thread accumulator 数量

重点盯：

- reg
- spill

### 第四轮：调 smem layout 和 global load

这一步常能拿到很直接的收益：

- padding
- swizzle
- vectorized load
- `cp.async`

### 第五轮：最后调 pipeline stage 和 unroll

这一步是榨汁阶段，不是第一阶段。

## 十三、正确性怎么测

至少要做：

- 和 CPU reference 比较
- 多组尺寸：
  - 小矩阵
  - 大矩阵
  - 非对齐矩阵
  - 边界尺寸
- 多组随机输入
- 看绝对误差和相对误差

特别是 Tensor Core / 混合精度路径，不要只测一个 case。

## 十四、性能怎么测

### 1. kernel 时间

用 CUDA Event 测 kernel 时间，不要把 H2D / D2H 混进去。

### 2. TFLOPS

公式：

```text
TFLOPS = 2 * M * N * K / (time * 1e12)
```

如果是 batched GEMM，再乘上 batch。

### 3. profiling 指标

常看：

- `sm__throughput`
- `dram__throughput`
- `eligible_warps_per_cycle`
- `stall_memory_dependency`
- `stall_long_scoreboard`
- tensor instruction 占比
- registers per thread
- smem usage

## 十五、最核心的判断逻辑

调 GEMM 时，本质上一直在判断三件事：

### 1. 是算力没吃满，还是内存没喂饱

- compute throughput 低
- DRAM throughput 也低

通常更像：

- 并行度问题
- 映射问题
- 流水问题

如果：

- DRAM throughput 很高
- compute 上不去

通常更像 memory-bound。

### 2. 是资源限制了，还是数据复用不够

- occupancy 低、reg 高、smem 高
  - 更像资源瓶颈
- occupancy 不低，但 DRAM traffic 很大
  - 更像 reuse 不够

### 3. 是 latency 问题，还是 bandwidth 问题

- `stall_memory_dependency` / `long_scoreboard` 高，但带宽没满
  - 更像 latency hiding 不够
- DRAM 已经接近峰值
  - 才更像真正 bandwidth-bound

## 十六、实战顺口版

可以直接这么记：

先定 `dtype` 和硬件路线，  
再定 `block tile`，  
再定 `warp/thread tile`，  
再看 `reg` 和 `smem` 能不能扛住，  
然后修 `global/smem` 搬运，  
最后用 pipeline 把 latency 藏起来。
