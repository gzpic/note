fp64，fp32,fp16,bf16的区别

|类型|总位数|指数位|尾数位|精度|范围|常见用途|
|---|--:|--:|--:|---|---|---|
|FP64|64|11|52|最高|很大|科学计算、数值仿真|
|FP32|32|8|23|高|大|传统深度学习、通用计算|
|FP16|16|5|10|较低|较小|深度学习训练/推理加速|
|BF16|16|8|7|比 FP16 更粗|和 FP32 接近|大模型训练、推理|

temperature调参怎么调整
常用推荐范围可以直接这么记呢：

- **0.0 ~ 0.2**：很稳  
    适合：代码、数学、抽取、JSON、tool call、agent执行链路
- **0.3 ~ 0.5**：稳中带一点自然  
    适合：技术问答、解释说明、总结
- **0.6 ~ 0.8**：比较自然  
    适合：普通聊天、泛化问答、改写润色
- **0.8 ~ 1.0**：偏发散  
    适合：创意写作、brainstorm、文案
- **1.0 以上**：一般不太推荐常用  
    更容易跑偏、胡说、重复、不稳定呢
### 具体含义

#### ① 符号位 sign

决定正负：

- 0：正数
- 1：负数

#### ② 指数位 exponent

决定这个数的**范围**，也就是能表示很大或很小的数。

#### ③ 尾数位 mantissa

决定这个数的**精度**，也就是小数部分能有多细。

## 架构与基础
- SIMT vs SIMD：SIMD 是单线程向量化；SIMT 是多线程同指令（warp 32线程）。
- CUDA 线程局部变量存哪：优先寄存器；寄存器不够会 spill 到 local memory（本质是显存）。
- gmem vs smem：gmem 容量大延迟高；smem 容量小延迟低，适合块内复用。
- block 内通信：`__syncthreads` + shared memory，或 warp 内 `shuffle`。
- warp shuffle：线程直接交换寄存器数据，低延迟，常用于规约/扫描。
- 向量化访存：用 `half2/float4` 提高吞吐，前提是地址对齐。
- compute-bound / memory-bound：分别受算力或带宽限制；还有 launch-bound、latency-bound、sync-bound。

## GEMM与参数选择
- GEMM 选参数：核心看 tile 大小、blockDim、寄存器压力、shared memory 占用和 occupancy 平衡。
- 确定最优线程数：从 128/256/512 起测，结合 occupancy + NCU 指标选最优。
- SM 没用满但增大矩阵时延增长：工作量确实增加，且可能触发更高带宽/缓存/同步开销。
- 分块如何保连续性：按行/列连续布局切 tile，保证 warp 访问 coalesced。
- 特定 shape 用 smem 异常排查：先查越界、bank conflict、对齐、同步遗漏、类型转换误差。

## FlashAttention与注意力
- FlashAttention 为什么快：重排计算顺序，在线 softmax，减少中间矩阵落显存。
- FA2 改进：更好的并行划分和内核调度，吞吐更高。
- FA3 改进：继续优化并行和硬件利用，面向新架构低精度路径更友好。
- FA4 是否了解：可答“关注方向是更强异步流水、低精度和长上下文性能”。
- Flash decoding combine kernel 占比：常见会是 decode 热点之一，比例依模型/序列长度而变，需用 NCU/NSYS 实测。
- MHA/GQA/MLA：MHA 每个 Q 头独立 KV；GQA 多个 Q 头共享 KV；MLA 目标是进一步降 KV 带宽与存储。

## 精度与硬件
- FP32/FP16/BF16/FP8/FP4：精度和范围依次变化，越低精度吞吐通常越高但误差风险越大。
- Nano 支持哪些：Jetson Orin Nano 常用 FP16/INT8 推理路径，是否支持更低精度看具体框架版本与内核支持。
- Tensor Core 使用前提：数据类型、维度对齐、内核实现都要匹配。

## 内存与传输
- bank conflict 原因：同 warp 多线程访问同一 bank 不同地址导致串行。
- padding/swizzle：通过布局扰动把冲突访问分散到不同 bank。
- 何时关闭 smem：当复用不足、冲突重、同步成本高时，直接走寄存器/L1 可能更快。
- H2D/D2H 能否重叠：可，需异步 memcpy + 不同 stream + pinned memory。
- 为什么 memcpy 常配 pinned：减少分页开销并支持更高 DMA 传输效率。
- cudaDeviceMaxConnections：控制主机到设备连接并发度，影响并发调度与重叠效果。

## 执行与编译
- kernel launch 开销：小 kernel 频繁发射时 CPU launch 会成瓶颈。
- CUDA Graph：把重复 kernel/memcpy 录图，减少 launch 开销。
- CUDA stream 使用前提：操作异步化、资源不冲突、正确同步依赖。
- CUDA 编译步骤：`nvcc` -> 生成 PTX/SASS -> 驱动加载执行。
- PTX vs SASS：PTX 是虚拟中间指令；SASS 是具体 GPU 架构机器码。

## 常用优化方法
- 算子融合：减少中间张量读写和 launch 次数。
- 张量复用：减少重复分配和访存，降低显存峰值。
- 算子融合难点：寄存器爆、occupancy 下降、数值一致性和调试复杂度上升。

## 工具与定位
- NSYS：看时间线、并发、空泡、H2D/D2H 与 kernel 重叠。
- NCU：看单 kernel 的 stall、带宽利用率、指令统计、occupancy。
- perf：看 CPU 侧热点，配合 NSYS 分析端到端瓶颈。
- CUDA 问题定位流程：先宏观（NSYS）再微观（NCU）再回归代码定位。
- profiling 关注项：带宽利用率、流水线空泡、指令耗时构成。
- 显存越界排查：开越界检测/ASan，检查索引、步长、对齐、batch 边界。

## 手撕与实战
- 手撕内存池：预申请大块显存，维护空闲链表/分级桶，减少频繁 `cudaMalloc`。
- prefill 优化：大算子吞吐、融合、批处理、图优化。
- decode 优化：KV 访问模式、小 kernel 合并、降低 launch 开销。
- transformer 最大计算量：通常集中在 GEMM（QKV/FFN）和 attention 核心。
