# 模型部署量化与推理速记

## Transformer与结构
- Transformer vs RNN：Transformer 并行强、长依赖建模好；RNN 时序强但并行差。
- 为什么 SwiGLU：门控提升表达能力，通常比 ReLU/GELU 在同算力下效果更好。
- Decoder-only 为什么多：生成任务统一、训练和推理链路简单、生态成熟。
- RoPE 是什么：旋转位置编码，把位置信息注入 Q/K，长上下文外推更友好。

## 模型规格速记

### Qwen3-4B 基本参数

- 类型：纯文本 `Causal Language Model`
- 参数量：`4.0B`
- 非 embedding 参数：`3.6B`
- 层数：`36`
- 注意力头：`Q 32`，`KV 8`，属于 `GQA`
- 原生上下文：`32,768`
- 使用 YaRN 后可扩到：`131,072`
- 特点：
  - 同一模型支持 thinking / non-thinking 两种模式
  - 支持 `100+` 种语言和方言

面试里可直接记成：

- `Qwen3-4B` 是一版 4B 级别的纯文本 dense 模型，36 层，GQA 结构，原生 32K，上下文通过 YaRN 可扩到 128K。

### Qwen3.5-4B 基本参数

注意：

- `Qwen3.5-4B` 官方卡是 `Causal Language Model with Vision Encoder`
- 所以它不是单纯和 `Qwen3-4B` 完全对位的纯文本模型

核心规格：

- 类型：带视觉编码器的因果语言模型
- 参数量：`4B`
- 层数：`32`
- Hidden Dimension：`2560`
- Token Embedding：`248320`
- Hidden Layout：
  - `8 × (3 × (Gated DeltaNet → FFN) → 1 × (Gated Attention → FFN))`
- Gated DeltaNet 头数：
  - `V 32`
  - `QK 16`
- Gated Attention 头数：
  - `Q 16`
  - `KV 4`
- 原生上下文：`262,144`
- 官方可扩展上下文：`1,010,000`
- 额外特性：
  - 默认支持 thinking
  - 支持 `MTP`
  - 可以 text-only 模式跳过 vision encoder 节省 KV cache 空间

面试里可直接记成：

- `Qwen3.5-4B` 不是简单的 Qwen3 文本版迭代，它官方是带 vision encoder 的 4B 模型，原生 256K，上下文官方写可扩到约 101 万，结构上是 Gated DeltaNet + Gated Attention 的 hybrid 路线。

### Qwen3.5 相对 Qwen3 的改进点

按官方描述，主要是下面几类：

#### 1. 从纯文本走向统一多模态

- `Qwen3-4B` 官方卡是纯文本模型
- `Qwen3.5-4B` 官方卡是带 vision encoder 的模型
- 官方强调：
  - `Unified Vision-Language Foundation`
  - 早融合 multimodal token

一句话：

- `Qwen3.5` 最大变化之一，是把视觉和语言统一到了一个基础模型框架里，不再只是单纯文本模型延长上下文。

#### 2. 架构从传统 attention 主体转向 hybrid 路线

- `Qwen3-4B` 是更常规的 Transformer / GQA 思路
- `Qwen3.5-4B` 官方写的是：
  - `Gated DeltaNet`
  - `Gated Attention`
  - sparse `Mixture-of-Experts`

一句话：

- `Qwen3.5` 的结构重点是更高吞吐、更低时延和更低成本，不只是堆参数。

#### 3. 上下文能力大幅增强

- `Qwen3-4B`：
  - 原生 `32K`
  - YaRN 后 `128K`
- `Qwen3.5-4B`：
  - 原生 `262K`
  - 官方写可扩到 `1,010,000`

一句话：

- 如果只看长上下文能力，`Qwen3.5-4B` 比 `Qwen3-4B` 提升是非常明显的。

#### 4. 多语言覆盖更广

- `Qwen3` 官方卡写 `100+` 语言和方言
- `Qwen3.5` 官方卡写扩到 `201` 语言和方言

#### 5. RL 和 agent 能力继续强化

- 官方提到：
  - `Scalable RL Generalization`
  - 面向复杂 agent 环境的 RL 扩展

一句话：

- `Qwen3.5` 不只是聊天模型升级，更偏向“长上下文 + agent + 多模态 + 大规模 RL”统一推进。

### 一个面试里更稳的答法

如果被问：

- `Qwen3.5` 相对 `Qwen3` 到底改了什么

可以直接答：

- 如果以官方 4B 卡来看，`Qwen3-4B` 是纯文本模型，重点是 thinking / non-thinking 双模式、GQA 和 32K 原生上下文；`Qwen3.5-4B` 则已经变成带 vision encoder 的多模态模型，结构上引入了 Gated DeltaNet 和 Gated Attention 的 hybrid 架构，原生上下文提升到 262K，并且官方给了百万级扩展上下文能力。
- 所以 `Qwen3.5` 相对 `Qwen3` 的变化不只是“参数再调一版”，而是同时推进了多模态、长上下文、推理架构效率、多语言覆盖和 agent/RL 能力。

### 一个要注意的细节

- `Qwen3` 官方支持 soft switch：
  - `/think`
  - `/no_think`
- `Qwen3.5` 官方卡里明确写了：
  - 不官方支持 `Qwen3` 那套 soft switch
  - 需要通过 API 参数关闭 thinking

所以更准确地说：

- `Qwen3.5` 是整体能力和架构上的升级
- 但接口行为并不是所有地方都和 `Qwen3` 完全一致

### Qwen3-4B vs Qwen3.5-4B 字符架构图

#### 1. Qwen3-4B

```text
输入 tokens
   │
   ▼
Token Embedding
   │
   ▼
+----------------------------------+
| 36 x Transformer Decoder Block   |
|                                  |
|  RMSNorm                         |
|    │                             |
|    ▼                             |
|  GQA Self-Attention              |
|  (Q:32, KV:8)                    |
|    │                             |
|    ▼                             |
|  Residual                        |
|    │                             |
|    ▼                             |
|  RMSNorm                         |
|    │                             |
|    ▼                             |
|  FFN / SwiGLU                    |
|    │                             |
|    ▼                             |
|  Residual                        |
+----------------------------------+
   │
   ▼
Final Norm
   │
   ▼
LM Head
   │
   ▼
输出 logits / token
```

一句话：

- `Qwen3-4B` 可以理解成一条比较标准的 dense decoder-only Transformer 主干，重点是 `36` 层、`GQA(Q32/KV8)`、`32K` 原生上下文，以及 thinking / non-thinking 双模式。

#### 2. Qwen3.5-4B

```text
文本 / 图像 / 视频输入
        │
        ├───────────────┐
        │               │
        ▼               ▼
 Text Tokens      Vision Encoder
        │               │
        └──── Early Fusion ───────┐
                                  ▼
                           Unified Token Stream
                                  │
                                  ▼
      +--------------------------------------------------+
      | 32 层 Hybrid Language Model                      |
      |                                                  |
      | 8 个 stage，每个 stage 结构：                    |
      |                                                  |
      |   3 x (Gated DeltaNet -> FFN)                    |
      |   1 x (Gated Attention -> FFN)                   |
      |                                                  |
      | Gated DeltaNet: V32 / QK16                       |
      | Gated Attention: Q16 / KV4                       |
      +--------------------------------------------------+
                                  │
                                  ▼
                           LM Head (tied)
                                  │
                                  ▼
                           输出 logits / token
```

一句话：

- `Qwen3.5-4B` 已经不是“纯文本 Qwen3 小改版”，而是“带 vision encoder 的统一多模态模型 + hybrid 主干”，核心层结构是 `Gated DeltaNet + FFN` 与 `Gated Attention + FFN` 交替组合。

#### 3. 放在一起怎么对比

```text
Qwen3-4B
  纯文本输入
    -> Token Embedding
    -> 36 x 标准 Transformer Decoder Block
    -> LM Head

Qwen3.5-4B
  文本/图像/视频输入
    -> Vision Encoder + Text Tokens
    -> Early Fusion
    -> 32 层 Hybrid 主干
       = 8 x [3 x (Gated DeltaNet -> FFN) + 1 x (Gated Attention -> FFN)]
    -> LM Head
```

#### 4. 最值得记的结构差异

- `Qwen3-4B`
  - 纯文本
  - 标准 dense decoder-only Transformer 路线
  - 36 层
  - GQA：`Q32 / KV8`
  - 原生 `32K`
- `Qwen3.5-4B`
  - 多模态统一输入
  - vision encoder + early fusion
  - 32 层 hybrid 主干
  - `Gated DeltaNet + Gated Attention`
  - 原生 `262K`

#### 5. 面试里可直接口述

- 如果只看结构，`Qwen3-4B` 更像标准的纯文本 decoder-only Transformer，只是在 Qwen3 里加入了 thinking / non-thinking 双模式和更强的 GQA 配置；而 `Qwen3.5-4B` 则明显更激进，它把 vision encoder 融进来了，同时把主干改成了 Gated DeltaNet 和 Gated Attention 混合堆叠的 hybrid 架构，目标是把多模态、长上下文和高吞吐统一起来。

## 模型转换与部署
- 模型转换：PyTorch/TF -> ONNX/TRT/其他后端图。
- 导出不支持算子：自定义插件、图替换、前后处理下沉、算子拆解。
- 精度掉点排查：先对齐预处理/后处理，再逐层比对中间激活与算子实现。
- 返回值不对：先查输入归一化、tokenizer、dtype、layout、mask 与后处理。
- 性能不达标：先分 prefill/decode，查热点算子、带宽、并发、缓存命中。
- 模型优化常见项：冗余消除、常量折叠、算子融合、内存规划、调度优化。

## 常用指标
- 时间：TTFT、decode 单 token 时延、端到端时延。
- 内存：峰值内存、稳态内存、KV cache 占用。
- 带宽：峰值带宽、有效带宽利用率。
- 功耗：平均功耗与温升。
- 稳定性：长时间运行是否抖动、是否 OOM、是否降频。

## 量化核心问题
- 为什么常用对称量化：实现简单、硬件友好、整数路径高效，尤其权重量化常见。
- 非对称何时用：激活分布偏移明显时更合适，但实现复杂度更高。
- GPTQ vs AWQ：GPTQ 偏后验误差最小化；AWQ 偏保护关键权重通道，实战常看模型和硬件适配。
- 激活值难量化：动态范围大、分布随输入变化，容易引起饱和和精度损失。
- 哪些参数能量化：线性层权重最常见；激活、KV、部分归一化层要谨慎。
- 量化步骤：校准数据 -> 统计分布 -> 选粒度/bit -> 量化/反量化插入 -> 验证精度与性能。
- 极不均匀分布（如最小值 -10000）如何量化：做 clipping、分段/对数缩放、离群值单独处理或混合精度保留。

## 推理阶段优化
- prefill 技术：大 batch、算子融合、FlashAttention、图优化。
- decode 技术：KV cache 布局、paged/cache 复用、小 kernel 合并、低 launch 开销。
- 什么时候优先调度优化：当 GPU 利用率不高且有明显空泡/等待时。
